qmemcpy

void *qmemcpy(void *dest, const void *src, size_t n);

qmemcpy 是一个内存拷贝函数，常见于 Qt 框架的底层实现中。它的作用和标准C库函数 memcpy 类似，用于将一块内存的数据复制到另一块内存区域。

基本用法：

void *qmemcpy(void *dest, const void *src, size_t n);
  • dest：目标内存地址（复制到这里）
  • src：源内存地址（从这里复制）
  • n：要复制的字节数

功能说明：

  • 从 src 地址开始，复制 n 个字节到 dest 地址。
  • 复制的区域不能重叠，否则结果未定义。
  • 返回值通常是 dest。

与 memcpy 的区别：

  • qmemcpy 主要用于 Qt 内部，可能有平台相关的优化。
  • 在大多数情况下，qmemcpy 和 memcpy 的用法和效果是一样的。

示例：

char src[10] = "123456789";
char dest[10];
qmemcpy(dest, src, 10);
// 现在 dest 和 src 内容一样

base64

原理

为什么叫Base64

Base64是一种二进制到文本的编码方式，常用于解决不同系统及传输协议中二进制数据的不兼容问题。它将8bit位一组的数据拆分为6位一组，映射到64个ASCII字符中，编码后数据长度为原来的4/3。它是基于(Base)64个字符的一种编码方式。使用其编码后的文本只包含64个ASCII码字符（偶尔加一个填充字符=）

base64表和示例

以下图的表格为示例，我们具体分析一下整个过程。

第一步：“M”、“a”、"n"对应的ASCII码值分别为77，97，110，对应的二进制值是01001101、01100001、01101110。如图第二三行所示，由此组成一个24位的二进制字符串。 第二步：如图红色框，将24位每6位二进制位一组分成四组。 第三步：在上面每一组前面补两个0，扩展成32个二进制位，此时变为四个字节：00010011、00010110、00000101、00101110。分别对应的值（Base64编码索引）为：19、22、5、46。 第四步：用上面的值在Base64编码表中进行查找，分别对应：T、W、F、u。因此“Man”Base64编码之后就变为：TWFu。

流加密算法

原理

流加密算法（Stream Cipher），是一种加密方式，它通过将明文与一个随机生成的密钥流进行逐位或逐字节异或操作来产生密文。

流加密算法的基本流程

密钥流生成器（Key Stream Generator）：流加密算法的核心是密钥流生成器，它使用一个短的密钥（种子密钥）来生成一个长的密钥流。这个密钥流决定了加密的随机性和强度。

异或操作（XOR Operation）：明文数据与密钥流进行异或操作，生成密文。解密时，密文与同一密钥流再次进行异或操作，恢复出明文。

成员

1. RC4 (Rivest Cipher 4)

  • 特点：简单高效，软件实现快，但存在严重安全漏洞。
  • 密钥长度：可变（通常40-256位）。
  • 问题：密钥重用或弱密钥会导致攻击（如WEP协议中的漏洞）。现已被弃用。
  • 应用历史：曾用于SSL/TLS、WEP等。

3. AES-CTR (AES in Counter Mode)

  • 原理：将AES块加密转换为流加密模式，通过加密递增的计数器生成密钥流。
  • 特点：
    • 结合AES的高安全性和流加密的并行性。
    • 无填充需求，适合加密任意长度数据。
  • 密钥长度：128/192/256位（与AES一致）。
  • 应用：磁盘加密、网络协议（如IPSec）。

4. Trivium

  • 类型：硬件优化的轻量级流密码。
  • 特点：适合资源受限设备（如IoT），设计简单但安全性较高。
  • 密钥长度：80位。
  • 标准化：eSTREAM项目最终算法之一。

5. Grain

  • 类型：超轻量级流密码（针对硬件优化）。
  • 特点：极低功耗，适合RFID标签等设备。
  • 版本：Grain-128（密钥长度128位）。

6. ZUC (祖冲之算法)

  • 起源：中国设计的3G/4G加密标准。
  • 特点：高安全性，通过国际标准化（3GPP LTE加密）。
  • 应用：移动通信（如中国4G/5G网络）。

7. E0 (Bluetooth加密)

  • 用途：蓝牙协议（如蓝牙2.0）的流加密。
  • 问题：存在已知攻击，后续蓝牙版本改用更安全的算法

原理

在密码学中，RC4是一种串流加密算法，密钥长度可变。他加密解密使用相同的密钥，因此也被称为“流加密算法”。RC4由伪随机数生成器和异或运算组成。RC4的密钥长度可变，范围是[1,256]。RC4一个字节一个字节地加解密。给定一个密钥，伪随机数生成器接受密钥并产生一个S盒。S盒用来加密数据，而且在加密过程中S盒会变化。由于异或运算的对合性，RC4加密解密使用同一套算法。

基本操作

1、初始化S-Box
2、KSA过程（置乱刚刚初始化完成的S表）
  - 初始化密钥（可无可有）
  - 置乱过程（KSA）
3、PRGA过程（生成密钥流，用于与明文进行异或生成密文）

def decrypt(s_box, cipher):
    """RC4变种解密函数（逆向加密逻辑：先异或后加20）"""
    s, i, j = s_box.copy(),  0, 0  # 复制S盒避免修改原数据 
    plain = bytearray()
    
    for byte in cipher:
        i = (i + 1) % 256         # 更新i索引（循环左移）
        j = (j + s[i]) % 256      # 更新j索引（基于S盒动态值）
        s[i], s[j] = s[j], s[i]   # 交换S盒中i和j位置的元素 
        
        # 计算密钥流字节（等效于C代码中的v8逻辑）
        key_byte = s[(s[i] + s[j]) % 256]
        # 解密：逆向操作（先减20再异或）
        plain.append(((byte  - 20) & 0xFF) ^ key_byte)
    
    return bytes(plain)

为何使用 & 0xFF：技术背景与必要性详解
1. 确保数值在单字节范围内（0-255）
字节的本质：在计算机中，一个字节（byte）由8位二进制组成，取值范围为 0-255。任何超过或低于此范围的数值都需要通过 模256运算（即 % 256）或 按位掩码（& 0xFF）进行修正。
示例场景：
若 byte = 5，则 5 - 20 = -15，直接异或操作会导致负数值参与运算。
通过 -15 & 0xFF，结果修正为 241（等价于 -15 % 256），确保数值在合法范围内。

DES

基于我搜索到的资料，以下是关于DES算法的详细解析：

一、算法基础

  1. 性质与用途
    DES（Data Encryption Standard）是一种对称分组密码算法，由IBM于1970年代设计，1977年被美国国家标准局（现NIST）确立为加密标准。主要用于民用敏感信息加密，如金融交易、电子文件保护等。

  2. 核心参数

    • 分组长度：64位（8字节）。
    • 有效密钥长度：56位（输入64位密钥，含8位奇偶校验位）。
    • 迭代轮数：16轮。

二、加密流程

  1. 初始置换（IP）
    明文分组通过固定置换表（IP表）重新排列，输出分为左半部分（L₀）和右半部分（R₀），各32位。

  2. 16轮迭代运算
    每轮操作（以第i轮为例）：

    • 左输入：Lᵢ = Rᵢ₋₁
    • 右输入：Rᵢ = Lᵢ₋₁ ⊕ F(Rᵢ₋₁, Kᵢ)
      其中 F函数 是关键步骤：
  • 扩展置换（E） ：将32位Rᵢ₋₁扩展至48位。
  • 与子密钥异或：48位结果与轮密钥Kᵢ按位异或。
  • S盒代换：48位数据分8组，每组通过6→4位非线性代换（共8个S盒），输出32位。
  • P盒置换：32位数据按固定表置换。
  1. 末置换（IP⁻¹）
    16轮后合并L₁₆与R₁₆（需交换顺序为R₁₆L₁₆），再通过IP⁻¹表置换得到64位密文。

三、密钥生成

  1. 子密钥派生

    • 初始压缩（PC-1） ：64位密钥→56位（去除校验位），分为左右28位（C₀/D₀）。
    • 循环左移：C₀/D₀按轮次左移1-2位，生成16组Cᵢ/Dᵢ。
    • 压缩置换（PC-2） ：56位CᵢDᵢ→48位轮密钥Kᵢ。

  2. 解密特性
    解密流程与加密相同，但子密钥使用逆序（K₁₆→K₁）。

四、安全性分析

  1. 优势

    • 算法公开透明，依赖密钥保密。
    • 混淆（S盒）与扩散（P盒）结合，增强非线性。

  2. 缺陷

    • 密钥过短（56位） ：易受暴力破解（1998年被22万美元设备攻破）。

    • 弱密钥存在：部分密钥导致加密结果可预测。
    • 分组长度限制：64位分组易受重放攻击。

  3. 改进方案

    • 3DES：三重加密（加密-解密-加密），密钥长度提升至112/168位。
    • AES替代：2001年起NIST推荐更安全的AES算法。

花指令

花指令（或者叫脏字节）是企图隐藏掉不想被逆向工程的代码块(或其它功能)的一种方法，在真实代码中插入一些垃圾代码的同时还保证原有程序的正确执行，而程序无法很好地反编译, 难以理解程序内容，达到混淆视听的效果。

如ida会呈现jmupout(xxxxx)，说明在xxxxx地址处ida反编译失败了所以我们要在xxxxx处分析汇编代码，找出花指令并nop掉

汇编指令

  1. jne：jum if not equal if(a=b)
  2. jl:jump if less if(a>b)
  3. jg:jum if greater
  4. jle: jump if less equal (<=)
  5. jz: Jump if Zero
  6. test 操作数1,操作数2 : 对操作数1和操作数2做按位与（AND），结果不保存，只影响标志位。

作用

运算

按位与运算（AND）：对两个二进制数对应的每一位进行操作，如果两个数的对应位都是 1，则结果为 1，否则为 0。

UPX

是什么

UPX（Ultimate Packer for eXecutables） 是一款先进的可执行文件压缩器，常用于对 .exe、.dll 等 PE 文件进行压缩。它广泛用于程序瘦身、代码保护，有时也被恶意软件用于免杀处理。

用途

减小文件体积，提高分发效率

防止文件被修改、逆向分析

保护安全软件的执行文件

恶意软件使用其隐藏真实代码，增加分析难度

技术原理

1.压缩：UPX将原始可执行文件的代码和数据段压缩，并将其封装在一个自解压引导段中。

2.加载时解压：运行时，封装的自解压模块首先运行，解压原始内容至内存，然后跳转到原始程序入口点执行。

UPX 加壳其实是利用特殊的算法，对 EXE、DLL文件里的资源进行压缩。类似WINZIP 的效果，只不过这个压缩之后的文件，可以独立运行，解压过程完全隐蔽，都在内存中完成。

解压原理，是加壳工具在文件头里加了一段指令，告诉 CPU，怎么才能解压自己。当加壳时，其实就是给可执行的文件加上个外衣。用户执行的只是这个外壳程序。当执行这个程序的时候这个壳就会把原来的程序在内存中解开，解开后，以后的就交给真正的程序。

apk结构

apk 全称 Android Package，它相当于一个压缩文件，只要在电脑上将apk后缀改为zip即可解压。

文件 注释
assets目录 存放APK的静态资源文件，比如视频，音频，图片等
lib 目录 armeabi-v7a基本通用所有android设备，arm64-v8a只适用于64位的android设备，x86常见用于android模拟器，其目录下的.so文件是c或c++编译的动态链接库文件
META-INF目录 保存应用的签名信息，签名信息可以验证APK文件的完整性，相当于APK的身份证(验证文件是否又被修改)。在编译生成一个api包时，会对所有要打包的文件做一个校验计算，并把计算结果放在META-INF目录下。在安装时，如果校验结果与META-INF下的内容不一致，系统就不会安装这个apk。保证了apk包里的文件不能被随意替换。
res目录 res目录存放资源文件，包括图片，字符串等等，APK的脸蛋由他的layout文件设计
AndroidManifest.xml文件 APK的应用清单信息，它描述了应用的名字，版本，权限，引用的库文件等等信息
classes.dex文件 classes.dex是java源码编译后生成的java字节码文件，APK运行的主要逻辑
resources.arsc文件 resources.arsc是编译后的二进制资源文件，它是一个映射表，映射着资源和id，通过R文件中的id就可以找到对应的资源

AndroidManifest.xml

文件是整个应用程序的信息描述文件，定义了应用程序中包含的Activity,Service,Content provider和BroadcastReceiver组件信息。每个应用程序在根目录下必须包含一个AndroidManifest.xml文件，且文件名不能修改。它描述了package中暴露的组件，他们各自的实现类，各种能被处理的数据和启动位置。

属性 定义
versionCode 版本号，主要用来更新，例如:12
versionName 版本名，给用户看的，例如:1.2
package 包名，例如：com.zj.52pj.demo
uses-permission android:name="" 应用权限，例如：android.permission.INTERNET 代表网络权限
\
android:label="@string/app_name" 应用名称
android:icon="@mipmap/ic_launcher" 应用图标路径
android:debuggable="true" 应用是否开启debug权限
