蓝牙灯泡重放攻击复现

仅用于授权测试与教学研究，禁止用于未授权设备与生产环境。

一、实验目标

本实验聚焦一个典型 IoT 安全问题：

  • 在 BLE 场景下，应用层控制指令如果缺少消息完整性保护、随机挑战或会话绑定，可能被捕获后直接重放。
  • 攻击者即使不理解完整协议语义，也可能通过“抓包 + 回放”复现关键控制动作（如开关灯）。

二、实验环境

  • 目标设备：Lifesmart 智能灯泡
  • 抓包工具：Wireshark 3.4 + nRF52832 Dongle
  • 交互工具：LightBlue App、Lifesmart 官方 App
  • 参考资料：https://blog.csdn.net/XiaoXiaoPengBo/article/details/115156929

实验前提：

  • 手机与灯泡可正常连接并完成基础控制。
  • 抓包网卡可稳定跟随 BLE 会话。
  • 手机保持前台连接状态，避免会话频繁断开。

三、最小必要背景（ATT/GATT）

BLE 常见控制链路是：

  • App 对某个 Characteristic 执行写操作（Write Request / Write Command）。
  • 设备在对应服务中解析 Value 字段并执行动作。

因此，若该 Value 没有被动态校验（如时间戳、随机数、签名、会话计数器绑定），就可能被重放。

四、攻击链路

1）识别目标设备

先用官方 App 连接灯泡，再用 LightBlue 观察附近设备 RSSI 变化。靠近设备时信号明显增强、远离时明显衰减的目标即为候选设备。

我这里识别到目标名称为 BULB。

2）抓取控制流量

使用 nRF52832 Dongle 配合 Wireshark 抓包，过滤 ATT 层数据：

btatt

抓包过程中选定目标设备地址，待 Dongle 指示状态稳定后进行多次“开灯/关灯”操作，构造可对比样本。

3）定位可复用指令

对比多次操作后可观察到：

  • 两条关键写入报文与开关动作强关联。
  • Value 字段存在规律变化，倒数第 7 位在连续操作中自增。
  • 开灯报文末尾常见 ...181，关灯报文末尾常见 ...180。

同时可定位到对应 Service/Characteristic 的 UUID，为后续回放提供写入位置。

4）执行重放验证

在 LightBlue 中对目标 Characteristic 进行写入，使用已捕获的关灯控制值：

0e000000000000000000000410020180

写入后灯泡关闭，说明捕获指令可被直接复用，重放成立。

五、漏洞本质分析

从现象看，该控制流程可能存在以下薄弱点：

  • 缺少一次性挑战值（nonce）或时序有效性校验。
  • 控制指令未与当前会话强绑定（抓到的历史包仍可用）。
  • 指令完整性保护不足，设备侧对报文新鲜度验证不严格。

这类问题在 BLE 轻量控制设备中较常见，属于“功能可用优先于安全设计”的典型落地风险。

六、防护建议

面向设备厂商与固件开发侧，可优先落地以下策略：

  • 在应用层指令引入随机挑战与响应机制，拒绝历史报文。
  • 指令中加入时间窗和递增计数器，并在设备端严格校验。
  • 对控制指令做消息认证（如基于会话密钥的 MAC），避免可伪造与可重放。
  • 强制敏感控制走加密链路，避免明文特征值可直接复写。
  • 增加异常行为检测（短时间重复同值写入、跨端突发写入告警）。

七、复现中的常见问题

  • 连接不稳定：手机需保持前台，避免系统回收蓝牙会话。
  • 抓不到关键包：优先保证开关动作节奏清晰，减少无关操作噪声。
  • 回放无效：检查写入句柄、UUID、Value 长度及大小端格式是否一致。

八、总结

本实验验证了 BLE 智能灯泡在控制指令新鲜度与会话绑定方面的风险：

  • 攻击门槛低，依赖公开工具即可复现。
  • 影响直观，直接触发物理控制行为。
  • 修复路径明确，关键在于“动态校验 + 完整性保护 + 会话约束”。

对车联网与 IoT 场景而言，这类“可重放控制链路”值得作为基线测试项长期纳入验证流程。
