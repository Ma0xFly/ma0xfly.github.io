数学计算漏洞：整型溢出与精度舍入 (Math & Precision Vulnerabilities)

标签: #漏洞类型/数学计算 | #协议类型/DeFi/核心业务逻辑 | #严重程度/High

📌 1. TL;DR (一句话快速回顾)

  • 本质成因：智能合约中的数学引擎存在两大历史级死穴：整型溢出（旧版本无边界检查，导致极值翻转现象）以及精度舍入（由于 EVM 不支持浮点数，所有除法默认向下取整截断，结合计算顺序或代币精度差异，导致价值被抹除或无中生有）。
  • 攻击路径：对于溢出，攻击者传入极大/极小值进行溢出构造（如 0-1 变成宇宙最大数）；对于精度舍入，攻击者利用精密计算的极小资产流转（例如经典 ERC4626 “1 wei 膨胀攻击”），人为推高系统兑换汇率，将普通用户的零头资产通过系统向下取整彻底侵吞。

🔍 2. 漏洞底层原理 (Root Cause)

正常业务逻辑

  • 代币体系运算：转账扣减（A - B），以及份额获取、AMM 自动做市商换算（通常涉及 A * B / C 的复合计算）。

缺陷剖析

缺陷A：整型溢出 (Integer Overflow/Underflow)

  • 致命原理：Solidity < 0.8.0 版本没有自带的 SafeMath，uint256 只能容纳 0 到 2^256 - 1。发生下溢出时（如 0 - 1），不仅不会报错，反而会因变成极大正数（type(uint256).max），使得本来没钱的黑客瞬间变为首富并绕过一切资金体量 Checks。

缺陷B：精度舍入与截断 (Precision Loss & Rounding)

  • 致命原理：因为 Solidity 没有小数，10 / 3 的结果是 3 而不是 3.33，这 0.33 因为向下取整被永远抹除了。
  • 为何产生冲突：
    • 除法截断先于乘法：如果公式是 A / B * C，那么 A / B 产生的小数部分截断会被最后的 * C 呈千百倍放大。
    • 单向舍入逻辑崩塌：在 ERC4626 金库或收益池中，公式往往是 获得的股份 = 充入的资产 * 池子总股份 / 池子总资产。 如果攻击者提前打入 1 笔极大的真实资产，把 池子总资产 分母撑得特别大。后面的散户存钱时，算出来的“获得股份”如果是 0.999，由于 Solidity 向下取整，散户最终获得的股份是 0！散户白交钱，却一分股份拿不到，这笔钱也就进入了底池，变成了黑客的利润。

⚔️ 3. 攻击执行路径 (Exploit Steps)

路径 1：古典的批量发送溢出攻击 (Batch Overflow)

  1. [触发阶段]: 攻击者向漏洞合约的高级转账函数一次传入天文数字金额，但通过巧妙乘以接收人数，刚好造出上溢出变成 0。
  2. [利用阶段]: 被欺骗的合约判定总需求额为 0，绕过余额检查，向下给每个人下发天文数字的钱。

路径 2：现代高阶 ERC4626 份额通胀攻击 (Inflation Attack)

  1. [准备阶段]: 攻击者敏锐地发现一个全新干净、还没有任何资产的借贷金库。
  2. [建立锚点]: 攻击者正常存入 1 wei 的资产，获得了初始的 1 wei 的池子股份 (Shares)。
  3. [拉伸分母]: 攻击者不通过正常的 Deposit 函数，而是直接强行向池子合约 transfer 转入了一大笔巨资（比如 10000 块钱）。此时，池子总共有 $10000 + 1$ 的资产，但系统登记的股份依然只有黑客手上的那 1 wei 的股份！这意味着：每一股现在价值 $10000$ 块！
  4. [无损收割]: 随后进来的羊毛党，如果存了 $9999$ 块钱。系统计算它应得的股份：9999 * 1 / 10001 = 0.9998。 接着致命的 向下取整 生效：0.9998 变成了 0 份股份！ 后面进来的这 $9999$ 块白白喂给了奖金池，黑客最后拿着自己绝版的 1 wei 股份，把池子全部提兑卷走。

💻 4. 关键代码示例 (Snippets)

漏洞代码 (精度计算截断篇)

// 最典型的反面公式：先除后乘 导致的毁灭灾难
function getReward(uint amount, uint totalReward, uint totalStake) public pure returns (uint) {
    // 假设散户 amount=100，totalStake=300，totalReward=1000
    // 如果先除： 100 / 300 = 0 (截断发生!) -> 0 * 1000 = 0。（散户一分钱拿不到）
    return (amount / totalStake) * totalReward;
}

// 金库(Vault) 份额向下取整的深渊
function deposit(uint assets) public returns (uint shares) {
    // 黑客提前拉大 totalAssets 会导致后进来的散户因为这个除法计算出小数被强行抹 0
    shares = (assets * totalSupply) / totalAssets;
    require(shares > 0, "Zero shares"); // 如果开发者连这一句都没写，散户的钱就灰飞烟灭了
}

🛡️ 5. 修复与防范方案 (Mitigations)

  • 针对整型溢出：
    • 版本控制：坚决升级迁移至 Solidity >= 0.8.0（自带底层 Panic 类型回滚保护）。
    • 防弹库：如果在老项目代码维护中，必须引用 OpenZeppelin 的 SafeMath。
  • 针对精度舍入：
    • 口诀：先乘后除 (Multiply before Dividing)：所有财务数学模型的运算里，必须要把乘法分子全部算尽，把除法放在公式的最后一步。
    • 金库防通胀底线（Virtual Shares/Dead Shares）：参照最新的 OpenZeppelin 修复策略：在金库第一次被创建充入时，强制向一个绝对黑洞地址（如 address(0)）铸造销毁极小一部分的初始股份（比如 1000 份），这使得黑客永远无法包揽全盘的 100% 且拉伸汇率成本飙升到极度不划算。
    • 提款与发股不同向：给用户发权益（比如放款/给股份）应当 向下取整(mulDivDown)，跟用户收债或者赎回计算他需归还多少时应当 向上取整(mulDivUp) 以保护协议永远不吃亏。

📖 6. 真实案例与靶场 (References)

  • 经典黑客事件：
    • 溢出向：BEC BatchOverflow (2018) 数十亿美元凭空印发出局。
    • 舍入向：由于未正确使用 mulDivUp/mulDivDown 计算精度引发的极小值榨干池子的 Balancer 闪电贷利用（2023）；无数新盘无防护 ERC4626 项目在开启的一瞬间首杀充入被抢跑挤压。
  • 靶场/练习：Damn Vulnerable DeFi - [Shards] (极为硬核的浮点/精度/向上向下取整混合灾难利用题)；Ethernaut - [Token], [Dex]。
  • 拓展阅读：精度计算不再只是单纯的漏洞，它逐渐成为了很多 DeFi 黑客通过数以万计笔交易“吸血”的无伤稳定套利通道（往往通过操纵汇集粉尘攻击实现）。深度熟悉 OpenZeppelin 的 Math.sol 中 mulDiv 工具是写好 DeFi 合约的基石。
