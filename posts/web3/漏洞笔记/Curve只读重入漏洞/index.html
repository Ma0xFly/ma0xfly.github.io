Curve只读重入漏洞 (Read-Only Reentrancy)

标签: #漏洞类型/只读重入 | #协议类型/DeFi/DEX/借贷 | #严重程度/High

📌 1. TL;DR (一句话快速回顾)

  • 本质成因：早期包含原生 ETH 的 Curve 池在移除流动性执行流中，存在先销毁代币更新部分账本、发还 ETH 打断执行流，随后才发送 ERC20 更新余下账本的顺序缺陷。在这个被打断的“安全真空期”内池子因状态未完全更新而陷入严重的计价极值。
  • 攻击路径：通过闪电贷巨资注水并抽干池子，触发回调时劫持执行流。趁着池子正处于畸形的中间状态，恶意调用把该池子当做预言机的倒霉第三方受害协议，让其读取出离谱的价格并执行错误判定。

🔍 2. 漏洞底层原理 (Root Cause)

正常业务逻辑

Curve 虚拟价格 get_virtual_price() 是评估 1 单位 LP 代币价值的关键函数，它的底层逻辑近似为：

$$\text{Virtual Price} = \frac{\text{Invariant } (D)}{\text{Total Supply}}$$

  • Invariant (D)：池子中所有资产的总价值评估（依赖于池子当前持有的 两种币的余额）。
  • Total Supply：LP 代币的总发行量。 正常的移除流动性操作应当同步成比例减少 Total Supply 和资产账户里的余额（D），从而保证 Virtual Price 的稳定。

缺陷剖析

  • 致命代码/机制：Curve 早期 Vyper 合约中的 remove_liquidity 执行顺序缺陷。
    1. 销毁 LP 代币：首先销毁用户的流动性代币会导致池子的 totalSupply 锐减。
    2. 退还原生 ETH：通过底层的 .call.value() 退还 ETH，这就将触发接收者（攻击智能合约）的 fallback() 或 receive() 函数，交出了控制权限。（致命点！）
    3. 退还 ERC20 代币：执行完回调后，最后合约才会转移池子中的 ERC20 代币（如 stETH）给用户，重新记录真实的余额。
  • 为何产生冲突：在步骤 2 中交出控制权时产生了一个致命的“时间停止”。
    • 预期状态：LP代币减少，连同池子资金取出，虚拟价格维持稳定。
    • 实际状态：攻击者的巨量 LP 已经在第 1 步被销毁（Total Supply 极小），但第 3 步还没执行，应退回的巨量 ERC20 资产仍然留在 Curve 池里（此时 Invariant (D) 维持极大）。如果此时拉取计算此时的价格数据，算出来的会是一个极其巨大的天文数字。 因为第三方协议读取数据时不属于修改状态的操作，哪怕 Curve 池子自身的修改状态函数有重入锁机制限制，也防不住跨合约通过 view 函数来畅通无阻的拉黑料。

🤔 为什么叫“只读重入”？

理解这个漏洞的前提是理清控制流：

  • 经典重入：攻击者在合约A的状态更新完成前，再次进入合约A的其他修改状态函数（如继续withdraw），目的是破坏并薅走原合约资产。
  • 只读重入：攻击者在合约A（如Curve池）状态异常的真空期，跑去调用第三方合约B（如借贷协议），而合约B为了计算价值，又再次进入了合约A的只读函数（get_virtual_price()）。

核心区别： 控制流依然绕回了原合约A，但触发的是 view 函数。攻击者不破坏合约A本身，而是利用A的“残疾状态”去欺骗并掠夺依赖它的第三方合约B。

⚔️ 3. 攻击执行路径 (Exploit Steps)

以受害者为借贷协议A（允许抵押 DVT 借出 LP，使用 Curve 判价）为例：

  1. [准备阶段]: 攻击者通过闪电贷借出巨量的 ETH 和 stETH 初始资金。
  2. [触发阶段]: 攻击者将等比例的资产注入 Curve 池获得巨量 LP 代币，马上调用 remove_liquidity 移除全额流动性以触发系统退回原生 ETH 走入攻击合约 receive()。
  3. [利用阶段]: 在 receive() 函数回调内部，抓取此时暂未被更新完毕处在价格畸高的时间窗口，攻击者调用借贷协议 A 的 liquidate() 去清算别的健康正常用户。借贷协议A请求读取 Curve 价格，判断所有借出 LP 的用户都已经严重资不抵债，允许攻击者进行低成本清算。
  4. [获利退场]: 攻击者的回调执行结束，Curve 继续将 stETH 退给攻击者状态回归正常。攻击者归还闪电贷并携夺来的抵押物获利离场。

漏洞代码 (Vulnerable Code)

# Curve 池中移除流动性 (Vyper 示意简例)
@external
@nonreentrant('lock')
def remove_liquidity(amounts: uint256[N_COINS]):
    # 1. 销毁 LP token (由于巨资撤离导致 totalSupply 暴跌)
    self.token.burnFrom(msg.sender, burn_amount)
    
    # 2. 发送原生 ETH，将执行流交给了外部攻击合约的 receive() -> 致命点！
    raw_call(msg.sender, b"", value=amounts[0]) 
    
    # 3. 此时外部正依靠重入胡作非为，等执行流回来后继续发送其应当被取走的 ERC20。太晚了！
    ERC20(coins[1]).transfer(msg.sender, amounts[1])

🛡️ 5. 修复与防范方案 (Mitigations)

  • 协议自身的修复：
    • DeFi 协议涉及到资产转移时应该严格遵循 CEI（Checks-Effects-Interactions）原则防范。
    • 将 view 等只读数据涉及关键状态的函数同样列入重入锁 @nonreentrant("lock") 保护机制之中，阻止其在系统状态扭曲时对外胡乱输出离谱的数据。
  • 第三方调用者的防范：
    • 不要单一依赖 DEX 现价预言机：绝不要在智能合约逻辑中直接结合 get_virtual_price() 或单一由现存的资金量级和函数逻辑预判和评估价格，因为这些往往是最容易随巨鲸注水受到干涉污染的数据源；应该融合 ChainLink 等去中心化专业第三方预言机价格网来做交叉对比。
    • 主动试探锁状态：去中心化借贷/保险等第三方协议如果在整合那些早期且源合约依然沿用老方法又改不了时，在任何 get_virtual_price() 的查询获取动作发生之前，加一句 try/catch：主动调用源资产池合约一个带着 nonReentrant 安全锁的简单低危函数作为测试验证手段。一旦处于遭劫持利用重入的区间中底层被落锁将导致 try 解析失败转而进入 catch，由此强行让自身受害者协议这一笔相关交易安全 revert。
