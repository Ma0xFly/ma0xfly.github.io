拒绝服务攻击漏洞 (Denial of Service - DoS)

标签: #漏洞类型/执行流中断 | #协议类型/DeFi/拍卖/分红机制 | #严重程度/High

📌 1. TL;DR (一句话快速回顾)

  • 本质成因：智能合约在处理核心业务逻辑时（例如退款、更新统治者、批量发工资），过度依赖于外部调用的成功执行，或者存在遍历无上限的数组导致 Gas 耗尽。
  • 攻击路径：攻击者通过部署一个没有 receive / fallback 函数的智能合约，或是编写一个在收款时恶意耗尽所有可用 Gas 的逻辑，强行参与系统的关键环节。当漏洞合约试图向它发送资金或调用其代码时，整个交易被迫失败回滚（revert），导致整个系统的业务被永久卡死阻塞。

🔍 2. 漏洞底层原理 (Root Cause)

正常业务逻辑

  • 在夺宝游戏、拍卖出价、或平台收益分红时，如果有新的人出价更高，系统需要把前一个人的资金退还，或者按名单群发分红。

缺陷剖析

缺陷A：未捕获异常导致全局崩溃 (Unchecked External Call DoS / Revert)

  • 致命原理：Solidity 使用 transfer() 在处理向智能合约账户发送以太币时，如果对方是一个不接收 ETH 的空壳合约，或故意在其 receive() 中写了 revert()，这条转账语句就会发生致命错误，并连坐中止整个外部调用者的后续代码，导致所有状态全部撤销归零。
  • 现象：国王游戏里，我想退钱给上一个国王，但上一个国王是个不可接收钱的合约，我的退钱动作永远抛异常，因此也没有人能把它的国王头衔抢走。

缺陷B：Gas 限额耗尽勒索 (Gas Limit DoS / Out Of Gas)

  • 致命原理：以太坊的每一个区块和每一笔交易都有 Gas 上限。如果在一笔交易中，漏洞合约试图向数十万人的数组里循环派发资金，或者攻击者利用外呼时 (call) 未限制转发的 Gas，恶意接收者就可以编写一个死循环，吞噬掉你全部的手续费。
  • 现象：当外部调用吃掉所有的 Gas 后，即便退出了调用，当前合约也没有多余的 Gas 能够用来更新关键的状态变量了，交易直接爆掉。

⚔️ 3. 攻击执行路径 (Exploit Steps)

以 Ethernaut 中经典的“King (被动退款 DoS)”和“Denial (耗尽 Gas DoS)”为例：

路径 1：Revert 霸座攻击 (King)

  1. [准备阶段]: 攻击者发现了一个“拍卖出价”合约，当前的标价是 1 ETH。合约逻辑是：谁出价比 1 ETH 高，合约就把现有的拍卖金退还给老国王，然后把新买家立为新国王。
  2. [触发阶段]: 攻击者不是用自己的钱包地址，而是专门部署一个带毒的智能合约 (AttackerContract)。这个合约极其简陋，连个接客退款的 receive() 函数都没写，然后它向拍卖合约出价 1.1 ETH，成功当上了新国王。
  3. [卡死全盘]: 第三个人试图花 1.5 ETH 当国王。拍卖合约想把之前的 1.1 ETH 退还给黑客的合约，结果因为黑客合约“拒收转账”引发底层的 revert。因为交易回滚，第三人永远当不上国王，黑客合约永久霸占在这个位置上。

路径 2：Gas 吸血鬼攻击 (Denial)

  1. [准备阶段]: 一个资金合伙池每次有资金流入，都会按照名单，使用 call 给所有合伙人发放利润。
  2. [埋藏炸弹]: 黑客把自己编写的吸血鬼智能合约注册成为了其中一名合伙人。这个智能合约的 receive() 里面只写了一个动作：陷入死循环 (如 while(true){})，或者疯狂重连，直至耗尽收到的执行燃料。
  3. [引爆卡死]: 资金池启动分发业务，发给其他人都很正常。轮到给黑客合约打款时，黑客合约像黑洞一样吸干了系统给予这笔交易的所有剩余 Gas，最终抛出 Out of Gas，池子剩余的业务代码全部无法执行。

💻 4. 关键代码示例 (Snippets)

漏洞代码 (被动退款卡死型)

contract Auction {
    address public currentLeader;
    uint256 public highestBid;

    // 致命逻辑点：把退款和更新新人的数据强制绑定在了一次执行里
    function bid() public payable {
        require(msg.value > highestBid, "Not enough ETH to outbid");

        if (currentLeader != address(0)) {
            // 这行如果失败抛出异常，整个函数全部连坐，下面的赋予新任领导人直接没戏
            payable(currentLeader).transfer(highestBid); 
        }

        currentLeader = msg.sender;
        highestBid = msg.value;
    }
}

攻击核心逻辑 (Attacker Contract)

contract BadKingAttack {
    // 强行出价抢占王座
    function hack(address target) external payable {
        (bool ok, ) = target.call{value: msg.value}("");
        require(ok);
    }

    // [致命防线]：无 receive，或者在此显式地写入 revert()
    // 以后只要有人给我退钱，我就当场死给你看，拽着对方的交易一起坠机
    receive() external payable {
        revert("I won't accept any ETH. HAHAHA!");
    }
}

🛡️ 5. 修复与防范方案 (Mitigations)

  • 改“主动推送 (Push)”为“被动拉取 (Pull)”机制：这是智能合约退款机制的黄金守则。不要在处理核心新业务（如上新买家）时夹带给老买家的主动发钱行为。
    • 将前任领导者的应退款项存入一个他自己的挂账账本 (pendingReturns[oldLeader] += amount)，然后立刻完成主流程的状态更新。如果他想提款，让他自己触发单独的 withdraw() 提现函数。这是最高级的隔断手段。
  • 限制未知调用的 Gas 或者不要使用 transfer：
    • 早年 transfer() 自带极其可怜的 2300 Gas 上限以防备重入，却非常容易因为对方执行极其简单的日志动作而引发 Out of Gas 的伪 DoS 故障。
    • 对于真正不受信任的调用环节，请严苛使用低级别的 .call() 并且有意识地限制它带过去的兜底 Gas 用量：victim.call{gas: 50000, value: 1 wei}("")。
  • 警惕无限数组循环遍历：所有对 for 和 while 数组遍历的操作，必须设置单次批处理上限或分页，以防止日后人数过多导致交易天然超过以太坊单块 Gas Limit。

📖 6. 真实案例与靶场 (References)

  • 经典黑客事件：GovernMental 资金盘锁死事件 (2016)。这是一个当时的庞氏游戏合约，规则是长时间没人接盘，最后一个人通吃。当其准备向 1100 名早期玩家分发以太币时，触发了庞大的数组解绑循环动作。这极巨的遍历所需 Gas 超过了当年以太坊网络硬编码的单区块 Gas 上限！导致那笔退款交易天然永远无法被矿工打包放入区块内，数万以太坊被永久“卡死”在代码里。
  • 靶场/练习：Ethernaut - [King] (经典的无 receive() 回滚霸座), [Denial] (调用 call 时外部耗尽所有 Gas 拖网)。
  • 拓展阅读：DoS 的广义变种还可以是“利用特定的抢跑逻辑锁死池子的汇率或开启状态”，甚至是人为塞满内存导致后续交易无法创建新插槽等极度冷门的底层攻击。总之，永远对别人的地址执行持悲观隔离态度。
