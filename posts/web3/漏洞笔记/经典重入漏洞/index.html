经典重入漏洞 (Classic Reentrancy)

标签: #漏洞类型/重入 | #协议类型/DeFi/智能合约底层逻辑 | #严重程度/High

📌 1. TL;DR (一句话快速回顾)

  • 本质成因：在修改关键状态（如扣减余额）之前，就将控制流或以太坊原生转账交由外部地址处理，导致防御机制形同虚设。
  • 攻击路径：攻击者在合约打款触发的 fallback/receive 回调中，趁着余额还没被清零，反手再次调用该提款函数，通过嵌套循环把池子掏空。

🔍 2. 漏洞底层原理 (Root Cause)

正常业务逻辑

  • 用户调用提款功能（如 withdraw()），合约核对其内部账本，发送资金，然后将账本中的该用户余额减掉。

缺陷剖析

  • 致命代码/机制：开发者未能遵守 CEI 模式。在 “Effects（修改状态）” 发生前，就去执行了 “Interactions（外部交互/转账）”。

🛡️ 核心防御：CEI 模式详解

CEI (Checks-Effects-Interactions) 是智能合约开发的“黄金法则”。它的本质是：在把控制权交给别人之前，先把自己家里的账结清。

  1. Checks (检查)：
    • 动作：验证所有的前置条件（如 require 余额足够、权限检查、输入验证）。
    • 目的：确保这笔交易是有权的、合算的，不给非法请求开门。
  2. Effects (效果/状态更改)：
    • 动作：修改合约内部的状态变量（如 balances[msg.sender] -= amount、标记 hasClaimed = true）。
    • 目的：提前记录下结果。即使后面的 Interactions 触发了重入，当攻击机再次回到 Checks 阶段时，由于状态已经更新（余额已扣除），攻击会被立即拦截。
  3. Interactions (交互)：
    • 动作：执行外部合约调用（如 call、send、调用其他合约的函数）。
    • 目的：完成最终的资金转账或跨合约操作。这是风险最高的步骤，因为它会交出 CPU 的执行控制权。

为什么顺序不能乱？

  • 错误顺序 (C-I-E)：就像在餐厅买单，服务生先把钱退给你，然后才去后台改账本。你拿到钱的那一秒就可以由于各种原因（比如瞬移）再次出现在前台领钱，因为账单结清状态还没更新。
  • 正确顺序 (C-E-I)：先在账本上划掉这一笔，然后再把钱递给你。即使你拿到钱瞬间反悔重入，前台查账也会发现：“对不起，您刚才已经结清了，余额为 0”。

⚔️ 3. 攻击执行路径 (Exploit Steps)

这里是经典的“反复套娃”抽干池子：

  1. [准备阶段]: 攻击者向漏洞合约存入一笔启动资金（例如 1 ETH），让合约记录下 balances[attacker] = 1 ETH。
  2. [触发阶段]: 攻击者调用 withdraw(1 ETH)。漏洞合约在发送这 1 ETH 给攻击者时，不小心触发了攻击者合约的 receive() 函数。
  3. [利用阶段]: 攻击者在自己的 receive() 函数里立刻写入：if (address(victim).balance >= 1 ETH) victim.withdraw(1 ETH);
    • 合约一看账本：哦，balances[attacker] 还是 1 ETH（因为上一次提款后扣余额的代码还没走到），接着发第二笔钱！
    • 再次触发 receive()，如此反反复复循环往下掏。
  4. [获利退场]: 掏空漏洞合约内的所有 ETH 后，循环终止。攻击者带着卷走的脏款全身而退。

💻 4. 关键代码示例 (Snippets)

漏洞代码 (Vulnerable Code)

// 最典型的反面教材！一定要肌肉记忆记住这种错误写法
function withdraw(uint256 amount) public {
    // 1. Checks: 检查余额
    require(balances[msg.sender] >= amount, "Insufficient balance");

    // 2. Interactions: 发送以太坊 (交出执行流！致命点！)
    (bool success, ) = msg.sender.call{value: amount}("");
    require(success, "Transfer failed");

    // 3. Effects: 更新状态 (此时外部由于没有更新状态，可以无限重入)
    balances[msg.sender] -= amount;
}

攻击核心逻辑 (Attack Logic)

contract Attack {
    Victim public victim;
    
    // 启动攻击！
    function attack() external payable {
        victim.deposit{value: 1 ether}();
        victim.withdraw(1 ether); // 触发第一次提款
    }

    // 重复套娃薅羊毛
    receive() external payable {
        // 趁着此时 victim 还没执行 balances[msg.sender] -= amount
        if (address(victim).balance >= 1 ether) {
            victim.withdraw(1 ether); 
        }
    }
}

🛡️ 5. 修复与防范方案 (Mitigations)

  • 协议自身的修复：
    • 必须严格遵循 CEI 模式 (Checks-Effects-Interactions)：一定是先查账，再把余额扣了，最后再对外发钱。（即使别人重入进来，因为余额已经是 0，就会被 Checks 拦在门外）。
// 正确写法
require(balances[msg.sender] >= amount); // Checks
balances[msg.sender] -= amount;          // Effects (先扣钱！)
(bool success, ) = msg.sender.call{value: amount}(""); // Interactions (最后发钱)
  • 引入防重入锁 (ReentrancyGuard)：继承 OpenZeppelin 的 ReentrancyGuard 合约，对敏感函数加上 nonReentrant 修饰符。底层原理是设置一个进入标记。

📖 6. 真实案例与靶场 (References)

  • 经典黑客事件：The DAO 黑客事件 (2016)，这一事件直接导致以太坊硬分叉为 ETH 和 ETC。造成了数千万美元的损失，也是人类历史上最具影响力的智能合约安全事件。
  • 靶场/练习：Damn Vulnerable DeFi - [Truster], [Side Entrance], Ethernaut - [Re-entrance], [Puzzle Wallet]。
  • 拓展阅读：重入攻击经过这么多年的发展，衍生出了跨函数重入（在一个函数套另一个函数）和跨合约只读重入（见另外一篇笔记）。核心思想都是“在状态未完全一致的真空期做小动作”。
