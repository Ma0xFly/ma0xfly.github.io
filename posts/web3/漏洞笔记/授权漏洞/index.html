权限/授权漏洞 (Authorization & Access Control)

标签: #漏洞类型/访问控制 | #协议类型/所有智能合约 | #严重程度/Critical

📌 1. TL;DR (一句话快速回顾)

  • 本质成因：由于开发者忘记给敏感函数（如提款、所有权转移、初始化、自毁）加上特权修饰符，或者基于错误的逻辑进行了身份校验（如错信了 tx.origin 而不是 msg.sender）。
  • 攻击路径：攻击者或者任意的平民账户，直接调用本不该暴露出来的特许管理接口，抢夺系统控制权甚至直接挪走资金池里的资金。

🔍 2. 漏洞底层原理 (Root Cause)

正常业务逻辑

  • 合约内存在“管理员专享”和“普通用户可用”两套不同的接口体系。比如系统暂停 pause()、修改代理合约 upgradeTo()、动用金库资金 rescueFund() 等关键行为必须要有强校验：只有管理员才能调。

缺陷剖析

  • 致命代码/机制：开发者最容易犯错的大类：
    1. 漏写修饰符：原本需要 onlyOwner 的函数写成了 public 或 external，变成了“人人可用”。
    2. 错误的身份认证：把 tx.origin（发出这条交易的初始人）作为身份判定基准以防御钓鱼。
    3. 未锁闭的初始化：像 initialize() 这样的函数用来代替构造函数（在做可升级合约升级时使用）。但由于代理模式在上线时由于粗心或者机制被绕过而没有调用，导致攻击者趁虚而入自己调用这个函数。
  • 为何产生冲突：
    • 预期状态：平民账户想乱搞，遭到 require(_owner == msg.sender) 的系统无情拒绝。
    • 实际状态：攻击者长驱直入，不仅自己当上了太上皇，还把真正的团队开发者给踢出了权限名单。

⚔️ 3. 攻击执行路径 (Exploit Steps)

以极其经典的“未初始化代理合约抢跑控制权”为例：

  1. [准备阶段]: 项目方部署一套极高上大的“代理与逻辑分离”智能合约框架，但在脚本里忘记马上调用一次 logicContract.initialize() 来锁定它的归属权。
  2. [触发阶段]: 攻击者通过链上扫描工具或者查阅其开源的 Etherscan 源代码，观察到 owner 变量还是一个 0x0000...00 （全零默认状态）。
  3. [利用阶段]: 攻击者毫不留情，直接组装一笔对该特许 initialize(address newOwner) 函数的调用，并把自己的地址传入了进去。
  4. [获利退场]: 漏洞合约直接把系统的顶级管理员设为了攻击者。攻击者随后立刻调用只能由管理员使用的提取金库函数，拿钱走人。开发团队直接被防在了自己的项目外面。

💻 4. 关键代码示例 (Snippets)

漏洞代码 (Vulnerable Code)

// 灾难教科书
contract Vault {
    address public owner;

    // 错误1：构造函数没有锁定权限 (代理合约模式的常见病)
    function initialize() public {
        owner = msg.sender;
    }

    // 错误2：漏了 onlyOwner 修饰符！这简直是个送钱函数！
    function emergencyWithdraw(uint amount) external {
        (bool success, ) = msg.sender.call{value: amount}(""); 
        require(success);
    }
    
    // 错误3：防钓鱼弄巧成拙。tx.origin 是这笔交易最初的签名者。
    // 黑客可以构造一个假冒钓鱼合约诱骗老板点击转一两块钱，但其底下隐藏的转账代码，通过 `tx.origin` 会直接把老板判定成操作者，把金库权限移走！
    function transferOwnership(address newOwner) external {
        require(tx.origin == owner, "Not owner");
        owner = newOwner;
    }
}

攻击核心逻辑 (Attack Logic)

// 最傻瓜式的直截了当
vault.initialize(); // 第一步：抢走皇帝位置成为 Owner
vault.emergencyWithdraw(address(vault).balance); // 第二步：奉旨拿钱

🛡️ 5. 修复与防范方案 (Mitigations)

  • 协议自身的修复：
    • 明确和标准化权限架构：不要自己写奇怪的权限校验，乖乖使用 OpenZeppelin 的 Ownable (单一管理员) 或 AccessControl (RBAC 多角色细分权限树)。
    • 永远不要用 tx.origin 处理权限：这是红线！在身份校验里，只能看且使用 msg.sender。（例外仅仅在于当你想完全拒绝别的智能合约调用你，只准真实的“人类钱包（EOA）”调用时，可以用 require(tx.origin == msg.sender)，但不要拿它做权限判定基底）。
    • 锁死 Initialize()：在使用 UUPS 等代理模式升级智能合约的设计中，要在声明 initialize() 时候加上 initializer 修饰符，或在对应的原逻辑合约内调用 _disableInitializers() 彻底锁死后路。

📖 6. 真实案例与靶场 (References)

  • 经典黑客事件：Parity 多签钱包被毁事件 (2017)。最抽象安全事故之一：有一位小白通过未受保护的 initWallet 取得了公共基础库合约的控制权，随后因为不知情，好奇地调用了仅所有者可用的 kill（selfdestruct 自毁操作），导致超过当时价值 1.5 亿美元（50多万以太坊）被永久锁死在以太坊虚空里。
  • 靶场/练习：Damn Vulnerable DeFi - [Climber] (复杂精妙的高阶基于时间锁的权限偷盗)，Ethernaut - [Fallback], [Fallout], [Delegation], [Telephone], [Naught Coin], [Dex Two]。
  • 拓展阅读：目前大多数单纯的忘记 onlyOwner 已经在基础编译层面越来越少见。当前的高阶实战里，更多是出在复杂 DeFi 乐高嵌套时上下文权限错位映射，或是如上文提到的在可升级治理时间所产生的 Delegatecall 的储蓄槽位权限冲突。
