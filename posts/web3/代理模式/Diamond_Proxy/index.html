Diamond Proxy（钻石代理 / ERC-2535）

一句话总结：Diamond Proxy 是一种**多面代理（Multi-Facet Proxy）**标准，将一个 Proxy 的后端拆分成多个独立的 Facet（切面）合约，每个 Facet 承载不同的功能模块。它通过函数选择器路由表实现函数级别的精准增删替换，彻底突破了单个合约 24KB 的大小限制。

📜 核心元数据与参阅资料 (Metadata & References)

  • 关联 EIP/ERC 编号：ERC-2535: Diamonds, Multi-Facet Proxy | EIP-1967: Standard Proxy Storage Slots
  • 权威参考实现：diamond-3-hardhat (Reference Implementation)
  • 底层基座依赖：EIP-1967 标准存储槽（可选）、ERC-165 接口自省（Loupe 部分）

1. 核心理念与解决的痛点 (Motivation)

  • 前代模式 (Transparent / UUPS) 的缺陷：
    1. 24KB 硬限制：EVM 规定单个合约最大 24KB。一旦 DeFi 协议功能爆炸式增长（如同时包含 ERC20、治理、质押、清算模块），一个 Implementation 根本装不下。
    2. 整体替换的暴力升级：Transparent / UUPS 升级时必须完全替换整个 Implementation，即使你只想修一个小 bug 或加一个小函数。
  • 破局思路：将”一个巨大的后端”拆分成多个小的 Facet 合约（类似微服务架构）。Diamond 主合约内部维护一张 “函数选择器 → Facet 地址”的路由映射表，每次调用到达 fallback 时，根据 calldata 前 4 字节去查表并 delegatecall 到对应的 Facet。
  • 专属核心概念：
术语定义
Diamond主合约（代理），通过 delegatecall 调用各个 Facet 执行函数。有状态，数据存储在 Diamond 中。
Facet无状态的合约或 Solidity 库，包含外部函数。一个 Facet 可被多个 Diamond 共用。
DiamondCut添加/替换/移除函数的操作，即”切割”钻石。
Loupe内省（Introspection）接口，用于查询 Diamond 有哪些 Facet 和函数。

                    │   • diamondCut() 增删改     │

   │  (ERC20逻辑) │ │  (治理逻辑)   │ │  (质押逻辑)   │
   │  transfer()  │ │  vote()      │ │  stake()     │

路由机制：Diamond 的 fallback() 中维护了一张 mapping(bytes4 => address)，根据函数选择器（前 4 字节）查找对应 Facet 地址，然后 delegatecall 到该 Facet。

B. Delegatecall 执行上下文表

属性值归属
代码 (Code)来自被路由到的那个 Facet
存储 (Storage)属于 Diamond 主合约
msg.sender保留原始外部调用者
msg.value透传

C. 升级逻辑归属

属性说明
升级函数在哪？Diamond 主合约自身（diamondCut()）
谁有权升级？取决于 diamondCut() 的权限控制（通常 onlyOwner 或 DAO）
升级的底层操作修改 Diamond 内部的选择器→Facet 路由映射表（增/删/替换条目）
升级粒度函数级别（可以只添加一个函数、只替换一个函数、只删除一个函数）

3. 核心接口与源码骨架 (Core Interfaces & Code)

A. IDiamondCut — 切割接口

interface IDiamondCut {
    enum FacetCutAction { Add, Replace, Remove }
    
    struct FacetCut {
        address facetAddress;       // Facet 合约地址
        FacetCutAction action;      // 操作类型
        bytes4[] functionSelectors; // 要操作的函数选择器列表
    }
    
    /// @dev 批量操作：添加/替换/移除函数。可选在操作后执行初始化调用。
    function diamondCut(
        FacetCut[] calldata _diamondCut,
        address _init,              // 可选：升级后执行初始化的合约
        bytes calldata _calldata    // 可选：初始化调用数据
    ) external;
    
    event DiamondCut(FacetCut[] _diamondCut, address _init, bytes _calldata);
}

B. IDiamondLoupe — 内省接口

interface IDiamondLoupe {
    struct Facet {
        address facetAddress;
        bytes4[] functionSelectors;
    }
    
    function facets() external view returns (Facet[] memory);
    function facetFunctionSelectors(address _facet) external view returns (bytes4[] memory);
    function facetAddresses() external view returns (address[] memory);
    function facetAddress(bytes4 _functionSelector) external view returns (address);
}

Loupe 接口让任何人都可以查询 Diamond 当前有哪些 Facet 和函数，提供完全的透明性和链上可审计性。

C. 关键函数源码骨架

fallback() — 选择器路由核心

fallback() external payable {
    // 从 Diamond Storage 中读取路由表
    DiamondStorage storage ds = diamondStorage();
    // 用 calldata 前 4 字节查找对应的 Facet 地址
    address facet = ds.selectorToFacet[msg.sig];
    require(facet != address(0), "Diamond: Function does not exist");
    // delegatecall 到对应 Facet
    assembly {
        calldatacopy(0, 0, calldatasize())
        let result := delegatecall(gas(), facet, 0, calldatasize(), 0, 0)
        returndatacopy(0, 0, returndatasize())
        switch result
        case 0 { revert(0, returndatasize()) }
        default { return(0, returndatasize()) }
    }
}

与 Transparent/UUPS 的本质差异：这里不是查 EIP-1967 的单一 Implementation 地址，而是查一张 mapping(bytes4 => address) 的路由映射表。

4. 存储管理与碰撞防御 (Storage Safety)

这是 Diamond 模式最具挑战性的部分。 由于多个 Facet 共享 Diamond 的存储空间（delegatecall 的本质），任何两个 Facet 如果不小心用了相同的 Slot，就会互相覆盖对方的数据（Storage Collision）。

  • 采用的存储标准：Diamond Storage Pattern（EIP-1967 之外的另一种哈希隔离方案）。
  • 核心思路：每个 Facet 定义自己的存储结构体，并将其锚定到一个由唯一字符串哈希生成的极远端存储槽位中。
library LibMyFacet {
    // 每个 Facet 用独特的字符串生成唯一的存储起始位置
    bytes32 constant STORAGE_POSITION = keccak256("diamond.storage.MyFacet");
    
    struct Storage {
        uint256 value;
        mapping(address => uint256) balances;
    }
    
    function getStorage() internal pure returns (Storage storage s) {
        bytes32 position = STORAGE_POSITION;
        assembly {
            s.slot := position
        }
    }
}

// Facet 内部使用：
function deposit() external payable {
    LibMyFacet.Storage storage s = LibMyFacet.getStorage();
    s.balances[msg.sender] += msg.value;
}
  • 碰撞风险点：如果两个 Facet 的 Library 使用了相同的 keccak256 输入字符串（如都写了 "diamond.storage.default"），它们会共享同一个存储区域并互相覆盖。开发者必须确保每个 Facet 的字符串标识全局唯一。

5. 安全攻击面与审计要点 (Security & Attack Surface)

A. 该模式的固有攻击面

  • 💀 diamondCut 权限劫持：此函数允许通过 delegatecall 在 Diamond 的存储上执行任意代码（_init + _calldata 参数），必须是整个合约最高级别的权限保护对象。劫持 diamondCut = 完全控制合约。
  • 函数选择器冲突：正确实现的 diamondCut 应阻止添加已存在的选择器（Add 操作时先检查该 selector 是否已被注册），杜绝同一个 selector 被映射到两个不同的 Facet。
  • selfdestruct 灾难：在 Facet 中使用 selfdestruct 会毁灭整个 Diamond 主合约（因为 delegatecall 的执行上下文是 Diamond 自己）。
  • 存储冲突：多个 Facet 共享存储空间时，如果 Diamond Storage 的字符串标识不唯一，会导致跨模块数据覆盖。

B. 安全最佳实践清单

  1. diamondCut 权限管控：使用多签 / DAO / 时间锁严格限制。
  2. 禁止 selfdestruct：在所有 Facet 中禁用此操作码。
  3. Diamond Storage 隔离：每个 Facet 的存储字符串标识必须全局唯一。
  4. 链上透明性：每次 Cut 操作必须发出 DiamondCut 事件，允许外部监控所有变更历史。
  5. Loupe 自检：部署后通过 Loupe 接口验证所有 Facet 和 Selector 的注册状态。

最佳适用场景

  • 巨型 DeFi 协议：合约功能远超 24KB 限制（如同时包含 AMM、借贷、治理、质押子模块）。
  • 模块化管理需求：不同功能模块由不同团队独立开发、独立升级，互不影响。
  • Facet 复用：已部署的 Facet 可被多个 Diamond 共用，节省部署成本。
  • 可选不可变性：Diamond 可以在后期冻结 diamondCut 权限，变为不可升级。

不适用场景

  • 简单的中小型合约（架构过度工程化，引入不必要的复杂性）。
  • 团队不熟悉 Diamond Storage 模式（极易犯存储冲突错误）。
