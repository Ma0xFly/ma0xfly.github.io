代理模式核心概念

一句话总结：代理模式（Proxy Pattern）是以太坊智能合约"可升级"的底层基石，利用 delegatecall 操作码实现代码与数据的物理分离——用户永远与同一个地址（Proxy）交互，而 Proxy 背后指向的业务逻辑代码（Implementation）可以随时被替换。

📜 核心元数据与参阅资料 (Metadata & References)

信息源原则：代理模式笔记必须以 EIP 官方提案原文 和 OpenZeppelin / Solady 等权威参考实现源码为第一手真理源。

  • 关联 EIP/ERC 编号：EIP-1967（标准存储槽）、ERC-1822（UUPS）、ERC-2535（Diamond）、ERC-1167（Minimal Proxy）
  • 权威参考实现：OpenZeppelin Proxy Contracts

1. delegatecall 的魔法

delegatecall 是以太坊中一种特殊的调用方式。它的核心规则是：

在调用方（Proxy）的存储空间和上下文中，执行被调用方（Implementation）的代码。

属性 值归属
代码 (Code) 来自 Implementation
存储 (Storage) 属于 Proxy
msg.sender 保留原始外部调用者（不变）
msg.value 透传（不变）

这就是为什么数据一直在 Proxy 里，代码可以换。

2. "升级"到底在干什么？

所谓的"升级合约"，不是修改已有的代码，而是：

  1. 部署一个全新的 Implementation 合约（包含新的业务逻辑/修复了 bug/增加了新功能）。
  2. 告诉 Proxy："从现在起，你不要再去找旧的 Implementation 了，改去找这个新的。"
  3. Proxy 内部把指向的地址从 旧Implementation 改成 新Implementation。

升级前后的变化

项目 升级前 升级后
Proxy 地址 0xABC... 0xABC...（不变！）
Implementation 地址 0x111...（旧逻辑） 0x222...（新逻辑）
存储数据（余额、状态等） 在 Proxy 里 还在 Proxy 里，原封不动！
代币余额 token.balanceOf(Proxy) = 1000万 还是 1000万，没动！
用户交互的地址 0xABC... 0xABC...（用户无感）

关于代币的疑问

"换了地址，内部的 token 又怎么办？"

答案是：Proxy 的地址不会换，所以 token 不受任何影响。

代币是存储在 ERC20 代币合约内部的 balances mapping 里的，key 就是持有者的地址。因为用户和外部合约看到的地址始终是 Proxy 的地址，Proxy 地址不变，所以代币余额、授权（allowance）、NFT 持有关系等都不会受到影响。

3. Initializable — 为什么不用 constructor？

在代理模式中，Implementation 合约的 constructor 只会在部署 Implementation 合约本身时执行一次，不会在 Proxy 的上下文中执行。

因此，代理模式的初始化不能用 constructor，而是改用一个带有 initializer 修饰符的普通函数（通常叫 initialize），并且确保它只能被调用一次：

function initialize(address admin, address proposer, address sweeper) external initializer {
    __Ownable_init(msg.sender);    // 初始化 Ownable
    __UUPSUpgradeable_init();      // 初始化 UUPS
    transferOwnership(address(new ClimberTimelock(admin, proposer)));
    _setSweeper(sweeper);
}

同时，Implementation 合约的 constructor 里通常会调用 _disableInitializers() 来确保没人能在 Implementation 合约本身上调用 initialize（只允许通过 Proxy 调用）：

constructor() {
    _disableInitializers();
}

4. EIP-1967：标准化存储槽

所有主流代理模式都遵循 EIP-1967 存储槽标准，将 Proxy 的内部变量（如 Implementation 地址、Admin 地址）存放在伪随机的存储槽中：

// Implementation 地址存储槽
bytes32 constant IMPLEMENTATION_SLOT = bytes32(uint256(keccak256("eip1967.proxy.implementation")) - 1);
// = 0x360894a13ba1a3210667c828492db98dca3e2076cc3735a920a3ca505d382bbc

// Admin 地址存储槽
bytes32 constant ADMIN_SLOT = bytes32(uint256(keccak256("eip1967.proxy.admin")) - 1);
// = 0xb53127684a568b3173ae13b9f8a6016e243e63b6e8ee1178d6a717850b5d6103

// Beacon 地址存储槽
bytes32 constant BEACON_SLOT = bytes32(uint256(keccak256("eip1967.proxy.beacon")) - 1);
// = 0xa3f0ad74e5423aebfd80d3ef4346578335a9a72aeaee59ff6cb3582b35133d50

这样做是为了避免 Proxy 的内部变量与 Implementation 的状态变量发生存储碰撞。详见 EIP-1967 专题笔记。

6. 通用安全红线

  1. 存储布局一致性：升级时新的 Implementation 必须保持与旧版完全一致的存储变量布局（只能追加，不能修改或删除已有变量的顺序和类型）。
  2. _disableInitializers()：Implementation 的 constructor 中必须调用此函数，防止裸合约被人恶意初始化。
  3. 多签/DAO 管理：掌控升级权限的 Admin/Owner 角色必须使用多签钱包或 DAO 治理，避免单点失败。
  4. 升级后测试：每次升级都应通过 storageLayout 工具（如 OpenZeppelin Upgrades Plugin）对比前后存储兼容性。
