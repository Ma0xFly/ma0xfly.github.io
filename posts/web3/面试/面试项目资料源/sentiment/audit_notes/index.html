Sentiment 2023-04-04 只读重入事件审计笔记

审计范围: src/sentiment/modules/AccountManager/src/core/AccountManager.sol、src/sentiment/modules/WeightedBalancerLPOracle/src/balancer/WeightedBalancerLPOracle.sol、test/poc.t.sol、根 README.md Solidity 版本: 主要模块 0.8.17，ReentrancyGuard 为 pragma >=0.8.0

合约概览

我先按 README 给的顺序看，结论很快就冒出来了：这次不是单点 bug，而是两个模块刚好在最糟糕的时机“握手成功”。

  • AccountManager 是资金边界。用户开账户、存抵押、借款、调用外部协议，基本都从这里过。
  • WeightedBalancerLPOracle 是价格边界。它不自己维护价格，而是临时去 Balancer Vault 读池子余额，再配上底层资产价格算 BPT 价格。
  • riskEngine 是风控边界。虽然仓库里只有接口，但从 borrow() / withdraw() 的调用关系能看出来，账户健康度最终要吃预言机价格。
  • test/poc.t.sol 也把攻击节奏点出来了：先建仓，再 exitPool()，最后在 fallback 窗口里借钱。

我一开始有个疑问：AccountManager.exec() 和 borrow() 都有 nonReentrant，那 fallback 里怎么还能借？后来把攻击路径重新捋了一遍就通了:

  • 攻击者把 Sentiment 账户里的 BPT 抵押头寸先铺好。
  • 真正用来制造 Balancer 中间态的“大额加池/退池”动作，是在 Sentiment 账户外、攻击者自己控制的上下文里做的。
  • 所以进入 Balancer 的 ETH 回调窗口时，Sentiment 这边并没有正在执行中的 AccountManager 调用，自然不会撞上本地 locked == 2。

这里最值得记住的一点是：本地 nonReentrant 只能防“我自己还没跑完又进我自己”，防不了“外部协议先进入一个脏状态，我在旁边读错了它”。

顺手把 nonReentrant 本身拆开记一下，免得后面脑子里只剩一个标签：

modifier nonReentrant() virtual {
    require(locked == 1, "REENTRANCY");
    locked = 2;
    _;
    locked = 1;
}
  • 进入函数前要求 locked == 1，表示当前没有别的受保护函数在执行。
  • 一旦进来，就先把 locked 改成 2，相当于挂上“施工中，别再进”的牌子。
  • 函数结束后再改回 1。
  • 所以如果某个外部调用在中途把控制权交出去，攻击者又想重新进入 AccountManager 的另一个 nonReentrant 函数，就会因为 require(locked == 1) 失败而回滚。

这套东西对“经典重入”很好使，比如我在 withdraw() 里先转账，转账时对方 fallback 又回头调一次 withdraw()。但这次不是这个故事：这次更像是 Balancer 自己正处在一个中间态，Sentiment 去读了它的脏数据。也就是说，锁保护的是 AccountManager 的执行栈，不是 Balancer 的状态一致性。

资金在哪

正常路径大概是：

用户 -> AccountManager.deposit() -> 账户内抵押

用户 -> AccountManager.borrow() -> LToken.lendTo(account, amt) -> 账户内拿到借款

riskEngine -> oracle -> 读取抵押价值 -> 决定借/提是否允许

逐函数分析

WeightedBalancerLPOracle.getPrice()

  • 干了什么：先 vault.getPoolTokens(poolId) 读池子底层余额，再读 getNormalizedWeights()，最后用底层资产价格和 totalSupply() 算 BPT 单价。
  • 谁能调用：任何依赖它取价的上层模块，典型就是风控。
  • 我注意到：它是 view，而且完全信任 Balancer 在这一瞬间返回的 balances 和 totalSupply() 彼此一致。
  • 坑点在于：这两个值不一定来自同一个“干净快照”。一旦 Balancer 正在 exitPool()，就可能出现“分母先缩了，分子还没缩”的中间态。
  • Slither 命中: 没有直接命中这个主因。静态扫描主要给了库代码噪音和 Helpers 里的低级调用提醒，说明这类“跨协议只读重入 + 脏读”很难靠通用 detector 直接抓出来。

这里顺手补两个紧贴这个函数的 Balancer 术语，不然后面一看到 BPT / exitPool() 很容易断片：

  • BPT = Balancer Pool Token 也就是 Balancer 池子的 LP 份额 token。用户往池子里放入 WBTC/WETH/USDC 这类底层资产后，会拿到 BPT。这次 getPrice(token) 里的 token，本质上就是这个 BPT。
  • exitPool() 就是从 Balancer 池子里退出流动性。用户把 BPT 还给池子，池子把底层资产退回来。等价于“烧掉 LP 份额，赎回底层资产”。

放到这次攻击里，这两个东西的角色很明确：

  • BPT 是 Sentiment 拿来当抵押物估值的对象
  • exitPool() 是攻击者用来制造 Balancer 中间态的动作

所以这次不是直接操纵某个底层币的喂价，而是：

  1. 先持有 / 建立会被 Sentiment 认可的 BPT 抵押
  2. 再用 exitPool() 制造 Balancer 的过渡状态
  3. 让 WeightedBalancerLPOracle.getPrice(BPT) 在这个窗口里错算 BPT 价格
  4. 最后把这个假价格变成真实借款能力

我把这个函数再按语句拆一遍，后面复盘时会更好用：

function getPrice(address token) external view returns (uint) {
    (address[] memory poolTokens, uint256[] memory balances,) =
        vault.getPoolTokens(IPool(token).getPoolId());

    uint256[] memory weights = IPool(token).getNormalizedWeights();

    uint length = weights.length;
    uint temp = 1e18;
    uint invariant = 1e18;
    for (uint i; i < length; i++) {
        temp = temp.mulDown(
            (oracleFacade.getPrice(poolTokens[i]).divDown(weights[i]))
                .powDown(weights[i])
        );
        invariant = invariant.mulDown(
            (balances[i] * 10 ** (18 - IERC20(poolTokens[i]).decimals()))
                .powDown(weights[i])
        );
    }
    return invariant.mulDown(temp).divDown(IPool(token).totalSupply());
}
  • 第一步：vault.getPoolTokens(...) 拿池子底层 token 列表和余额，也就是“库存快照”。
  • 第二步：getNormalizedWeights() 拿每个 token 的权重，比如 33/33/33 池里每个权重大概是 0.333e18。
  • 第三步：循环里的 temp 在算“价格部分”。 oracleFacade.getPrice(poolTokens[i]) 先取每个底层资产价格； divDown(weights[i]) 再按权重归一； powDown(weights[i]) 最后做加权幂运算。 我脑子里会把它记成“底层资产价格的加权几何均值”。
  • 第四步：循环里的 invariant 在算“库存部分”。 balances[i] 是池子里真实余额； 10 ** (18 - decimals) 是把不同 token 精度统一到 18 位； 然后同样做按权重的幂运算。 这一段我会把它记成“池子库存规模的加权几何均值”。
  • 第五步：最后 invariant.mulDown(temp).divDown(totalSupply())，本质就是拿“整个池子的理论价值”去除以 “BPT 总供应量”，从而得到 1 个 BPT 值多少钱。

一句话版就是：

BPT 价格 = 池子底层余额的加权几何均值 * 底层资产价格的加权几何均值 / BPT总供应量

这个写法平时没问题，坑就坑在它默认假设 balances 和 totalSupply() 来自同一个干净快照。但在 exitPool() 中间态里，可能出现：

  • totalSupply() 已经变小
  • balances 还没同步扣减

于是最后这一除就炸了：

  • 分子偏大
  • 分母偏小
  • 算出来的 BPT 价格被高估

这两个关键值还不是从同一个地方读的，我得单独记一下来源：

  • balances 来自 vault.getPoolTokens(IPool(token).getPoolId()) 也就是先拿 BPT 对应的 poolId，再去 Balancer Vault 读池子底层 token 余额。
  • totalSupply() 来自 IPool(token).totalSupply() 也就是直接把这个 token 当成 BPT 合约本身，去读它当前的总供应量。

换句话说：

  • balances 是 Vault 侧 的读数
  • totalSupply() 是 BPT 合约侧 的读数

正因为这两个值不是同一个接口一次性返回的，而是分开从两个地方读，所以只要 Balancer 在 exitPool() 过程中没有把这两个面向外部的状态同时更新，就会出现短暂不同步。这里也是这次事故最阴的点：公式没毛病，问题出在它拼接的是两个不同步的数据源。

我在本地 fork 上顺手做了一个很小的验证，也帮我把“正常变化”和“漏洞窗口”区分开了：

  • priceBefore：220120691697985632
  • priceDuring：221005560649599490
  • priceAfter：220122172406069243

这里有个细节值得记：

  • priceDuring > priceBefore 才是只读重入窗口的核心证据，因为这是回调中读到的异常高价。
  • priceAfter 比 priceBefore 略高一点不奇怪，这不一定说明漏洞还在持续，而更像是“真实退池后的新平衡价格 + 单币退出带来的池子组成变化 + 舍入误差”。

也就是说：

  • before -> after 的小幅变化，可以是正常状态更新
  • before -> during 的明显抬升，才是中间态脏读

所以我会把这个函数的风险标签记成：不是数学公式错了，而是它把外部协议的瞬时读数误当成了原子快照。

AccountManager.borrow()

  • 干了什么：检查目标资产是否有 LToken，必要时把借款 token 记到账户资产列表里，随后直接 lendTo(account, amt)，最后才调 riskEngine.isAccountHealthy(account)。
  • 谁能调用：账户 owner。
  • 我注意到：这里的风控校验是“借完再看健不健康”，而不是“先算这笔借款能不能借，再放款”。
  • 我也专门对着 nonReentrant 想了一遍：如果攻击者是在某个还没结束的 AccountManager.exec() 里重进 borrow()，这把锁会挡住；但如果 Balancer 的 fallback 窗口发生在 Sentiment 外部上下文里，那 borrow() 还是能正常进。
  • 这个顺序单独看不一定立刻出事，因为健康检查失败会整笔 revert；但一旦健康检查吃到的是被操纵的价格，它就会把假的抵押价值兑换成真的借出资产。
  • Slither 命中: 也没有直接命中这里的组合风险。工具没法从单个函数里推断“外部预言机此刻可能正在读脏状态”。

这里的 LToken 我也顺手记一下，不然第一次看很容易把它当成普通 ERC20：

  • LToken 可以理解成“某个可借资产对应的借贷池合约”。
  • 用户要借 USDC，AccountManager 就先去 registry.LTokenFor(USDC) 找到对应的 USDC LToken。
  • 然后调用 lendTo(account, amt)，由这个 LToken 把资产借给账户。
  • 还款时也是回到 LToken.collectFrom(...)，借款余额则通过 getBorrowBalance(...) 这类接口来算。

所以我脑子里会把它当成：

  • 不是用户账户
  • 不是抵押物 token
  • 而是“某个底层资产的放贷池 + 借款记账入口”

如果类比别的协议，它扮演的角色有点像“每个资产各自一套借贷市场的池子合约”。

AccountManager.exec()

  • 干了什么：允许账户 owner 通过受控 target 和 calldata 代表账户跟外部协议交互。
  • 谁能调用：账户 owner。
  • 我注意到：它的 nonReentrant 只能保护 AccountManager 自己的执行栈。换句话说，它能拦“我还没跑完，你又回来调我”，但拦不住“我去读的那个外部协议，自己现在正处于脏状态”。
  • 这里有意思的是：攻击者确实需要 exec() 这类能力去铺 Sentiment 账户内的 BPT 头寸，但真正制造只读重入窗口的关键 exitPool() 不一定非要在这条持锁路径里做。

AccountManager.withdraw()

  • 干了什么：先 isWithdrawAllowed()，再把 token 从账户提给 owner。
  • 谁能调用：账户 owner。
  • 我注意到：它和 borrow() 一样，安全性本质上也绑定在 riskEngine 对抵押价格的判断上。价格一旦是假的，提款门槛也会一起变假。

🔍 发现的问题

问题1: Balancer BPT 预言机在 exitPool() 中间态读取到不一致快照

  • 严重程度: Critical
  • 怎么发现的: 我先从 README 里的“先放款、后健康检查”往回找，发现真正能把账户健康度抬上去的只有预言机。再看 WeightedBalancerLPOracle.getPrice()，它一边读 Vault.getPoolTokens()，一边读 IPool(token).totalSupply()，马上就警觉了：如果 Balancer 在某个流程里这两个值不是同时更新，BPT 价格就会飘。官方复盘也明确点名了 exitPool() 的 ETH 回调窗口正好会制造这个中间态。
  • 问题本质: 预言机把 Balancer 的瞬时状态当成原子快照使用，但 exitPool() 期间实际上会出现“totalSupply() 已减少、balances 尚未减少”的不一致窗口，导致 BPT 价格被严重高估。
  • Slither 命中: 无直接命中。这次核心问题是跨协议读脏状态，不是本地合约里传统的写路径重入。
  • 问题代码:
function getPrice(address token) external view returns (uint) {
    (address[] memory poolTokens, uint256[] memory balances,) =
        vault.getPoolTokens(IPool(token).getPoolId());

    uint256[] memory weights = IPool(token).getNormalizedWeights();

    for (uint i; i < weights.length; i++) {
        temp = temp.mulDown(
            (oracleFacade.getPrice(poolTokens[i]).divDown(weights[i]))
                .powDown(weights[i])
        );
        invariant = invariant.mulDown(
            (balances[i] * 10 ** (18 - IERC20(poolTokens[i]).decimals()))
                .powDown(weights[i])
        );
    }

    return invariant.mulDown(temp).divDown(IPool(token).totalSupply());
}
  • 攻击叙事: 我是攻击者。我先用闪电贷准备大笔 WBTC/WETH/USDC，再在 Sentiment 账户里留一份 BPT 抵押。接着我在 Balancer 池外面做一笔大额 exitPool()，并且故意走 ETH 这条退出路径。原因很简单：Balancer 如果退给我的是普通 ERC20，标准 transfer 不会把执行权交还给我；但如果退给我的是原生 ETH，它会对我的合约做一次 value call，于是我的 receive()/fallback() 就能在 exitPool() 尚未走完时执行。此时池子的 BPT 已经烧掉一部分，所以 totalSupply() 变小了，但 Vault.getPoolTokens() 看到的底层余额还没扣掉。我就在这个窗口里让 Sentiment 去查 BPT 价格。结果分子偏大、分母偏小，BPT 价格直接被吹上去，后面的风控检查就形同虚设。
  • 代码位置: src/sentiment/modules/WeightedBalancerLPOracle/src/balancer/WeightedBalancerLPOracle.sol:43
  • 修复思路: 不要把 Balancer view 读数当成天然安全。至少要按 Balancer 官方建议，在取价前做一次 no-op 的 manageUserBalance([])/等价 reentrancy check；更稳一点是把这种可重入状态下的 spot 价格移出借贷风控主路径，改用有边界条件的缓存价、TWAP、或显式暂停借贷。

问题2: borrow() 直接把可操纵的抵押估值兑现成真实放款

  • 严重程度: High
  • 怎么发现的: 在确认预言机会高估 BPT 之后，我回头看资金边界，最想问的一句就是“谁先拿钱？”答案是 borrow() 里先 lendTo(account, amt)，再 isAccountHealthy(account)。也就是说，这里不是先算额度，而是先把钱推进账户，再把生死交给风控判断。
  • 问题本质: 借款路径没有把“预言机可信”当成前置条件，而是直接用风控结果为已经发生的放款背书；只要健康检查恰好吃到脏价格，协议就会把虚高抵押价值立刻兑换成真资产。
  • Slither 命中: 无直接命中。它看不到这里依赖的是一个可被外部中间态污染的 oracle 路径。
  • 问题代码:
function borrow(address account, address token, uint256 amt)
    external
    nonReentrant
    whenNotPaused
    onlyOwner(account)
{
    if (registry.LTokenFor(token) == address(0))
        revert Errors.LTokenUnavailable();

    if (IAccount(account).hasAsset(token) == false) {
        if (IAccount(account).getAssets().length > assetCap)
            revert Errors.MaxAssetCap();
        IAccount(account).addAsset(token);
    }

    if (ILToken(registry.LTokenFor(token)).lendTo(account, amt))
        IAccount(account).addBorrow(token);

    if (!riskEngine.isAccountHealthy(account))
        revert Errors.RiskThresholdBreached();
}
  • 攻击叙事: 我在 Balancer 的回调窗口里，不需要碰 Sentiment 自己的重入锁，只要直接调用 borrow() 就行。此时 riskEngine 看到的是被高估十几倍的 BPT 抵押，所以我可以连续借出 USDC / USDT / WETH / FRAX。这些都是真资产，不是账面数字。等窗口结束、价格恢复正常，坏账已经铸成了。
  • 代码位置: src/sentiment/modules/AccountManager/src/core/AccountManager.sol:224
  • 修复思路: 借款前先做额度计算和价格安全校验，不要让 lendTo() 成为风控前置检查的一部分。更具体一点，可以把“借后健康度”预演成纯检查逻辑，或者对特定 LP 抵押启用单独的 oracle freshness / reentrancy guard / borrow cooldown。

💡 收获与心得

  • view 不是安全词。只要它读的是外部协议的中间态，它一样可以成为攻击入口。
  • 这里的“中间态”我给自己下个直白定义：一次操作还没完整做完时，合约短暂处在的过渡状态。也就是本来应该一起保持一致的几个量，在执行过程中不是同一瞬间更新，于是外部刚好在半路读取时，就会读到“半新半旧”的假快照。
  • 放到这次 Balancer 场景里，就是 exitPool() 过程中，totalSupply() 和 Vault 里的 balances 没有在同一个时刻对外表现为一致。于是 WeightedBalancerLPOracle.getPrice() 读到的不是最终状态，而是中途状态，这就是这次只读重入能成立的基础。
  • 本地 nonReentrant 很容易给人安全错觉，但它只覆盖当前合约自己的调用栈，不覆盖依赖协议的“脏读”问题。
  • 借贷协议最怕的是把“可操纵的价格”直接连到“真实资产转出”。一旦中间没有缓冲层，任何短暂价格错乱都会被放大成真实亏损。
  • Slither 这次更多是帮我确认“常规坑没有特别亮”，但主问题还是得靠手工把跨协议状态机拼起来看。
  • 这份仓库里的 test/poc.t.sol 目前还是骨架，不过它把关键节奏都留好了：Phase 2 建仓，Phase 3 在 fallback 窗口借款。后面真要补 PoC，重点就该放在还原这两个阶段之间的最小调用链。
