MemeScan 面试辅导资料（贴当前代码版）

先记住这句话

这不是一个“纯 LLM 审计工具”，而是一个以链上仿真为主、轻量规则系统评分为骨架、LLM 做补充解释和追加分析的 Web3 风险扫描系统。

面试时最重要的是两点：

  1. 不要把设计稿里的能力当成全部都已经落地。
  2. 要把“为什么这样设计”讲出来，而不只是背功能列表。

一句话定位

MemeScan 是一个面向新发 Meme 代币的安全扫描系统。它会监听新交易对，拉起本地 Anvil 分叉环境，真实模拟买入和卖出，再结合轻量规则系统和源码分析生成风险报告。

30 秒自我介绍

我做了一个叫 MemeScan 的 Web3 安全项目，主要解决新发 Meme 币的蜜罐和高风险权限问题。它不是只看源码，而是会在本地分叉链里真实模拟买卖交易，验证能不能卖出、税率是多少；如果需要，再结合源码和 LLM 做进一步解释，最后生成结构化审计报告。

2 分钟项目介绍

MemeScan 的目标是解决一个很实际的问题：新发 Meme 币上线快、风险高，普通用户很难在短时间判断是不是蜜罐，或者合约里有没有项目方随时作恶的权限。

我这个项目的核心思路不是只做静态分析，而是做“动态验证 + 规则判断 + AI 辅助解释”。

具体流程是这样的：

  1. 监控层监听 DEX 工厂合约的 PairCreated 事件，发现新的代币交易对。
  2. 协调层收到代币后，会先补齐代币元数据，比如 name、symbol、decimals、owner。
  3. 然后进入沙盒仿真：我会启动 Foundry 的 Anvil，分叉主网状态，在本地用 cast send 真实执行买入、approve、卖出，计算买卖税率，判断是不是蜜罐。
  4. 如果基础仿真通过，还会尝试做一轮“Rug-Pull Replay”，也就是在沙盒里模拟项目方身份，尝试黑名单、改税率、暂停交易之类的常见恶意操作，看能不能把正常代币打成不能卖。
  5. 审计层会把仿真结果喂给轻量规则系统，打风险标签和风险分；在有源码和 LLM 配置时会生成解释摘要，如果命中了某些不确定的高风险场景，还会再追加一轮更深的源码分析。
  6. 最后报告层把结果输出成 Markdown 报告，前端用 Streamlit 展示，支持查看实时报告、历史报告，以及基于最新报告做问答。

我觉得这个项目最有价值的地方，是把“能不能卖”这种用户最关心的问题，从静态猜测变成了动态验证。

当前项目的真实架构

Agent / Service 分层怎么讲

我会把这个项目描述成“Agent 编排 + Service 落地执行”。

  • ScannerAgent 封装 MonitorService，负责监听PairCreated(...)事件，发现新交易对
  • CoordinatorAgent 负责编排流程
  • SandboxAgent 封装 SimulationService，负责链上仿真和攻击重放
  • AuditorAgent 封装 AnalysisService 和 EtherscanService
  • ReporterAgent 负责报告生成

但是要注意，面试时不要把它吹成很重的 autonomous agent system。当前实现本质上还是明确流程编排，Agent 之间通过函数调用和结构化字典传参协作，不是消息队列，也不是完全自治系统。

如果面试官追问“那 Agent 之间到底怎么通信”，你不要只回答一句“函数调用”，建议展开成下面这版：

当前项目里，Agent 之间的通信方式本质上是：

  • 同进程内的异步方法调用
  • 通过结构化字典传递上下文
  • 返回结构化结果给上游 Agent

也就是说，它不是：

  • Kafka / RabbitMQ 这种消息队列
  • Event Bus 广播
  • 多进程 Actor 模型
  • Agent 自己互发自然语言消息

而是一个更轻量的、工程上更可控的通信模型。

你可以直接举主链路里的例子：

  • CoordinatorAgent 收到 {"token": token}
  • 调用 SandboxAgent.run({"token": token})
  • SandboxAgent 返回 {"simulation": SimulationResult, "all_results": ...}
  • CoordinatorAgent 再把结果组装成 {"token": token, "simulation": simulation} 传给 AuditorAgent
  • AuditorAgent 返回 {"report": AuditReport, "decision": "...", "llm_analysis": ...}
  • 最后 CoordinatorAgent 再把 {"report": report} 交给 ReporterAgent

这套通信的核心不是“聊天”，而是数据契约。
每个 Agent 关心自己的输入和输出字段，上游 Agent 负责组装上下文，下游 Agent 负责返回结构化结果。

你还可以继续往下讲：

为什么这里不用消息队列

因为当前系统的主流程是强顺序依赖的：

  1. 先仿真
  2. 再审计
  3. 最后生成报告

后一步明确依赖前一步结果，所以没有必要为了“看起来高级”引入 MQ。
如果过早引入 Kafka/RabbitMQ，反而会带来：

  • 运维复杂度
  • 消息幂等问题
  • 重试和死信处理
  • 调试链路变长

对于当前这个单机工具型系统，同进程异步调用已经足够。

这种通信方式的优点

  • 简单直接：调用链清晰，调试方便
  • 类型边界明确：虽然用的是 dict，但里面承载的是 Token、SimulationResult、AuditReport 这类结构化对象
  • 上下文组装灵活：Coordinator 可以决定给下游 Agent 传哪些字段
  • 适合当前规模：不需要额外基础设施就能跑通完整审计链路

这种通信方式的边界

  • Agent 之间是紧耦合调用，不适合跨进程横向扩展
  • 没有天然的持久化消息队列，所以失败恢复能力有限
  • 当前并不是“分布式多 Agent 系统”，而是“单进程编排式多 Agent”

这句你可以直接背：

“我这里的 Agent 通信不是 MQ，也不是 LLM 对话式消息传递，而是 Coordinator 主导的异步函数调用加结构化上下文传参。它更像一条可追踪的编排流水线，而不是分布式自治系统。”

如果面试官再追问“那为什么还叫 Agent”，你可以这样答：

“因为每个模块不只是 service 封装，它还有自己的任务边界和局部决策能力，比如 Sandbox 会决定是否值得继续做攻击重放，Auditor 会决定是否追加深度分析。只是当前这套系统的通信机制仍然是工程上更轻量的函数编排，而不是复杂的消息总线。”

真实审计链路

当前最核心的一条链路是：

Coordinator -> Sandbox -> Auditor -> Reporter

其中：

  • ScannerAgent 只在实时监控模式下参与“发现代币”
  • 手动扫描时，用户输入地址后会直接进入 CoordinatorAgent
  • CoordinatorAgent 自己的 decide() 目前只是占位，真正的“是否追加深度分析”决策在 AuditorAgent.decide()

前端怎么讲

前端是 Streamlit，不是前后端完全分离架构。因为 Streamlit 本身是同步渲染，所以我用了一个后台线程跑独立的 asyncio event loop，把监控、仿真、LLM 调用这些异步任务提交到后台执行，再把结果同步回 session_state。

这套设计的重点不是“高并发”，而是在 Streamlit 约束下保持 UI 可用，同时能跑异步链上任务。

你可以重点强调的亮点

1. 动态仿真而不是只看源码

很多安全工具只是扫 ABI 或源码特征，但 MemeScan 会在 Anvil 分叉里真实走一遍：

  • 买入
  • 查询实际收到的 token 数量
  • approve
  • 卖出
  • 对比预期和实际结果，算税率

这个能力能直接回答用户最关心的问题：这个币到底能不能卖。

2. Rug-Pull Replay

如果基础仿真没发现问题，系统还会尝试模拟项目方权限：

  • 读取 owner()
  • 在本地分叉链中 impersonate 该账户
  • 尝试调用常见恶意函数，比如黑名单、改税、暂停交易
  • 攻击后再次仿真买卖

如果攻击后卖不出去，说明这个合约不是“现在安全”，而是“项目方还没动手”。这个视角比单纯看当前状态更有说服力。

3. 轻量规则系统和 LLM 不是替代关系

我的设计不是让 LLM 决定一切，而是分层：

  • 轻量规则系统负责稳定、便宜、可解释的基础判断
  • LLM 负责看源码、做语义解释、降低误报

这样做的原因是：链上风险判断里，确定性的执行结果应该优先于自然语言推理。

但这里有个面试口径一定要收紧：
当前实现更准确地说是“规则集合 / 轻量规则系统”，还不算成熟的规则引擎。

为什么这么说？因为现在代码里主要是：

  • 一组独立的 if-else 规则负责打标签
  • 一张权重表负责把标签汇总成分数
  • 一个 is_dangerous 快捷判断负责把关键标签提到更高优先级
  • 一个 UNKNOWN_RISK -> 追加 LLM 的 fallback 机制负责兜底不确定场景

但它还缺少成熟规则引擎常见的能力：

  • 规则隔离和规则编排机制
  • 显式的优先级 / 冲突消解策略
  • 更系统的 fallback / confidence / uncertainty 表达

所以面试时不要说：

“我实现了一个完整规则引擎。”

而要说：

“当前版本实现的是一个轻量的 rule-based scoring system。它已经有独立规则、标签汇总、权重评分和不确定场景的 LLM fallback，但还没抽象成完整的规则引擎框架。”

4. Streamlit 场景下的异步工程化处理

项目不是学术 demo，而是一个能跑的工具。为了让 Streamlit 这种同步框架承载异步链上扫描，我做了：

  • 后台线程 + 独立事件循环
  • 模块级共享列表传递结果和日志
  • 启动时从磁盘报告恢复历史数据
  • 自动刷新实时报告

这部分很适合体现工程意识。

5. 安全岗一定要讲“状态、断言、失败条件”

这一点非常关键。面试官如果问你“auditor怎么判断风险”，你不能回答“AI 觉得危险”或者“规则打分出来的”。
安全分析的表达必须落到三个层面：

  • 状态是什么
  • 断言是什么
  • 失败条件是什么

你可以把 MemeScan 的判断逻辑概括成：

“我不是先让 AI 给结论，而是先采集一组可验证状态，再对这些状态做确定性断言，最后才把断言结果映射成风险标签和分数。LLM 只负责补充解释，不负责定义真值。”

这里最值得你背熟的是下面这张表：

场景 读取的状态 核心断言 失败条件 / 风险结论
买入检测 buy_receipt.success、actual_tokens 买入交易必须成功，且实际到账 token > 0 买入 revert 或到账为 0，说明异常，至少不是正常可交易代币
卖出检测 sell_receipt.success 如果买得进，就应该卖得出 买得进但卖不出，是蜜罐/卖出受限的重要证据
买入税 expected_tokens vs actual_tokens 实际到账不应显著低于理论报价 差值过大，说明存在高买入税或隐藏扣费
卖出税 expected_eth vs actual_eth_received 实际收到 ETH 不应显著低于理论报价 差值过大，说明存在高卖出税或隐藏扣费
ownership owner() 调用结果 如果 owner 仍存在，说明权限未放弃 标记 OWNERSHIP_NOT_RENOUNCED
mint 权限 字节码是否包含 mint selector，且 owner 未放弃 有增发能力且权限集中，需要提高警惕 标记 HIDDEN_MINT
Rug-Pull Replay owner 提权后恶意调用是否成功；攻击后二次卖出结果 如果项目方一旦行权就能阻断卖出，说明后门可利用 标记为重放成功，风险等级显著提高

这套表达比“规则系统判断”更专业，因为它告诉面试官：
你知道自己在验证什么，不是在堆关键词。

你可以直接背这一句：

“我的审计核心不是主观判断，而是状态驱动的断言。先采集交易执行状态、余额变化、owner 状态和字节码特征，再用确定性规则断言哪些安全性质被破坏，比如‘可买不可卖’、‘税率显著偏高’、‘owner 未放弃且具备 mint 能力’。评分只是这些断言的汇总，不是判断依据本身。”

6. 现象和根因一定要分开讲

这是安全面试里另一个高频坑。
“卖不出去”只是现象，不是根因。
真正成熟的回答应该分两层：

  1. 先证明现象成立
  2. 再说明根因定位到什么粒度

你要避免这种说法：

  • “不能卖，所以高风险”
  • “像黑名单”
  • “可能被限制了”

你应该改成这种结构：

  • 现象层：我观测到什么
  • 证据层：我拿到了什么可复现证据
  • 根因层：我能定位到什么程度
  • 边界层：哪些根因当前系统还不能唯一证明

你可以直接背这一句：

“我会先把‘卖出失败’当成运行时现象，再去找根因证据。当前系统对现象判断是强确定性的，因为它来自真实仿真；对根因定位是分层的，能证明到哪一层，我就只说到哪一层，不会把推测当结论。”

在你这个项目里，根因证据主要来自 4 类：

  • revert_reason 如果卖出交易直接 revert，系统会记录 revert reason。这能帮助判断是 TRANSFER_FROM_FAILED、权限不足，还是其他执行失败。

  • 税率异常 如果交易不 revert，但实际到账远低于理论报价，说明更接近“高税/隐藏扣费”这种根因，而不是单纯的硬性禁卖。

  • ownership / mint / 权限状态 如果 owner 没放弃、且存在增发或管理权限，这能说明“项目方具备干预能力”，但还不能单独证明就是哪一个函数在导致 sell fail。

  • Rug-Pull Replay 命中的 payload 如果我 impersonate owner 后，调用 setBlacklist、setTax、pauseTrading 这类函数，随后卖出立刻失败，那我就能把根因从“卖不出去”进一步收敛到“某类 owner 权限可导致卖出失败”。

最重要的表达是：

  • 我能强证明的：卖出失败、高税、owner 权限存在、某个 payload 触发后用户无法卖出
  • 我不能强证明的：一定是 transfer 里的哪一行代码、一定是 blacklist 还是 pair 锁仓，除非 revert reason 或重放结果直接给到证据

如果面试官直接问你：

“所以你实际代码里有做根因定位这一层吗？”

你建议这样回答：

“有，但只是部分根因定位，还不是完整的 root-cause engine。
当前版本已经能区分失败发生在 buy、approve 还是 sell 阶段，也会保留 revert reason；
另外我还能通过理论报价和实际到账差值识别高税/隐藏扣费，通过 owner 状态和 Rug-Pull Replay 识别可利用的权限后门。
但我现在还做不到对所有卖出失败场景，唯一定位到源码里的具体函数和具体状态变量，所以这部分我会诚实地说是部分归因，不会说成完整根因定位。”

已经做到的根因定位：

  • 能区分买入失败、approve 失败、卖出失败，说明问题发生在哪个执行阶段
  • 能记录 revert_reason，作为直接错误证据
  • 能通过理论值和实际值差异识别高税/隐藏扣费这一类根因方向
  • 能通过 owner 状态和攻击重放，把根因收敛到“某类 owner 权限可导致卖出失败”

还没做到的根因定位：

  • 不能稳定唯一判断一定是 blacklist、router 限制、pair 状态异常还是 transfer 里的某一条 require
  • 没有做源码级控制流分析，也没有做系统化的 root-cause taxonomy
  • Auditor 当前更偏规则映射和评分，不是一个完整的根因分析器

这类回答在面试里是加分的，因为它说明你知道：

  • 哪些是动态执行能证明的
  • 哪些是归因推断
  • 哪些目前还需要更强的程序分析能力

高频面试题

Q1：为什么做这个项目？

建议回答：

Meme 币市场上线快、生命周期短，很多风险发生在用户还没来得及查资料之前。传统审计报告覆盖不到这些新币，而纯静态规则又很容易漏掉运行时行为。所以我想做一个更贴近真实交易场景的系统，用本地分叉仿真去验证“用户买了之后到底能不能卖”，再补充源码和权限层面的分析。

Q2：为什么要做成 Multi-Agent，而不是一个大函数？

建议回答：

我这里的 Multi-Agent 不是为了追求概念，而是为了把职责拆开。监控、仿真、审计、报告这几块变化频率和复杂度不同，拆成 Agent 以后更容易替换和扩展。比如审计层可以换规则、换 LLM，报告层可以换展示方式，而不会动到仿真逻辑。

但我也会坦诚，当前实现还是偏“编排式 Agent”，不是完全自治系统。主要收益是模块边界清晰，而不是 Agent 自己相互谈判。

Q3：Coordinator 真正做了什么？

建议回答：

Coordinator 现在主要做三件事：

  1. 收到 token 后补齐元数据
  2. 串行调用 Sandbox、Auditor、Reporter
  3. 记录决策链，最终交给前端展示

它目前不是最强的决策中心。真正“要不要追加深度分析”的规则，写在 AuditorAgent.decide() 里。这个我会在面试里主动讲清楚，因为这体现了我对自己代码边界是清楚的。

Q4：沙盒仿真为什么用 Anvil 分叉？

建议回答：

因为我要的是“接近真实主网状态，但又绝对不上链”。Anvil 用 --fork-url 启动后，会把主网状态映射到本地 EVM 环境里。这样我既能读取真实流动性池和合约状态，又可以在本地随便试买卖、试攻击，不会动真实资产。

Q5：你的蜜罐检测怎么做？

建议回答：

不是只看某个函数签名，而是完整模拟一次买卖链路：

  1. 先用 Router 的 getAmountsOut 估算预期输出
  2. 用 swapExactETHForTokens 真买
  3. 查 token 余额，算买入税
  4. 调 approve
  5. 再用 swapExactTokensForETH 卖出
  6. 对比卖出前后 ETH 余额，算卖出税

如果买得进但卖不出，或者税率异常高，就会标记高风险或蜜罐。

你也可以把这段再展开一点讲：

  • getAmountsOut 这是 Uniswap Router 的只读报价函数，用来问 AMM 池子：“如果我拿这么多 ETH 去换 token，理论上能换到多少？”
    它给的是按池子储备量算出来的理论值，可以当作基准线。

  • swapExactETHForTokens 这一步才是真正执行买入。也就是用户把 ETH 打进 Router，Router 再沿着交易路径把 token 换回来。
    如果这一步直接 revert，说明连买入都不正常，风险已经很高。

  • 查 balanceOf 买完以后不能只看 Router 的理论报价，还要看钱包里实际到账多少 token。
    如果理论上该拿 100 万枚，实际只拿到 80 万枚，那中间差的 20% 往往就是买入税或者隐藏扣费。

  • approve 卖出前必须先授权。因为卖出 token 的时候，真正帮你把 token 转进池子的是 Router，它本质上是在代表你花这笔 token。
    按 ERC-20 标准，合约不能直接动你的币，除非你先 approve(router, amount)。
    所以没有这一步，后面的卖出通常会因为 transferFrom 权限不足而失败。

  • swapExactTokensForETH 这一步是真正执行卖出，也就是把 token 换回 ETH。
    如果买得进但这里卖不出去，这就是最典型的蜜罐特征之一。

  • 对比卖出前后 ETH 余额 卖出成功后，再看钱包里实际多回来了多少 ETH，然后和 getAmountsOut 的理论值对比。
    如果理论上该拿回 1 ETH，结果实际只回来 0.6 ETH，那可以估算卖出税大概是 40%。

面试时你可以用一句更口语的话概括：

“本质上就是先拿 DEX 理论报价做参照，再执行一遍真实买卖，看理论值和实际到账之间差了多少，这个差值就是税率和隐藏扣费最直观的证据。”

如果面试官继续追问“这里的断言是什么”，你要把话讲成这样：

  • 我先记录交易执行状态：买入是否成功、卖出是否成功
  • 再记录资产状态：钱包实际收到多少 token、卖出后实际回多少 ETH
  • 再做断言：
    • 断言 1：正常代币应该可买且可卖
    • 断言 2：实际到账不应该明显低于 DEX 理论报价
    • 断言 3：如果买得进但卖不出，这是强风险信号，不是“感觉不对”

所以我不是“看起来像蜜罐”，而是基于一组执行后状态做确定性判断。

如果面试官再追问“那你怎么区分现象和根因”，你要这样答：

  • can_buy=True 且 can_sell=False，这是现象层结论
  • revert_reason 能告诉我交易是在什么阶段失败，这是直接根因证据
  • 如果没有明确 revert reason，但卖出税极高，那我更偏向“高税/隐藏扣费”这一类根因
  • 如果 owner 提权后执行 setBlacklist 或 pauseTrading 能稳定复现卖不出，那我就能把根因收敛到“owner 可利用的权限后门”

所以更严谨的表达不是：

“不能卖，所以就是黑名单。”

而是：

“不能卖是现象；黑名单、暂停交易、高税、授权限制等是候选根因。当前系统通过 revert reason、税率差值和攻击重放，对其中一部分根因做进一步归因。”

Q6：为什么要做 Rug-Pull Replay？

建议回答：

因为有些币上线初期是“能买能卖”的，但项目方保留了关键权限，等用户进场后再拉黑、改税、暂停交易。只检测当前状态会漏掉这类风险。所以我会在本地分叉链里 impersonate owner，主动尝试几种典型攻击，再重新模拟卖出。这相当于把“静态权限存在”进一步验证成“能不能真的作恶”。

Q7：LLM 在这里到底起什么作用？

建议回答：

LLM 在这个项目里主要做两类事：

  1. 给报告生成更易读的解释摘要
  2. 在某些高风险但不够确定的场景下，结合源码做进一步判断

当前实现里，如果配置了 API Key 且拿到了源码，标准分析阶段也可能先调用一次 LLM 生成摘要；如果命中特定条件，还会再追加一次更深的分析。我不会把它当最终裁判，真正高优先级的证据还是仿真结果和链上规则判断。

这里还有一个很容易被问到的点：LLM 分析的是目标代币合约源码，不是 Uniswap Router。

更准确地说：

  • 交易执行层面，我是通过 Uniswap V2 Router 去做买卖仿真
  • 但源码分析层面，拉取的是 token.address 对应的 ERC-20 合约源码
  • 如果该币没有完成区块浏览器源码验证，LLM 就拿不到可读源码，也不会去分析 Uniswap，而是根据仿真结果和风险标签做补充说明

这里你还可以补一句更专业的解释：

  • 区块链上公开的是合约字节码，不等于公开了 Solidity 源码
  • 如果项目方没有在 Etherscan/BscScan 提交并验证源码，系统拿到的主要还是 bytecode，而不是可直接阅读的源码文件
  • 只有字节码时可以做逆向分析，但不能稳定、唯一地还原出原始源码，因为编译过程会丢失变量名、注释、文件结构、继承写法，以及编译器和优化参数信息

所以面试时更严谨的说法不是“这个合约没开源”，而是：

“这个合约没有完成源码验证，所以我拿不到已验证的可读源码，只能更多依赖链上行为和规则判断。”

Q7.5：Auditor 怎么判断？有没有断言？

建议回答：

有，而且我会把 Auditor 讲成一个状态到断言的映射器，而不是一个“AI 给分器”。

Auditor 的输入主要有两类状态：

  1. 来自 Sandbox 的运行时状态

    • can_buy
    • can_sell
    • buy_tax_pct
    • sell_tax_pct
    • buy_gas
    • is_honeypot
    • revert_reason

  2. 来自链上读取的静态状态

    • owner() 是否为零地址，判断 ownership 是否已放弃
    • 字节码里是否出现 mint(address,uint256) 的 selector，判断是否存在增发能力

然后 Auditor 按确定性规则做断言：

  • 如果 is_honeypot=True，断言该币具备蜜罐特征
  • 如果 can_buy=True 且 can_sell=False，断言存在卖出受限
  • 如果 buy_tax_pct 或 sell_tax_pct 高于阈值，断言存在高税风险
  • 如果 ownership 没放弃，断言存在中心化权限
  • 如果同时“ownership 没放弃 + 存在 mint 能力”，断言存在高风险增发权限

最后才会做两件事：

  • 把这些断言映射成 RiskFlag
  • 再根据标签权重算一个风险分数

所以分数不是判断来源，断言才是判断来源。

你可以直接用这句回答面试官：

“Auditor 不是凭感觉判断，也不是让 AI 直接下结论。它先吃进一组可观测状态，再做确定性断言，比如‘买得进但卖不出’、‘税率高于阈值’、‘owner 未放弃且存在 mint 能力’。风险标签是断言结果，风险分只是对这些断言做加权汇总。”

如果面试官再问“失败条件是什么”，你可以继续答：

“失败条件就是断言被破坏时的具体表现，比如买入 revert、卖出 revert、到账为 0、税率异常高、owner 未放弃、攻击重放后无法卖出。这些都是可复现实验结果，不是主观印象。”

Q7.6：这算规则引擎吗？

建议回答：

严格说，当前版本还不算成熟的规则引擎，更准确地说是一个轻量规则系统 / rule set。

它现在已经有的能力是：

  • 每条规则相对独立地从状态中产出 RiskFlag
  • 用权重表把多个标签汇总成风险分
  • 用 is_dangerous 和关键标签集合做一层高危优先判断
  • 用 UNKNOWN_RISK 和 AuditorAgent.decide() 触发更深的 LLM 分析，作为不确定场景的 fallback

但它没有完全做到的部分也要主动说清楚：

  • 规则还没有被抽象成独立对象或 DSL，本质上还是 if-else
  • 没有显式的规则优先级图和冲突消解框架
  • 没有正式的 confidence score，只能通过 UNKNOWN_RISK + 追加分析 这种方式表达不确定性

所以我在面试里不会把它吹成 full rule engine。
我会说：

“当前是一个轻量的 rule-based analyzer，已经具备规则判定、标签汇总、权重评分和 fallback 分析的雏形，但还没演进到可配置、可组合、带显式优先级和置信度管理的完整规则引擎。”

如果面试官继续问“那高危和低危同时命中怎么办”，你可以这样答：

“当前实现的处理方式比较直接：先保留所有命中的标签，再通过权重表汇总成分数；另外在报告层有一个 is_dangerous 快捷判断，把 HONEYPOT、CANNOT_SELL、HIDDEN_MINT 这类关键标签视作更高优先级。也就是说，现在是‘保留全部标签 + 关键标签优先解释’，还不是正式的冲突消解系统。”

如果面试官问“不确定怎么办”，你可以这样答：

“当前没有单独的 confidence 字段，但有一个轻量 fallback 机制：如果命中了 UNKNOWN_RISK，或者一些高风险标签但证据还不够强，就会追加源码和 LLM 分析，把纯规则判断提升成‘规则 + 解释性分析’。所以我更愿意把它叫 uncertainty fallback，而不是置信度引擎。”

如果面试官继续压你：“那你能不能说清是哪个函数、哪个状态导致的？”
你不要硬扛，也不要虚化，要诚实地这样答：

“当前版本能稳定证明的是现象和部分根因，不是所有场景都能唯一定位到源码中的具体函数和具体状态变量。
如果有明确 revert reason，或者 owner 攻击重放命中了某个 payload，我可以把根因收敛到比较具体的权限路径；
但如果只是卖出失败而没有更多链上错误信息，我不会把 blacklist、router 限制或 pair 问题硬说成已证实根因，只会说这是待进一步归因的候选原因。”

这类回答在安全面试里反而是加分项，因为它说明你知道：

  • 哪些是观测事实
  • 哪些是推断
  • 哪些还需要额外证据

Q9：异步部分是怎么处理的？

建议回答：

监控、RPC 调用、LLM 请求这些都适合异步；但 Streamlit 自身不是 async-first 框架，所以我做了一个后台线程，在线程里跑独立 event loop。前端通过 asyncio.run_coroutine_threadsafe() 把任务提交进去，后台跑完后再把报告和日志同步回前端。

如果面试官继续追问“异步到底是什么”，你可以再展开成这一版：

异步的核心思想是，遇到 RPC、HTTP、子进程这种要等待外部响应的操作时，不要阻塞整个程序，而是先把执行权让出去，等结果回来再继续。
在 Python 里，我主要是用 async def 定义协程，用 await 等待这些 I/O 操作，再由 event loop 统一调度。

你这个项目里的代码落地，大致是 4 层：

  1. 先准备一个后台 event loop 在 app.py 里，我没有让 Streamlit 主线程直接跑异步逻辑，而是先用 asyncio.new_event_loop() 创建一个新的事件循环，再起一个守护线程，让这个 loop 在后台一直 run_forever()。
    这样做的原因是 Streamlit 每次交互都会 rerun 脚本，如果把异步任务直接塞在主线程里，状态会很乱，也不方便长期运行监控。

  2. 把真正耗时的任务写成协程 比如：

    • _on_new_pair()：发现新交易对后，异步调用 CoordinatorAgent.run()
    • _manual_scan()：用户手动输入地址后，异步跑完整审计流程
    • MonitorService.start()：异步轮询链上事件
    • CoordinatorAgent.run()：异步串联元数据获取、沙盒仿真、审计和报告生成

    这些函数的共同点是内部都有大量 await，因为它们都在等链上 RPC、HTTP 请求或者子进程执行结果。

  3. 前端通过 thread-safe 方式把任务丢给后台 在 UI 线程里，真正提交任务时我用的是 asyncio.run_coroutine_threadsafe(coro, loop)。
    例如手动扫描弹窗里，用户点“扫描”后，前端会把 _manual_scan(token_address, chain_name) 这个协程扔到后台 loop 去跑，然后拿到一个 Future。
    这样前端就不用自己 await，而是继续刷新页面、展示日志、轮询任务有没有完成。

  4. 后台跑完后，把结果同步回前端 因为后台线程不能直接安全地改 st.session_state，所以我用了模块级共享列表 _shared_reports 和 _shared_log 作为中转。
    后台任务完成后先把报告和日志写进共享列表；主线程每次渲染时，再通过 _sync_shared_to_session() 把这些数据搬进 session_state，最终显示在页面上。

你可以把这套实现总结成一句话：

“主线程只负责渲染 UI，后台线程负责跑 event loop；前端把协程安全地提交给后台执行，后台把结果写回共享区，主线程再同步到页面。”

如果面试官问“为什么不用纯多线程”，你可以这样答：

因为这个项目主要是 I/O 密集型，不是 CPU 密集型。RPC、Etherscan、LLM、Anvil 子进程这些场景，用 asyncio 统一调度会比开很多线程更轻，也更容易管理超时、重试和任务生命周期。
我这里保留一个后台线程，不是为了用线程做并发计算，而是为了给 Streamlit 提供一个稳定存在的异步运行环境。

如果面试官问“代码里最关键的几个异步落点在哪”，你可以直接点名：

  • app.py 里的 get_or_create_loop()：创建后台事件循环
  • app.py 里的 asyncio.run_coroutine_threadsafe(...)：提交协程到后台
  • services/monitor.py 里的 start()：异步轮询链上事件
  • agents/coordinator.py 里的 run()：异步串联整个审计流水线
  • app.py 里的 _sync_shared_to_session()：把后台结果同步回前端

Q10：数据是怎么持久化的？

建议回答：

当前真正稳定使用的持久化是 Markdown 报告落盘到 reports/。数据库层已经建了 SQLAlchemy 模型和 Repository，也会在启动时建表，但业务结果还没有完整接到数据库写入链路里。所以这一块我会实话实说，已经把基础设施搭好了，但当前主要还是文件级持久化。

这题不要说成“已经全量落库”。

Q11：这个项目里最难的工程问题是什么？

建议回答：

我觉得最难的是把链上仿真、后台异步任务和 Streamlit 这种同步 UI 拼到一起，还要保证用户看到的是连续、可理解的执行过程。链上调用不稳定、Anvil 启动有成本、UI 又会频繁 rerun，这几个约束叠在一起，最难的是把执行链路做稳。

面试时不要说错的点

1. 不要说“Coordinator 决定是否触发深度分析”

更准确的说法：

Coordinator 负责编排，真正的规则决策现在写在 AuditorAgent.decide()。

5. 不要说“数据库已经完整承接了审计结果”

真实情况是：

  • DB 模型和 Repository 已经有
  • 应用启动时会建表
  • 但主业务链路当前主要还是把报告写到 reports/

7. 不要把它说成“完全自治的智能体系统”

更准确的说法：

这是一个带有限决策能力的多 Agent 编排系统。它的价值在于职责拆分和局部决策，而不是完全自主规划。

面试时可以主动承认的局限

这部分非常重要。面试官通常不怕你项目有边界，怕的是你自己都不知道边界在哪。

局限 2：规则系统比较轻量

可以这样说：

现在的规则主要覆盖蜜罐、税率、ownership、mint 和部分未知风险，还没有做更系统的黑名单函数扫描、代理合约识别、权限图分析。它现在更像一个 MVP 级的 rule-based scorer，而不是成熟规则引擎，但框架已经留好了扩展点。

局限 3：数据库没完全接入主链路

可以这样说：

数据库基础设施已经有了，但为了先跑通端到端体验，我优先做了 Markdown 报告落盘。后续如果做查询分析、报表聚合、历史回溯，DB 一定要接上。

如果面试官追问“你最满意的技术决策是什么”

建议回答：

我最满意的是把“静态风险猜测”升级成“动态执行验证”。在 Web3 里，很多风险不是源码里有没有某个关键词，而是交易真正执行时会不会被拦、税率是不是异常、项目方能不能临时改状态。用分叉链去跑真实买卖和攻击重放，让系统输出更接近用户真实体验，这个方向我觉得是对的。

如果面试官追问“你最想优化哪一块”

建议回答：

我会优先做三件事：

  1. 把仿真层完全做成链配置驱动，真正打通 ETH/BSC
  2. 把数据库接入主链路，支持历史聚合和查询
  3. 把当前比较硬编码的攻击 payload 和风险规则，升级成更系统的权限分析和函数扫描

这样项目就会从一个“能跑的风险扫描器”升级成一个“可持续演进的审计平台”。

面试时最好的状态不是“像背文档”，而是“像你真的维护过这套系统”。

所以你的表达顺序尽量遵循：

  1. 先讲目标和问题场景
  2. 再讲主链路
  3. 再讲最有价值的技术点
  4. 最后主动讲边界和下一步优化

只要你能把“亮点”和“边界”同时讲清楚，这个项目就足够支撑一轮不错的面试交流。
