Euler Finance 审计笔记

审计范围: src/euler/ 的核心架构与模块，重点关注 Storage.sol、BaseLogic.sol、modules/EToken.sol、modules/DToken.sol、modules/Liquidation.sol、modules/RiskManager.sol Solidity 版本: ^0.8.0（vendor/ 下存在混合 pragma）

合约概览

这套代码是典型的“单存储 + 多模块逻辑”架构。真正持有状态的是 Euler.sol 和 Storage.sol，真正执行业务的是各个模块合约。

我先从 Euler.sol 看入口，马上能确认这是一个强模块化系统：

  • dispatch() 会把来自受信任 proxy 的调用转成 delegatecall，把执行路由到对应模块。
  • 这意味着所有模块共享一套全局存储；任何一个模块只要少做一次安全检查，影响的就不是“这个函数自己”，而是整个协议账本。
  • 高危特征非常集中：
    • delegatecall 路由
    • 借贷协议核心账本
    • flashLoan
    • 清算折价机制
    • 自借 / 自抵押路径

Slither 说明

  • 这次我实际跑了 slither . --exclude-dependencies，已经能直接扫当前项目。
  • 这轮输出里有几类命中我会这样看：
    • controlled-delegatecall @ Base.callInternalModule()
      • 我的判断：这是 Euler 模块化架构的设计特征，不是这次 exploit 的根因，但它提醒我“任一模块漏检都会放大成全协议风险”。
    • reentrancy-balance @ DToken.flashLoan()
      • 我的判断：值得记一下，但更像“闪电贷回调前后余额检查”的通用风险提示，不是这次 DAI 攻击链主因。
    • arbitrary-send-erc20 @ BaseLogic.pullTokens()
      • 我的判断：这是通用 transferFrom 封装带来的高噪声命中，单看这个告警不足以说明真实漏洞。
  • 更关键的是：Slither 这次没有替我直接指出 donateToReserves() 的问题，真正把漏洞钉死的还是人工对同类函数的差异审计。

资金在哪

钱和债都锚定在 Storage.sol 的 AssetStorage / UserAsset 上：

Storage.sol 是 Euler 的“全局状态定义文件”。它不做业务逻辑，核心功能是定义整个协议要存哪些状态，以及这些状态的结构和映射关系确定各变量内存槽位置

struct UserAsset {
    uint112 balance;              // 用户持有的 eToken 内部记账余额，本质上代表这名用户在该资产池中的抵押份额
    uint144 owed;                 // 用户在该资产池中的债务余额，对应 dToken / borrow 侧的欠款
    uint interestAccumulator;     // 用户上次计息时记录的利息累计器，用来把历史 owed 换算成当前应还债务
}

struct AssetStorage {
    uint96 reserveBalance;        // 协议储备金余额，手续费和 donateToReserves 一类的价值最终会沉到这里
    uint112 totalBalances;        // 全池 eToken 总余额，也就是所有用户抵押份额之和
    uint144 totalBorrows;         // 全池总债务
    mapping(address => UserAsset) users; // 每个用户在这个资产池上的资产/债务账本
}

我读到这里时，脑子里的第一反应是：

“这协议真正的命门不是 token transfer，而是所有会改 balance、owed、reserveBalance 的函数。”

所以我顺着账本保护链继续往下看：

  1. BaseLogic.getAccountLiquidity() 通过内部模块调用进入 RiskManager.computeLiquidity()
  2. BaseLogic.checkLiquidity() 在状态变更后强制做健康度检查
  3. RiskManager.requireLiquidity() 最终执行两个硬约束：
    • 不能违反 borrow isolation
    • collateralValue >= liabilityValue

这条保护链很清楚：

“凡是会让用户更危险的状态变化，做完以后必须调 checkLiquidity(account)。”

补充：underlying 是什么

这里专门补一句，避免把变量语义看混：

  • EToken 里的 underlying 是底层 ERC20 的地址（比如 DAI 地址），不是金额变量。
  • 它来自 CALLER() 解包后的 assetStorage.underlying，而不是临时计算出来的数值。

来源链我在代码里按顺序对上了：

  1. Markets.doActivateMarket() 把底层币地址写进 assetStorage.underlying
  2. Storage.AssetStorage 用 address underlying; 持久化保存这个市场对应的底层币
  3. EToken.CALLER() 通过 eTokenLookup[proxyAddr] 拿到该市场的 assetStorage，再读出 underlying

所以这一行：

(address underlying, AssetStorage storage assetStorage, address proxyAddr, address msgSender) = CALLER();

本质是在说：“先确认当前调用对应哪个 eToken 市场，再拿这个市场底层币的地址。”

这也解释了为什么一份 EToken 逻辑可以服务多资产市场：

  • 逻辑代码是共享的
  • 市场差异由 proxy + storage 映射出来的 underlying 决定

另外容易混淆的点也顺便钉住：

  • underlying（变量）= 地址
  • amount in underlying units（注释语义）= 数量单位（后续会按 decimals 做缩放）

我把 BaseLogic.checkLiquidity() 这几行单独拆开看了一遍：

function checkLiquidity(address account) internal {
    uint8 status = accountLookup[account].deferLiquidityStatus;

    if (status == DEFERLIQUIDITY__NONE) {
        callInternalModule(MODULEID__RISK_MANAGER, abi.encodeWithSelector(IRiskManager.requireLiquidity.selector, account));
    } else if (status == DEFERLIQUIDITY__CLEAN) {
        accountLookup[account].deferLiquidityStatus = DEFERLIQUIDITY__DIRTY;
    }
}

这里的含义其实很直接：

  • 先读 accountLookup[account].deferLiquidityStatus
    • 也就是先看这个账户当前是不是处于“允许延迟流动性检查”的上下文里。
  • 如果状态是 DEFERLIQUIDITY__NONE
    • 就说明现在没有任何豁免，必须立刻做健康度校验。
    • 于是它直接 callInternalModule() 调到 RiskManager.requireLiquidity(account)。
    • 真正的硬约束检查也就在那边执行：不能违反借贷隔离，且抵押价值必须覆盖债务价值。
  • 如果状态是 DEFERLIQUIDITY__CLEAN
    • 说明当前正处在一个允许“先做一串状态变更，最后统一检查”的流程里。
    • 这时不会立刻校验，而是把状态标成 DEFERLIQUIDITY__DIRTY，意思是：“这个延迟检查上下文已经被改脏了，后面必须补做总检查。”

所以这段代码的安全意义是：

  • checkLiquidity() 不是装饰性函数，它就是账本修改后的统一闸门。
  • 正常路径下，它会立刻把账户送去 RiskManager 做硬校验。
  • 延迟检查路径下，它至少也会留下“这个账户已经被改动过，别忘了稍后结算”的标记。

也正因为如此，donateToReserves() 漏掉这一步才会这么致命：

  • 它改了用户 balance
  • 但既没有立即把账户送去 requireLiquidity()
  • 也没有把账户标成需要延迟结算

等于直接绕过了账本变更后的统一风控闸门。

逐函数分析

EToken.deposit()

  • 代码位置：src/euler/modules/EToken.sol:139
  • 干了什么：
    • 拉入 underlying
    • 换算成内部记账单位
    • 增加用户 balance
  • 我注意到：
    • 这一路是“加抵押”，按理说风险较低
    • 但如果账户已经在这个市场有债务，会形成 self-collateralized loan，所以依然会在特定条件下补一发 checkLiquidity(account)

EToken.withdraw()

  • 代码位置：src/euler/modules/EToken.sol:193
  • 干了什么：
    • 计算用户想提多少
    • 向外转 underlying
    • 减少用户 balance
  • 我注意到：
    • 这就是一个标准的“减少抵押物”路径
    • 做完 decreaseBalance() 之后，立即调用 checkLiquidity(account)，非常符合预期

EToken.mint()自借杠杆函数。

  • 代码位置：src/euler/modules/EToken.sol:221
  • 干了什么：
    • 先把输入金额从 external units 解码到内部精度：decodeExternalAmount(...)
    • 再用 underlyingAmountToBalanceRoundUp(...) 换算出要铸造的 eToken 内部份额（这里是向上取整）
    • 接着又把 amountInternal 反算回 underlying 数量：balanceToUnderlyingAmount(...)，确保后续债务侧和资产侧对齐
    • 然后执行两步核心记账：
    • increaseBalance(...)：给用户加 eToken 余额（资产侧）
    • increaseBorrow(...)：给用户加 dToken 债务（负债侧）
    • 最后 checkLiquidity(account) 校验这次“自借杠杆”后账户仍健康
  • 我注意到：
    • 这个函数最危险的点在于：它不需要真实转入新的 underlying，就能同时把资产和负债放大
    • 对攻击者来说，这正好是“把头寸先膨胀到很大规模”的工具；头寸越大，后面 donateToReserves() 抹掉抵押后的坏账绝对值越大
    • 也正因为风险高，作者在流程末尾强制做了 checkLiquidity(account)；说明他们知道这步是高危动作
    • 从 exploit 视角看，mint() 本身不是漏洞点，但它是漏洞收益放大的前置放大器

EToken.transferFrom()

  • 代码位置：src/euler/modules/EToken.sol:352
  • 干了什么：
    • 把 eToken 从 from 挪到 to
  • 我注意到：
    • 对 from 来说是减少抵押，对 to 来说可能形成自抵押债务
    • 所以这里对 from 和部分场景下的 to 都做了流动性检查

EToken.donateToReserves()

  • 代码位置：src/euler/modules/EToken.sol:390
  • 干了什么：
    • 从用户 balance 里扣除 eToken
    • 同时把同额 value 记入 reserveBalance
  • 这里要特别澄清一个很容易误解的点：
    • 这不是“用户再额外转一笔新钱进协议”
    • 用户捐掉的是自己已经持有的 eToken 份额，也就是自己原本在池子里的抵押权益
    • 换句话说，donateToReserves() 本质上是一条“主动放弃自己抵押份额，把它改记到协议储备”的路径
  • 我注意到：
    • 这其实也是“减少用户抵押物”的路径
    • 从账本影响上看，它和 withdraw()、transferFrom(from, ...) 没有本质区别，甚至更危险，因为它不会把资产发回用户，而是直接沉到协议储备
    • 但函数末尾完全没有 checkLiquidity(account)
  • Slither 命中:
    • 这次 Slither 没有直接把 donateToReserves() 标成核心漏洞
    • 真正高价值的发现仍然来自人工对 withdraw() / mint() / burn() / transferFrom() / donateToReserves() 的 post-state check 差异比对

Liquidation.computeLiqOpp() / executeLiquidation()

compute Liquidation Opportunity，计算一笔清算机会的参数”，比如能还多少（repay）、能拿多少抵押物（yield）、健康度（healthScore）、折价（discount）等

  • 代码位置：
    • src/euler/modules/Liquidation.sol:55
    • src/euler/modules/Liquidation.sol:224
  • 干了什么：
    • 先算健康度
    • 健康度 < 1 时给清算折价
    • 再把违约人的债务转给清算人，并把抵押物折价划走
  • 我注意到：
    • 80 行开始，健康度低于 1 直接进入折价路径
    • 91-102 行会把折价和 conversion rate 算出来
    • 242-246 行把债务转给或铸给清算人，274 行把 collateral 划给清算人
    • 也就是说，一旦有人能“合法地”把自己打成深度资不抵债，清算模块就会很诚实地帮他把坏账货币化

🔍 发现的问题

问题1: donateToReserves() 销毁抵押物后遗漏流动性检查

  • 严重程度: Critical
  • 怎么发现的:
    • 我不是先盯 exploit 交易，而是先盯“所有会让用户更危险的函数”。
    • 然后把几条同类路径摆在一起比：
      • withdraw() 结尾有 checkLiquidity(account)
      • mint() 结尾有 checkLiquidity(account)
      • burn() 结尾有 checkLiquidity(account)
      • transferFrom() 对 from 也有 checkLiquidity(from)
    • 结果扫到 donateToReserves() 时，发现它同样减少 balance，却在结尾直接 logAssetStatus(assetCache) 返回。
    • 这时候我再回头看 BaseLogic.checkLiquidity() 和 RiskManager.requireLiquidity()，就能确认：协议原本的安全模型确实要求这种操作在 post-state 做健康度验证。
  • 问题本质:
    • donateToReserves() 允许用户主动减少自己的抵押资产，却不验证这次减少之后账户是否还满足 collateralValue >= liabilityValue。
    • 这里减少的不是“额外赠送的一笔外部资产”，而是用户已经持有的抵押份额本身。
    • 换句话说，攻击者可以保留债务、抹掉抵押、制造坏账。
  • 问题代码:
    function donateToReserves(uint subAccountId, uint amount) external nonReentrant {
    (address underlying, AssetStorage storage assetStorage, address proxyAddr, address msgSender) = CALLER();
    address account = getSubAccount(msgSender, subAccountId);

    updateAverageLiquidity(account);
    emit RequestDonate(account, amount);

    AssetCache memory assetCache = loadAssetCache(underlying, assetStorage);

    uint origBalance = assetStorage.users[account].balance;
    uint newBalance;

    if (amount == type(uint).max) {
        amount = origBalance;
        newBalance = 0;
    } else {
        require(origBalance >= amount, "e/insufficient-balance");
        unchecked { newBalance = origBalance - amount; }
    }

    assetStorage.users[account].balance = encodeAmount(newBalance);
    assetStorage.reserveBalance = assetCache.reserveBalance = encodeSmallAmount(assetCache.reserveBalance + amount);

    emit Withdraw(assetCache.underlying, account, amount);
    emitViaProxy_Transfer(proxyAddr, account, address(0), amount);

    logAssetStatus(assetCache);
}
  • 攻击叙事:
    • 我是攻击者，我先找一个流动性很深的市场，比如 DAI。
    • 我用闪电贷拿到第一笔大资金。
    • 然后我调用 mint()，把自己的 eDAI 和 dDAI 一起放大，先把杠杆堆起来。
    • 这一步之后，我虽然债很大，但抵押也很大，所以 checkLiquidity() 还过得去。
    • 接着我调用 donateToReserves()，把一大块 eDAI 从自己账上抹掉，直接记入协议储备。
    • 因为这里没有 post-state 健康度检查，我的账户就能在链上成功变成“抵押远小于债务”的坏账账户。
    • 一旦进入这个状态，清算模块就会按设计给出高额折价。
    • 我再用第二个账号或第二个合约来清算自己，把系统给出的折价吃掉。
    • 最后还掉闪电贷，折价差额就是利润。
  • 代码位置:
    • 漏洞函数：src/euler/modules/EToken.sol:390
    • 漏掉的统一防线：src/euler/BaseLogic.sol:585
    • 健康度硬约束：src/euler/modules/RiskManager.sol:374
    • 利润兑现路径：src/euler/modules/Liquidation.sol:80
  • 修复思路:
    • 最直接的修法：在 donateToReserves() 完成账本更新后、返回前加入 checkLiquidity(account)。
    • 更稳一点的修法：把“任何减少 balance 或增加 owed 的路径都必须调用流动性检查”抽成显式开发规范，并补一组 invariant / regression test。
    • 审计侧我也会建议做一个“同类函数差异对比”测试：枚举所有会写 users[account].balance / users[account].owed 的函数，确保危险方向变更后都走到统一健康度校验。

结论

这轮 scoped 审计里，最核心、最确定、也最贴合真实攻击路径的问题，仍然是：

donateToReserves() 在减少用户抵押后没有做 post-state liquidity check。

我没有在这轮人工审计里再看到第二个同级别的关键漏洞；真正让这次事故爆炸的，不是复杂的价格操纵，而是一个非常朴素的“不一致性”：

  • 同类危险函数大多会调用 checkLiquidity()
  • 唯独 donateToReserves() 没有

这就是典型的“边缘功能绕开核心不变量”。

💡 收获与心得

  1. 审计借贷协议时，先盯账本，不要先盯交易。 真正关键的是谁在改 balance、谁在改 owed、谁在改 reserveBalance。

  2. 同类函数 diff 是挖这种漏洞的高性价比方法。 尤其是那种“大家最后都调一个 guard，只有一个函数没调”的情况，命中率很高。

  3. 清算模块常常不是根因，但经常是利润放大器。 这次真正的漏洞在 EToken，但真正把坏账变成利润的是 Liquidation。

  4. 如果要长期用 Slither，源码树就别为了“好看”随便重排。 目录直观性可以靠 README 和笔记补，工具链稳定性只能靠源码布局本身保证。
