Euler Finance 漏洞复现说明

1. 项目目标

本项目复现的是 Euler Finance 在 2023-03-13 的经典攻击路径，核心目标不是“把交易跑出来”这么简单，而是把下面三件事都讲清楚：

  1. 漏洞为什么会发生
  2. 攻击者如何把漏洞变成可兑现利润
  3. 如何用 Foundry + Tenderly 在指定历史区块稳定复现

2. 漏洞根因

本次漏洞的根因在 EToken.donateToReserves()。

问题本质是：

  • 用户调用 donateToReserves() 后，自己的抵押物会减少
  • 但对应债务不会减少
  • 且函数末尾没有像 withdraw()、mint() 那样执行 checkLiquidity(account)

这意味着攻击者可以主动把自己从“健康仓位”推成“可清算仓位”，然后再通过清算模块吃到折价。

一句话总结：

Euler 允许攻击者在不触发风控校验的情况下，销毁自己的抵押，从而制造一个可以被折价清算的坏账仓位。

为什么用户抵押了钱还能 donateToReserves()

这里容易误解成“用户又额外拿钱捐给协议”，但实际不是。

更准确地说：

  • 用户先把 DAI 存进 Euler
  • 协议不会一直把这笔底层 DAI 以“原样归属”的方式记在用户名下
  • 协议会给用户记一笔 eDAI 份额，代表他对池子的索取权

所以 donateToReserves() 捐掉的，不是用户钱包外面新转进来的一笔钱，而是：

用户已经持有的 eToken 份额，也就是他原本的抵押权益。

正常情况下，用户减少自己的抵押权益之后，协议应该立即检查这笔账户是否还健康。

本次漏洞的关键不是“用户为什么能捐”，而是：

用户捐掉自己的抵押份额之后，Euler 没有立即执行流动性检查。

3. 攻击路径

复现出来的攻击链如下：

  1. 从 Aave V2 闪电贷借入 30,000,000 DAI
  2. 向 Euler 存入一部分 DAI 作为初始抵押
  3. 通过 mint() 放大 eDAI 和 dDAI 仓位
  4. 偿还一部分债务，调整仓位形态
  5. 再次 mint()，把仓位进一步做大
  6. 调用 donateToReserves()，减少抵押但不减少债务
  7. 账户健康度跌破 1，进入可清算状态
  8. 由另一个地址发起清算，拿到折价抵押
  9. 从 Euler 池中提取剩余真实 DAI
  10. 归还 Aave 闪电贷本息，保留净利润

4. 为什么要拆成两个角色

项目里把攻击者拆成：

  • violator：持有不健康仓位
  • liquidator：发起清算的一方

这样做的原因是 Euler 的清算逻辑明确禁止 self-liquidation。

所以在 PoC 里，必须把“被清算账户”和“清算执行账户”分开。

5. 复现环境

本项目使用两套方式验证：

A. Foundry fork test

使用 Tenderly RPC 作为 fork 状态源，在历史区块复现：

source .env && forge test --match-test testExploit -vvv --rpc-url $TENDERLY_RPC_URL --fork-block-number 16817995

特点：

  • 代码在本地执行
  • 状态来自 Tenderly fork
  • 适合快速验证 PoC 是否成立

B. Tenderly Virtual TestNet 广播

通过脚本把真实 exploit 交易发到 Tenderly，方便在 Debugger 中逐层查看：

source .env && forge script script/EulerTenderly.s.sol:EulerTenderlyScript --rpc-url $TENDERLY_RPC_URL --broadcast --slow -vvvv

这里使用 --slow 的原因是：

  • 脚本会先部署 exploit 合约
  • 再调用 exploit 合约
  • 如果不加 --slow，第二笔调用可能在第一笔部署确认前发出，导致打到“当时还没代码的地址”

6. 复现结果

当前 PoC 已经稳定跑通，关键输出如下：

  • liquidator DAI after withdraw: 38904507
  • Aave amount owing: 30027000
  • Final DAI balance: 8877507

这说明：

  1. 利用链已经完整闭环
  2. 闪电贷本息已经成功归还
  3. 剩余约 8.88M DAI 为本次 PoC 中复现出来的净利润

7. 代码结构

本项目里与复现最相关的文件：

  • test/poc.t.sol
    • 主 PoC 测试文件
    • 直接展示完整攻击链
  • script/EulerTenderly.s.sol
    • Tenderly 广播脚本
  • test/interfaces/
    • 最小外部接口定义
  • src/euler/modules/EToken.sol
    • 核心漏洞点
  • src/euler/modules/Liquidation.sol
    • 清算与折价逻辑

8. 这个项目体现的能力

如果从面试角度总结，这个复现体现的不是“会跑工具”，而是以下能力：

  1. 能从源码层定位漏洞根因，而不是只抄攻击脚本
  2. 能把协议业务逻辑翻译成最小可验证的 PoC
  3. 能区分“本地 fork test”和“真实广播调试”的不同语义
  4. 能使用 Foundry、Tenderly、Aave、Euler 模块接口完成跨协议复现
  5. 能把复杂 exploit 拆成可解释的阶段，并输出可验证结果
