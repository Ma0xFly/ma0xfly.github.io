Day 1：Storage 布局与代理模式

题卡 1：Solidity 的 storage layout 是怎么分配的？

问题

  • Solidity 状态变量在 storage 里是怎么排布的？

短答

  • Solidity 的状态变量默认按声明顺序分配 storage slot，每个 slot 是 32 字节。
  • 多个小于 32 字节的值类型会尽量紧凑打包到同一个 slot。
  • struct 和 array 通常从新 slot 开始，mapping 自己占一个逻辑 slot，但真正的数据存到哈希派生位置。

展开

  • EVM 的 storage 可以理解为一个 slot => 32 bytes 的大字典。
  • Solidity 编译器会按变量声明顺序给状态变量分配 slot。
  • 对于 uint128、bool、address 这类小于 32 字节的值类型，编译器会尝试把它们塞进同一个 slot，这就是紧凑打包。
  • 如果当前 slot 剩余空间不够，就会开下一个 slot。
  • struct 和 array 一般会从新的 slot 开始，避免布局歧义。
  • mapping 比较特殊，它声明位置的 slot 更像一个“种子槽位”，真正的 value 不直接放在这个 slot，而是通过 keccak256 计算出实际存储位置。

题卡 2：什么是紧凑打包，为什么它重要？

问题

  • Solidity 的紧凑打包是什么？为什么面试里经常问？

短答

  • 紧凑打包就是把多个较小的状态变量放进同一个 32 字节 slot 里，减少 storage 占用。
  • 它重要是因为会影响 gas、slot 编号和升级合约时的存储兼容性。

展开

  • storage 写入是昂贵操作，所以多个小变量如果能共用一个 slot，通常更省空间。
  • 例如两个 uint128 可以共用一个 slot，一个 address 加一个 bool 在某些顺序下也可能打包。
  • 但打包不是“越多越好”，因为变量顺序会直接影响布局。
  • 在代理升级场景里，旧版本和新版本如果布局不兼容，就可能出现 slot 错位，最终覆盖关键数据。

题卡 3：mapping 的 storage slot 怎么算？

问题

  • mapping(key => value) 的值到底存在什么位置？

短答

  • 假设 mapping 变量本身的 slot 是 p，某个 key 对应 value 的存储位置通常是 keccak256(abi.encode(key, p))。
  • 也就是说 mapping 的数据不是顺序排的，而是通过 key 和槽位一起哈希出来的。

展开

  • mapping 不能像数组一样顺序存储，因为 key 的类型和分布都不固定。
  • 所以 Solidity 不会把所有值线性铺开，而是把 mapping 声明所在的 slot 作为基准点。
  • 当你访问 m[key] 时，编译器会把 key 和 p 一起编码，再做一次 keccak256，得到最终 storage slot。
  • 如果 value 本身还是复杂类型，比如 struct，那这个哈希出来的位置通常是 struct 起始位置，后续字段再按偏移继续排。
  • 面试里最好强调一点：mapping 的声明 slot 本身通常不直接存 value，而是参与地址推导。

题卡 4：dynamic array 的 storage slot 怎么算？

问题

  • 动态数组在 storage 中怎么存？长度和元素位置分别在哪？

短答

  • 动态数组变量本身所在的 slot p 一般存数组长度。
  • 数组元素的起始位置通常是 keccak256(p)，第 i 个元素位于 keccak256(p) + i。

展开

  • 和普通值类型不同，动态数组长度不固定，所以 Solidity 需要把“长度”和“数据区”分开。
  • 数组变量自己的 slot 主要用于记录数组长度。
  • 真正的数据不会紧挨着长度继续放，而是从 keccak256(p) 开始放置。
  • 这样设计的好处是能避免和其他变量的 slot 冲突，并且支持不定长扩展。

题卡 5：什么是 storage collision？

问题

  • 代理模式里为什么会发生 storage collision？

短答

  • 因为代理模式下常用 delegatecall 执行逻辑合约代码，但写入的是代理合约自己的 storage。
  • 如果逻辑合约和代理合约对 slot 的理解不一致，就会把不该写的数据写到关键 slot 上，造成存储碰撞。

展开

  • delegatecall 的核心特征是“用别人的代码，操作自己的状态”。
  • 所以当 proxy 通过 delegatecall 调用 implementation 时，implementation 里的状态变量并不会写到 implementation 自己的 storage，而是写到 proxy 的 storage。
  • 如果 proxy 把某个 slot 留给 admin、implementation、beacon，而 implementation 恰好也把这个 slot 当成普通业务变量，就会发生覆盖。
  • 这就是 storage collision，本质不是“两个合约变量重名”，而是“同一个 slot 被两套布局解释成了不同含义”。
  • 这类问题在升级合约里尤其危险，因为它可能直接导致管理员地址被改掉、升级入口失控或资产状态被破坏。

题卡 6：为什么 delegatecall 会导致代理合约写自己的 storage？

问题

  • delegatecall 到底和普通 call 差在哪？

短答

  • call 是去对方上下文执行，对方用自己的 storage。
  • delegatecall 是借对方代码执行，但保留当前合约上下文，所以 storage、msg.sender、msg.value 都沿用调用者一侧的环境。

展开

  • 代理模式之所以可升级，本质上就是把“代码”和“状态”拆开。
  • proxy 保存状态，implementation 保存逻辑。
  • 如果使用普通 call，那 implementation 会在自己的上下文里执行，根本碰不到 proxy 的状态，这样代理模式就没法工作。
  • 所以要用 delegatecall，让 implementation 代码在 proxy 的 storage 上运行。

题卡 7：Transparent Proxy 和 UUPS 的核心区别是什么？

问题

  • Transparent Proxy 和 UUPS 分别是什么？差异在哪？

短答

  • Transparent Proxy 把升级逻辑放在 proxy 里，proxy 更重，但升级控制更集中。
  • UUPS 把升级逻辑放在 implementation 里，proxy 更轻、更省 gas，但如果实现写坏了，可能失去升级能力。

展开

  • Transparent Proxy 的思路是：proxy 不只是转发，还自己持有升级入口和管理员判断逻辑。
  • 它的好处是升级逻辑始终留在 proxy 端，不容易因为 implementation 出错而彻底失去升级入口。
  • 代价是 proxy 更重，且每次调用通常需要经过额外判断身份耗费gas。
  • UUPS 则更轻，proxy 主要负责 delegatecall 转发，升级逻辑由 implementation 提供。
  • 好处是结构更简洁、部署和运行更省一些。
  • 风险是如果新的 implementation 没有正确保留升级接口或校验逻辑，proxy 可能还能转发，但系统已经无法继续升级，甚至直接变砖。

题卡 8：为什么可升级代理不能靠 constructor 初始化，而必须用 initialize？

问题

  • 可升级代理里为什么不能把关键状态写在 logic contract 的 constructor 里？
  • 为什么一定要通过 proxy 去调用 initialize？还要防什么额外风险？

短答

  • 代理模式的核心是逻辑与状态分离：logic 存代码，proxy 存状态。
  • constructor 只会在 implementation 自己部署时执行，写入的是 implementation 自己的 storage，不会初始化 proxy 的 storage。
  • 所以必须通过 proxy 用 delegatecall 执行 initialize，把状态写进 proxy 上下文；同时还要防止 implementation 自身被人直接调用初始化并接管。

展开

  • 这题要先抓住代理模式的本质：
  • proxy 持有状态
  • implementation 提供逻辑
  • 用户平时交互的不是 implementation，而是 proxy
  • 关键点在 delegatecall：proxy 调 implementation 代码时，执行环境仍然是 proxy，所以 storage 写入落在 proxy 上
  • 这也解释了为什么 constructor 不适合做可升级状态初始化：
  • constructor 只在 implementation 部署那一刻跑一次
  • 当时还没有通过 proxy 的 delegatecall 上下文
  • 因此它改到的是 implementation 自己的 storage，不是 proxy 的 storage
  • 结果就是：logic 看起来“初始化过了”，但真正承载业务状态的 proxy 其实还是未初始化状态
  • 正确做法是提供 initialize / reinitializer 这类外部初始化入口，再由 proxy 去调用
  • 这样代码虽然在 implementation 里，但状态写入发生在 proxy 的 storage，上下文才是对的
  • 这里还有第二层安全点：
  • 如果 implementation 自身没有被禁用初始化，攻击者可能直接调用 implementation 的 initialize
  • 一旦把 owner / admin / upgrader 之类权限变量初始化到 implementation 自己名下，就可能为后续攻击链打开入口，尤其在 UUPS 场景里风险更高
  • 所以工程上通常要同时做两件事：
  • 通过 proxy 调用初始化函数，确保业务状态落在 proxy
  • 在 implementation 构造阶段调用 _disableInitializers()，防止实现合约本体被单独初始化
  • 面试里更好的纠偏说法是：
  • storage collision 讨论的是“布局冲突”
  • 初始化问题讨论的是“状态写入发生在哪个上下文”
  • 两者有关，但不是一回事

题卡 9：EIP-1967 为什么能降低 storage collision 风险？

问题

  • EIP-1967 解决了什么问题？为什么它常和代理模式一起被问？

短答

  • EIP-1967 规定了 implementation、admin 等关键地址要放到标准化的特殊 slot 里，而不是随便放在普通业务 slot。
  • 这样可以显著降低和业务变量发生碰撞的风险，也方便工具统一识别代理合约。

展开

  • 早期代理模式很容易因为 slot 选得随意而和业务变量冲突。
  • EIP-1967 的思路是把关键元数据放到经过约定的特殊 slot 中，例如通过 bytes32(uint256(keccak256("eip1967.proxy.implementation")) - 1) 这种方式生成。
  • 这样做有两个好处：
  • 第一，这些 slot 很难和普通编译器按顺序生成的业务 slot 重合。
  • 第二，区块浏览器、审计工具和升级框架都能更容易识别代理结构。
  • 所以它不是“从根本上消灭所有碰撞”，而是通过标准化把高风险冲突面大幅缩小。

题卡 10：为什么高 TVL 协议的 Proxy Admin 不应该是 EOA？

问题

  • 一个 DeFi 协议为什么不能让单个 EOA 直接掌控升级权限？

短答

  • 因为代理升级权本质上接近系统最高权限，单个 EOA 风险太集中。
  • 私钥泄露、单人作恶或被胁迫都可能导致协议被恶意升级，所以更合理的是多签、治理和时间锁组合。

展开

  • 对高 TVL 协议来说，升级权并不是普通运维权限，而是接近“根权限”。
  • 如果这个权限由单个 EOA 掌控，那么攻击面非常直接：盗私钥、社工、木马、内部作恶都可能导致全协议被接管。
  • 更稳妥的做法至少是多签，把单点失败变成多点共谋。
  • 再往上是 DAO 治理，把升级决策交给治理流程。
  • 但即使是多签或 DAO，也仍然需要时间锁，因为治理本身也可能被操纵。
  • 所以成熟做法通常是 Proxy Admin + Multisig/DAO + Timelock 组合，而不是把希望押在一个人身上。

题卡 11：如果面试官让我 90 秒解释钻石代理，我该怎么答？

问题

  • 什么是 Diamond Proxy？什么时候会考虑它？

短答

  • Diamond Proxy 可以把系统功能拆成多个 facet，由同一个代理统一转发。
  • 它适合功能很多、需要模块化扩展的大型系统，但复杂度和治理成本都比普通代理更高。

展开

  • 普通代理通常是一个 proxy 对应一个 implementation，升级时整体替换。
  • Diamond 的思路是把不同函数选择器映射到不同 facet 合约，相当于一个代理挂载多个功能模块。
  • 这样做的好处是模块拆分更细，升级粒度更灵活，不需要每次替换整个实现。
  • 但代价也很明显：路由更复杂，审计面更大，storage 管理更容易出错。
  • 所以 Diamond 不是“更高级的默认选择”，而是适合超大合约系统的工程化方案。
  • 面试里如果被追问，你可以补一句：它本质上是在代理模式上继续做函数级路由和模块化拆分。

Day 2：调用上下文、Gas 与签名授权

题卡 1：call、delegatecall、staticcall 的核心区别是什么？

问题

  • 面试里如果让我快速区分 call、delegatecall、staticcall，我该怎么答？

短答

  • call 是在被调合约上下文执行，使用被调方的 storage。
  • delegatecall 是借用被调合约代码，但保留调用方上下文，所以 storage、msg.sender、msg.value 都沿用调用方一侧语义。
  • staticcall 和 call 类似，但强制只读，不能发生状态修改。

展开

  • call 最像普通外部调用，代码、storage 和执行环境都切到被调方。
  • delegatecall 最特殊，它只借代码，不切上下文，所以经常被用来做代理升级、library 风格复用。
  • staticcall 则是在调用层面加了“只读约束”，即便底层代码里试图写状态，也会 revert。

题卡 2：call 会怎么影响 storage、msg.sender 和 msg.value？

问题

  • 普通 call 执行时，上下文到底怎么变？

短答

  • call 会切到合约 B 的 storage 上下文。
  • 对 B 来说，msg.sender 变成调用它的合约 A。
  • msg.value 则是 A 显式传给 B 的值，没有传就是 0。

展开

  • 假设外部用户先调用合约 A，A 再用 call 调 B。
  • 这时 B 看到的 msg.sender 不是最初那个用户，而是 A。
  • B 使用的是自己的 storage，所以 B 对状态的写入不会直接改 A 的状态。
  • msg.value 也不是自动沿用上层值，必须由 A 在 call{value: ...} 时显式指定。

题卡 3：为什么 delegatecall 会保留原始 msg.sender？

问题

  • 为什么 delegatecall 下 msg.sender 不会变成被调合约地址？

短答

  • 因为 delegatecall 的设计目标就是“在当前上下文里执行别人的代码”。
  • 它不会创建新的完整执行上下文，而是保留调用方的 storage 和消息语义，所以 msg.sender、msg.value 都继续沿用上层调用链中的值。

展开

  • 还是假设用户调 A，A 再 delegatecall 到 B。
  • B 中执行的代码其实像是“贴到 A 里运行”，因此读写的都是 A 的 storage。
  • 同理，B 看到的 msg.sender 仍然是最外层调用 A 的那个用户，而不是 A 自己。
  • 这也是代理模式能成立的原因，因为逻辑合约必须像自己就是 proxy 一样操作状态和权限。
  • 但风险也很大，因为一旦逻辑合约里有危险代码，它操作的是 proxy 的状态，不是自己的一块隔离区。

题卡 4：为什么 staticcall 写状态会 revert？

问题

  • staticcall 为什么能保证只读？如果里面调用写状态函数会怎样？

短答

  • 因为 staticcall 在 EVM 层面携带了静态上下文约束。
  • 在这个上下文里，一旦发生状态修改类操作，就会触发异常并 revert。

展开

  • staticcall 不是“开发者约定俗成的只读”，而是 EVM 执行语义上的硬约束。
  • 只要当前执行路径处在 static context 中，像 SSTORE、创建合约、销毁合约等会引起状态变化的操作都不被允许。
  • 所以即便目标函数没有标 view，只要你用 staticcall 去调，它也不能真正写入状态。
  • 这类设计的意义在于：上层调用者可以强制要求“我这次外部读取必须是只读的”，降低意外交互风险。

题卡 5：call{value: ...} 的潜在风险与防御是什么？

问题

  • 如果面试官追问“call 携带 msg.value 转账时，除了重入还有什么风险”，我该怎么答？

短答

  • 风险不只是常规重入，还包括触发目标合约 fallback() / receive() 中的危险逻辑、低级调用静默失败，以及 gas 转发行为带来的边界问题。
  • 防御上不能只背 CEI，还要补一句：低级 call 必须检查返回值，必要时主动 revert，不能让“账本已改、转账没成”这种状态静默发生。

展开

  • call{value: ...} 的危险点主要有三层：
  • 第一层，控制权交出去。
  • 只要你把 ETH 通过低级 call 发给目标地址，对方就可能在 receive() 或 fallback() 里立即执行逻辑。
  • 这会带来：
  • 常规重入
  • 跨函数重入
  • 在某些依赖外部读值的场景里，还可能把中间态暴露给别的路径消费
  • 第二层，低级调用不会自动替你处理失败。
  • call 返回的是 (bool success, bytes memory data)。
  • 如果你不检查 success，就可能出现：
  • 你的账本已经减了余额，但 ETH 实际没有成功转出去
  • 或者目标逻辑失败了，你这边还当成成功继续往后跑
  • 这类问题比单纯的 revert 更危险，因为它会制造“静默失败”。
  • 面试里可以直接给一句标准写法：
  • (bool success, ) = target.call{value: amount}(data);
  • require(success, "call failed");
  • 第三层，gas 转发不是无限自由的。
  • call 默认的 gas 转发行为会受 EIP-150 的 63/64 规则影响。
  • 这意味着你不能简单假设：
  • 对方一定有足够 gas 把复杂逻辑跑完
  • 失败一定会被优雅处理
  • 回调里的行为一定和你想象得一样
  • 所以安全分析时，要把“目标 fallback 可能干什么”和“它到底有多少 gas 能干这些事”一起看。
  • 综合起来，最稳的防御回答应该是：
  • 先做检查
  • 先改自己账本
  • 最后再做外部转账（CEI）
  • 低级 call 必须检查返回值

题卡 6：EIP-150 的 63/64 规则是什么？

问题

  • 什么是 63/64 规则？

短答

  • EIP-150 规定外部调用时，实际可转发给子调用的 gas 不能简单把当前 gas 全部送出去，而是通常最多只能转发大约 gasleft() * 63 / 64。
  • 它的目的之一是保证上层调用者在子调用结束后还留有 gas 处理返回值、错误和后续逻辑，而不是被下游调用把 gas 完全吃光。

展开

  • 这里最容易混淆的是两件事：第一，整笔交易给链上的 gas limit 够不够；第二，当前调用帧把多少 gas 转发给下一层。
  • EIP-150 解决的主要不是“用户整笔交易 gas 不够”这个问题，而是“调用者不能把自己这一层的 gas 毫无保留地全送给子调用”。
  • 假设合约 A 在执行过程中还剩 64000 gas，如果没有这条规则，A 理论上可以把几乎所有 gas 都转给 B。
  • 如果 B 把这些 gas 全部消耗掉并 OOG(Out Of Gas)，B 这一层当然会失败，但问题是 A 返回之后可能已经几乎没有 gas 了。
  • 这时 A 连最基本的收尾动作都做不了，比如检查 success、走备用分支、记录失败、主动 revert，甚至连后续几条指令都可能跑不完。
  • 结果就会从“子调用失败，但调用者还有机会决定怎么处理”变成“下游失败顺便把上游也拖死”，这就是为什么要强制给上层留一点 gas。
  • EIP-150 引入 63/64 规则后，A 最多只能把大约 64000 * 63 / 64 = 63000 gas 送给 B，自己至少还能保留大约 1000 gas。
  • 这部分 gas 不一定很多，也不保证足够做复杂逻辑，但至少让 A 不会因为一次外部调用就被完全抽干，仍有机会处理返回结果。
  • 如果不做这种保留，典型后果有三个：
  • 第一，A 会失去处理失败的能力，连 if (!success) 这种最基本的收尾代码都可能跑不完。
  • 第二，本来只是 B 这层调用失败，最后却会升级成 A 也 OOG，严重时整笔交易一起回滚。
  • 第三，会放大 DoS 或 gas griefing 风险，让下游合约有机会通过“吃光 gas”干扰上游的控制流。
  • 所以这个规则不是单纯的费用优化，也不是“失败了反正就 revert”的简单逻辑，而是在调用语义上保护调用者的控制权。
  • 另一个副作用是，深层嵌套调用时 gas 会层层衰减，这让调用深度天然受到软约束，也让 gas griefing 和复杂回调链分析变得更现实。

题卡 7：EIP-150 对攻击和防御分别意味着什么？

问题

  • 63/64 规则除了是机制题，对安全分析有什么意义？

短答

  • 对攻击者来说，它限制了“把所有 gas 压给下游”的某些操作预期。
  • 对防御者来说，它提醒你不能假设外部调用一定拥有足够 gas，也不能把 gas 行为想得过于理想化；更重要的是，要区分“子调用失败”和“调用者还有没有 gas 处理失败”。

展开

  • 在复杂调用链里，下游合约实际拿到的 gas 往往比调用者直觉里更少。
  • 这会影响回调函数能做多少事、fallback 是否跑得完、某些分支会不会因 gas 不足失败。
  • 安全上常见的误区有两个：
  • 第一，以为“我传了很多 gas，所以外部逻辑一定能完整执行”。
  • 第二，以为“对方一定还能执行某段逻辑”，但实际在深层调用下可能已经没那么多 gas。
  • 还有第三个常见误区：觉得“如果下游 OOG，那不就是直接 revert 吗”。其实低级 call 下更准确的语义是：子调用失败，返回 success = false，然后由调用者决定接下来怎么办。
  • 如果调用者自己也没 gas 了，它就失去了这个决定权。于是本来应该是“可控失败”的地方，可能变成上游一起 OOG，或者根本没法走备用逻辑。
  • 所以分析攻击链时，除了状态和权限，也要考虑 gas 是否支持这条链真实跑通。
  • 它不是万能防线，但会改变很多边界条件。

题卡 8：permit 的完整链路该怎么讲？

问题

  • EIP-2612 permit 到底在做什么？如果面试官让我从头讲一遍，我该怎么答？

短答

  • permit 允许用户先离线签名，再由链上交易提交授权，从而不需要用户自己先发一笔 approve。
  • 它的核心链路是：构造 EIP-712 消息、签名、链上验证签名、检查 nonce 和 deadline，然后更新 allowance。

展开

  • 传统 approve 需要用户自己发链上交易，体验差，而且需要先有 gas。
  • permit 的思路是把“授权意愿”先写进结构化消息里，由用户离线签名。
  • 提交者可以是用户自己，也可以是第三方 relayer。
  • 合约收到请求后，一般会验证以下几件事：
  • 签名是否对应 owner
  • 域分隔是否正确，避免跨合约或跨链混用
  • nonce 是否匹配，避免重放
  • deadline 是否过期
  • 全部通过后，再更新 allowance
  • 所以 permit 的本质不是“省一次 approve”这么简单，而是把授权从普通交易改成了“签名 + 提交”的双阶段模型。

题卡 9：为什么 permit 必须有 nonce 和 deadline？

问题

  • permit 里为什么几乎总会有 nonce 和 deadline？

短答

  • nonce 主要用来防重放，确保同一份签名不能被无限次复用。
  • deadline 主要用来限制签名有效期，降低用户长期暴露在钓鱼或延迟执行风险中的概率。

展开

  • 如果没有 nonce，攻击者拿到一份合法签名后，就可能反复提交同样的授权请求。
  • nonce 的作用就是把“这次签名”绑定到一个唯一序号，用过一次就失效。
  • 如果没有 deadline，签名理论上可以长期有效，这会带来很大的尾部风险。
  • 比如用户今天签了一份消息，几个月后市场环境变了、资产变多了，这份旧签名却仍可能被人拿出来执行。
  • 所以 permit 安全里，nonce 解决的是“能不能重复用”，deadline 解决的是“能用到什么时候”。
  • 面试里再补一句更完整：它们都不能防止用户主动签给钓鱼消息，但能收缩被利用的窗口和方式。

题卡 10：permit 和 EIP-1559 分别最容易被追问什么？

问题

  • 如果 Day 2 最后被追问到 permit 风险或 EIP-1559 基础，我该怎么稳住？

短答

  • permit 最容易被追问的是：签名验证链路、nonce/deadline 作用、钓鱼和重放风险。
  • EIP-1559 最容易被追问的是：base fee 会被销毁，priority fee 是给打包者的激励，两者共同决定交易费用结构。

展开

  • 对 permit 来说，面试官常常不是要你背标准原文，而是看你是否真的理解“签名授权会引入哪些新攻击面”。
  • 你要能讲：
  • 域分隔为什么重要
  • 为什么 relayer 模式会带来提交时序问题
  • 为什么用户可能被诱导签下危险消息
  • 对 EIP-1559 来说，一般不需要讲到特别深的实现细节，但至少要能区分：
  • base fee 是协议根据区块拥堵动态调整的基础费，并被销毁
  • priority fee 是用户给打包者的小费
  • max fee 则是用户愿意支付的上限
  • 如果时间紧，Day 2 优先保住调用语义、EIP-150 和 permit，EIP-1559 作为保底基础题即可。

题卡 11：Multicall + delegatecall 为什么会出现“多次存款、只付一次钱”？

问题

  • 如果面试官给你一个 Multicall 例子，问为什么循环里能多次 deposit，却只进来一次 ETH，你怎么解释？

短答

  • 因为 delegatecall 会保留外层调用者的执行上下文，msg.sender 和 msg.value 在整个调用链里都保持一致。
  • msg.value 不是“每次循环重新扣一次的钱”，它只是这笔顶层交易进入合约时附带的只读环境变量。
  • 如果 Multicall 内部把多个 payable 调用都当成“各自收了一次 ETH”，就会把同一份 msg.value 重复记账。

展开

  • 这类漏洞的核心不是“余额没扣”，而是把环境变量当成了可消费资源。
  • 在 EVM 里，原生 ETH 的实际转移发生在 call 的 value transfer 阶段；合约内部读取 msg.value 只是看到本次外层调用携带了多少 ETH。
  • delegatecall 不会重新创建一层独立的 msg.value，它只是让被调用代码在当前合约上下文里执行，所以在 for 循环里每一次迭代看到的仍然是同一个值。
  • 这就会出现经典错误：
  • 外层只在第一次进入时带入 1 ETH
  • Multicall 循环里多次 delegatecall 到同一个 deposit/payable 逻辑
  • 每次逻辑都检查到 msg.value == 1 ETH
  • 于是余额检查如果只看“msg.value > 0 就记一次存款”，就会重复铸造份额或重复记账
  • 所以面试时要讲清楚两个层次：
  • 业务现象：看起来像多次 deposit、只付一次钱
  • 底层原因：delegatecall 保留上下文，msg.value 在一次顶层交易内只读且全局一致，真正的价值转移不是在读取 msg.value 时发生的

工程视角修复方案

  • 更标准的做法不是“相信 Multicall 会自己处理好”，而是直接限制 payable 逻辑的可重入调用方式：
  • 在 Multicall 中禁止 msg.value 被多次透传给内部子调用
  • 约束 payable 函数只能由顶层入口单次消费，禁止循环里重复进入同一记账路径
  • 如果必须支持批量操作，改成内部记账模型，例如先把 ETH 包装成 WETH，或者记录内部 shares / credits，再在批量结束时统一结算
  • 这样能把“真实资金转移”与“逻辑调用次数”解耦，避免把同一份 msg.value 记多次

关键词 / 面试金句

  • delegatecall 保留 msg.sender / msg.value
  • msg.value 是环境变量，不是循环里可重复扣费的余额
  • 价值转移发生在 call，不是读取 msg.value 的瞬间
  • 标准防御：禁止多重传递 msg.value，或把 ETH 先转成 WETH / 内部 shares 再批量处理

Day 3：重入、UTXO vs Account、Slither

题卡 1：如果不用 DeFi 术语，怎么重新定义重入？

问题

  • 不提 AMM、借贷、提款，单从程序与状态机角度看，重入到底是什么？

短答

  • 重入本质上不是“某个函数被再次调用”这么简单，而是系统在一次状态更新尚未完成时，被外部控制流重新插入执行。
  • 更抽象地说，它是时序失控和临界资源竞争，最终破坏了状态更新的原子性。

展开

  • 很多人第一次学重入，会把它记成“先转账后改余额”，但这只是常见表象，不是本质。
  • 更本质的描述是：合约在处理某个关键状态变化时，把执行权暂时交给了外部，而外部又在中间重新进入了本系统。
  • 于是系统内部原本假设的“这一步做完再做下一步”被打断了。
  • 一旦关键状态还没更新完，或者某个资源还没被正式扣减，新的调用就可能读到旧状态、重复消费同一资源，或者触发不该通过的校验。

题卡 4：什么是跨函数重入？

问题

  • 如果不是重进同一个函数，也算重入吗？

短答

  • 算。跨函数重入是指攻击者在一个函数执行过程中，通过回调进入同一合约的另一个函数，而那个函数依赖的状态同样还没更新完成。

展开

  • 很多人以为只要给某个危险函数加锁就够了，但真实系统里，多个函数可能共享同一批核心状态。
  • 例如函数 A 正在处理中间态，外部调用发生后，攻击者不一定非要再打 A，也可以打函数 B。
  • 如果 B 同样读取那批尚未最终落盘的状态，或者能操作同一份资源，就可能绕开“只防同函数”的防护。
  • 所以跨函数重入比单函数重入更能体现你是否真的理解“问题在状态共享，而不在函数名字相同”。
  • 面试里最好强调：只要多个入口共享同一临界状态，就要按“一个状态机”来防，而不是按“一个函数一个函数”来防。

题卡 6：什么是只读重入，为什么“只读”也危险？

问题

  • 没有直接写坏状态，为什么只读重入仍然能造成经济损失？

短答

  • 因为“只读”不代表“无影响”。如果某个协议在错误时机读取了一个暂时失真的状态或价格，再用它做定价、清算或借贷判断，就可能造成真实损失。

展开

  • 只读重入最容易迷惑人的地方在于：被重入的那一侧可能没有直接执行状态写入。
  • 但问题在于，另一个协议把它当成预言机、报价源或健康度输入时，读到的可能是一个尚未完全同步的中间状态。
  • 也就是说，危险不一定来自“把状态写坏”，而可能来自“把错误快照当成真实世界”。
  • 例如池子在执行流动性移除过程中，某些内部变量已经变化，某些还没变化；这时外部协议来读取价格，就可能得到一个短暂偏离真实值的结果。
  • 如果这个读值被拿去做借贷、清算或兑换判断，后果就会落到真钱上。
  • 所以只读重入的本质是：状态快照失真被外部系统消费，而不是传统意义上的重复扣款。

题卡 7：UTXO 和 Account 模型的核心差异是什么？

问题

  • 如果面试官让我从底层模型上比较 UTXO 和 Account，我该怎么答？

短答

  • UTXO 更像一组彼此独立、一次性消耗的离散输入，天然更局部、更易并行。
  • Account 模型则是所有交易共享一个全局状态机，表达力更强，但更容易出现竞争、排序依赖和全局状态冲突。

展开

  • UTXO 全称是 Unspent Transaction Output，可以理解成“未花费的输出票据”。
  • 在 UTXO 模型里，系统不直接维护“某个账户现在余额是多少”这类中心视角，而是记录：你手里现在有哪些还没花掉的输出。
  • 当你发起一笔交易时，本质不是“把账户余额减掉一点”，而是“消耗若干旧输出，再生成若干新输出”。
  • 例如你有一个 10 BTC 的 UTXO，要转给别人 3 BTC，常见做法不是把余额从 10 改成 7，而是：
  • 消耗那个 10 BTC 的旧 UTXO
  • 生成一个 3 BTC 给对方的新 UTXO
  • 再生成一个 7 BTC 的找零 UTXO 给自己
  • 这意味着 UTXO 更像局部、离散、一次性的资源消费模型。
  • 只要两笔交易不花同一个 UTXO，它们天然彼此隔离，因此验证与并发处理都更自然。
  • Account 模型则完全不同。它直接维护账户余额、nonce、合约 storage、池子储备、借贷头寸等全局状态。
  • 交易执行时，本质是在修改这棵共享的全局状态树。
  • 例如以太坊转账时，系统会直接改发送者余额、接收者余额和发送者 nonce；如果是合约调用，还会继续修改 mapping、数组、储备量、债务等状态。
  • 所以 Account 模型可以概括成“全局共享状态机”，表达力更强，也更适合复杂智能合约。
  • 这里说“表达力更强”，具体体现在几个层面：
  • 第一，它更容易表达持久化的共享可变状态。例如一个 AMM 池子的储备量、一个借贷协议的总债务、一个治理系统的投票状态，都可以长期挂在链上并被多个用户持续读写。
  • 第二，它更容易表达复杂状态机。例如订单从 Created -> Filled -> Cancelled，提案从 Pending -> Active -> Succeeded -> Executed，这些都可以直接用合约状态变量维护。
  • 第三，它更容易表达跨用户、跨事务累积的逻辑关系。例如 allowance、抵押率、清算阈值、累计利息、LP 份额，这些都依赖历史状态不断滚动更新。
  • 第四，它更容易表达合约之间的同步组合。一个合约可以直接调用另一个合约，立即读取它的状态、拿返回值继续计算，再决定下一步怎么做，这对 DeFi 组合性特别关键。
  • 也正因为如此，Proxy、Vault、AMM、Lending、DAO 这类复杂系统在 Account 模型下实现会更自然。
  • 反过来看，传统 UTXO 更擅长表达“消费哪些输入、产生哪些输出”，但对这种持续演化的共享状态、复杂权限关系和多合约同步交互表达起来会更别扭。
  • 但代价就是状态耦合更强。大家不再是只消费各自独立的票据，而是在争夺“谁先改全局状态、改哪一部分状态”。
  • 这也是为什么 EVM 世界里更容易出现排序依赖、共享状态竞争、清算争夺和 MEV。
  • 从防重放角度看，两者也不一样：
  • UTXO 更依赖“输入只能被花一次”，也就是同一个 UTXO 一旦消费掉，再拿同一笔交易重放通常会因为输入已失效而失败。
  • Account 更依赖 nonce，即同一个账户的交易按序执行，同一个 nonce 只能成功一次。
  • 所以如果面试官让你抽象总结，最值得说的一句就是：
  • UTXO 是局部资源消费模型，Account 是全局共享状态模型。
  • 这句总结后面可以自然连到并发能力、重放防御、MEV 和状态竞争。

面试加强版

  • 如果只说“比特币用 UTXO，以太坊用账户模型”，这只是记名词，不够深入。
  • 更强的答法是把后果一起讲出来：
  • UTXO 因为状态边界清晰，所以更容易并行验证，双花问题也更容易形式化判断。
  • Account 因为所有人共享同一个全局状态，所以交易顺序本身会改变结果，排序权也就变成了有经济价值的资源。
  • 这就是为什么 Account 模型更容易产生 MEV、夹子攻击、清算竞争和预言机依赖问题。
  • 如果面试官追问“那表达力更强具体强在哪”，就落到四个词：
  • 共享状态
  • 复杂状态机
  • 跨事务累计逻辑
  • 合约同步组合
  • 也可以再补一句 tradeoff：
  • UTXO 更清晰、更局部、更适合并发
  • Account 更灵活、更强表达、更适合智能合约
  • 代价则是全局状态竞争和时序敏感性更强

题卡 8：为什么 Account 模型更容易产生 MEV 和临界资源竞争？

MEV 现在通常指 Maximal Extractable Value，最大可提取价值。

简单说，就是：区块生产者或能影响交易排序的人，通过控制一笔交易：

  • 先后顺序
  • 插队
  • 打包与否

从中额外榨取出来的利润。

问题

  • 为什么 EVM 的全局状态模型天然更容易产生 MEV？

短答

  • 因为交易不是在彼此独立的局部状态上运行，而是在共享的全局状态上串行执行。
  • 当多个用户都想抢同一个价格、清算机会或池子状态时，执行顺序本身就变成了可套利资源。

展开

  • 在 EVM 里，一笔交易的结果往往依赖它前面那笔交易刚刚把状态改成了什么。
  • 对 DeFi 来说，这种依赖尤其强，因为价格、储备量、健康因子、套利空间都和前一笔交易的结果紧密相关。
  • 所以内存池里的竞争，不只是“谁先上链”，而是“谁先改全局状态”。
  • 这就让排序权本身具有经济价值，MEV 由此出现。
  • 从更抽象的角度讲，热门合约其实就是高价值的临界资源。
  • 谁先访问、谁插队、谁夹在中间，都会改变最终收益分配。

题卡 9：Slither 到底在做什么？为什么它不是“神奇扫漏洞”？

问题

  • Slither 这种静态分析工具，底层到底在干嘛？

短答

  • Slither 本质上是在源码和字节码语义之间构建中间表示，再结合控制流、数据流和规则匹配做静态近似分析。
  • 它很强，但不是“扫一下就知道一定有漏洞”的神谕。

展开

  • 静态分析工具不会像真实链上执行那样把每条路径都跑一遍。
  • 它更像是在“代码还没运行前”，先建立程序结构图，然后检查某些危险模式是否存在。
  • 常见步骤包括：
  • 把 Solidity 代码转成更适合分析的中间表示
  • 建立控制流图 CFG，看函数和基本块怎么流转
  • 建立数据流 / 污点传播关系，看危险输入会不会流到危险操作
  • 用 detector 规则去匹配潜在问题，比如重入、未检查返回值、权限缺失等
  • 所以 Slither 提供的是高价值线索，不是终局判决。
  • 它特别适合做“第一轮筛查”和“代码味道定位”，但最后仍需要人结合业务语义判断是否真能被利用。

题卡 10：为什么静态分析常常会误报，面试里该怎么答？

问题

  • 工具都报出来了，为什么还说不一定真 exploitable？

短答

  • 因为静态分析看到的是“代码形状”，不一定完整理解真实执行约束、权限条件、上下文前提和协议业务语义。
  • 所以它擅长发现可疑路径，但不保证这些路径在现实中真的能走通。

展开

  • 误报的来源很多。
  • 有些路径在语法上存在，但现实中被权限控制、初始化顺序、调用前置条件或外层框架约束住了。
  • 有些 detector 看到“外部调用 + 状态更新”就会提示重入风险，但实际代码可能有锁、快照或不变量保护。
  • 还有些问题在单合约维度看像漏洞，但放进完整协议流程里并不能形成可利用攻击链。
  • 所以一个成熟的安全工程师不会把工具输出当结论，而会继续追问：
  • 调用路径真实可达吗
  • 攻击者权限足够吗
  • 是否存在经济收益
  • 成本是否合理
  • 是否会被其他机制拦住
  • 面试里如果你能把“工具发现”与“人工验证”区分开，通常会显得比只会跑命令更专业。

题卡 11：静态分析在 AST/CFG 和跨合约污点分析上的边界是什么？

问题

  • 如果面试官追问“你说 Slither 有边界，那具体边界在哪”，该怎么答？

短答

  • 静态分析在单合约结构化问题上很强，比如基于 AST、CFG 的模式识别和路径筛查。
  • 但它在跨合约污点传播上有天然硬伤：真实执行依赖动态调用、代理升级、外部协议状态和运行时上下文，静态图很难完整复原。
  • 所以它擅长给线索，不擅长单独给最终 exploit 结论。

展开

  • 可以按三层讲清：
  • 第一层，AST 解决“代码长什么样”。它擅长抓语法和结构模式，比如危险调用点、权限修饰器缺失、变量读写关系。
  • 第二层，CFG 解决“代码可能怎么走”。它能近似分析函数内与函数间的控制流，辅助发现可疑路径。
  • 第三层，污点分析解决“危险输入会流到哪里”。它尝试跟踪用户输入、外部返回值、价格数据等是否流入关键汇点。
  • 真正的难点出在跨合约场景：
  • delegatecall、代理合约和可升级实现会让真实代码边界在运行时变化
  • 外部协议返回值受链上瞬时状态影响，静态阶段难以准确建模
  • 多协议组合里的利用链常跨多个事务和角色，不是单次函数路径能表达
  • 别名分析、路径爆炸和约束求解成本会导致精度与性能必须权衡，结果要么漏报要么误报
  • 所以面试里更专业的表述是：
  • 静态分析 = 高召回的前置筛查器
  • 动态验证（fork test / PoC）= 可利用性证明器
  • 人工语义审计 = 业务约束与经济可行性的最终裁决
  • 这也解释了为什么“AI 大模型可以辅助”但不能替代这三层闭环：模型能帮你提速归纳，不会自动消除跨合约污点分析的可达性与上下文不完备问题。

关键词 / 面试金句

  • AST 看结构，CFG 看路径，Taint 看数据流
  • 单合约高效，跨合约易失真
  • 先静态高召回，再动态证实，再人工定性

Day 4：AMM、TWAP、Flash Loan

题卡 1：Uniswap V2 的核心做市逻辑是什么？

问题

  • 如果面试官让我用一句话讲 Uniswap V2，我该怎么答？

短答

  • Uniswap V2 是恒定乘积做市商，核心约束是池子中两种资产的储备乘积尽量满足 x * y = k。
  • 用户买入一种资产时，必须卖出另一种资产，池子通过储备变化自动给出价格。

展开

  • Uniswap V2 不是传统订单簿，而是把两种资产放进同一个池子里。
  • 池子维护两份核心状态：reserve0 和 reserve1。
  • 交易者往池子里放入一种资产，就能取走另一种资产。
  • 但池子不会随便给价格，而是通过储备关系约束输出数量。
  • 最经典的抽象是 x * y = k，其中 x 和 y 是两边储备，k 是某种意义上的不变量。
  • 当一种资产被买走，它的储备减少；另一边储备增加，于是边际价格立刻发生变化。
  • 所以 V2 的本质是：价格不是预先喂给池子的，而是由储备比例和交易本身共同决定。

题卡 4：Uniswap V3 的 Tick 和区间流动性是什么？

问题

  • V3 比 V2 高级在哪？Tick 和区间流动性要怎么讲？

短答

  • V2 的流动性默认铺满整个价格区间，V3 则允许 LP 只在自己选定的价格区间提供流动性。
  • Tick 可以理解为离散化的价格刻度，流动性在不同 Tick 区间之间切换。

展开

  • V2 的流动性是“全范围”的，不管价格跑到哪里，理论上你的资金都在场。
  • 这听起来简单，但资本利用率不高，因为很多资金其实长期躺在根本用不到的价格区间里。
  • V3 的改进是让 LP 自己指定一个价格范围，只在这个范围内提供流动性。
  • 为了让这个机制可执行，协议把连续价格离散成很多 Tick。
  • 你可以把 Tick 理解成“价格刻度”，流动性会在 Tick 区间之间分布和切换。
  • 当价格落在你的区间内，你的资金参与做市、赚手续费。
  • 当价格跑出你的区间，你的流动性会变成单边资产，暂时不再提供有效双边做市。
  • 所以 V3 的本质是：牺牲简单性，换取更高资本效率和更精细的流动性管理。

题卡 5：为什么说 V3 的资本效率更高，但风险也更复杂？

问题

  • V3 为什么被说成“更高效但更难管”？

短答

  • 因为 LP 不再把资金平均铺在整个价格轴上，而是集中部署在更可能发生交易的区间里，所以单位资金能产生更高做市深度。
  • 但代价是头寸管理、再平衡、无常损失和跑出区间后的风险都更复杂。

展开

  • 在 V2 里，资金铺得很散，优点是简单，缺点是很多资本闲置。
  • V3 允许你把资金压缩到更窄区间，所以同样 100 万美元流动性，在活跃价格附近能提供比 V2 更深的可交易深度。
  • 这就是资本效率更高的来源。
  • 但风险并没有消失，而是从“简单平均暴露”变成了“带主动管理要求的精细暴露”。
  • 价格一旦跑出你的区间，你就不再像预期那样赚手续费，头寸还可能变成单边资产。
  • 这要求 LP 更关注再平衡、市场波动和策略管理。
  • 所以面试里可以说：V3 让 LP 更像策略交易员，而不只是被动存钱的人。

题卡 6：TWAP 是什么，为什么比 Spot Price 更难操纵？

问题

  • 什么是 TWAP？为什么大家常说它比现价更安全？

短答

  • TWAP 是时间加权平均价格，它不是看某一个瞬时价格点，而是看一段时间内的平均价格。
  • 因此攻击者如果想操纵 TWAP，就需要在更长时间窗口内持续维持异常价格，成本通常比操纵单点现价更高。

展开

  • Spot Price 是你现在这一刻读到的价格，非常敏感，也很容易被单笔大额交易短暂拉歪。
  • TWAP 的思路是：不要只信这一瞬间，而是把一段时间里的价格轨迹综合起来。
  • 这样一来，攻击者不能只在一个区块里做个短暂拉盘就立刻骗过依赖方。
  • 如果协议读的是过去几分钟或更长时间窗口的均价，攻击者就要把异常价格维持得更久，或者在多个区块反复投入资金。
  • 这会显著提高操纵成本。
  • 所以 TWAP 不是“绝对安全”，而是“让攻击从短时低成本变成更长时、更昂贵”。
  • 这正是它的工程价值所在。

题卡 7：为什么 TWAP 不是万能的？评估它还要看什么？

问题

  • 面试里如果被追问“TWAP 也会被操纵吗”，我该怎么答？

短答

  • 会。TWAP 只是提高操纵成本，不是消灭操纵可能。
  • 判断它是否安全，还要结合流动性深度、窗口长度、更新频率和攻击者资本规模一起看。

展开

  • 如果池子本身很浅，哪怕做 TWAP，攻击者维持异常价格的成本也可能没有高到不可承受。
  • 如果时间窗口太短，比如只看最近几个区块，那它和现价相比并没有本质安全跃迁。
  • 对预言机安全来说，关键不只是“有没有用 TWAP”，而是：
  • 窗口够不够长
  • 池子够不够深
  • 交易成本够不够高
  • 被操纵后能造成的收益够不够大
  • 所以面试里你可以一句话总结：
  • TWAP 解决的是“瞬时价格可被低成本拉歪”的问题，但面对低流动性池或高收益攻击目标时，仍然可能失守。

题卡 8：Flash Loan 到底是什么，为什么它不是漏洞本身？

问题

  • 闪电贷到底是什么？为什么很多人说“闪电贷不是漏洞，只是放大器”？

短答

  • Flash Loan 是一种要求“同一笔交易内借出并归还”的无抵押借贷机制。
  • 它本身不是漏洞，真正的漏洞是协议对价格、状态或权限做了错误假设，闪电贷只是把攻击者的瞬时资金能力放大了。

展开

  • 正常借贷需要抵押，因为出借方担心你不还。
  • 闪电贷之所以能无抵押，是因为它要求在同一事务里完成借款、使用、还款。
  • 如果事务结束时钱没还上，整笔交易回滚，就像借款从未发生过。
  • 这意味着攻击者可以在一个区块内瞬间获得巨额资金，而不用长期持有这些资本。
  • 但这不等于“闪电贷让本来安全的协议凭空失守”。
  • 更准确地说，闪电贷会把原本存在但要求巨额本金才能利用的问题，变成人人可利用的现实攻击面。
  • 所以它是资金放大器、攻击加速器，不是根因。

题卡 9：Callback 在闪电贷攻击链里扮演什么角色？

问题

  • 为什么面试里老要问 callback？它和攻击到底是什么关系？

短答

  • Callback 是闪电贷协议把控制权交还给借款人的那一步。
  • 攻击者正是利用这段“已拿到钱、尚未归还”的窗口，去操纵价格、清算头寸或触发其他协议的错误逻辑。

展开

  • 闪电贷不是把钱打给你就结束了，它通常会在放款后立即调用你指定的回调函数。
  • 这一点非常关键，因为从那一刻开始，到你归还贷款之前，你拥有了一笔巨大但短暂的可用资本。
  • 攻击链往往就写在这个 callback 里：
  • 先借钱
  • 再拿大钱去推价格、搬池子、打清算、做跨协议套利
  • 最后在函数末尾还钱
  • 所以 callback 不是一个“语法细节”，而是闪电贷机制里最关键的攻击执行窗口。
  • 面试里如果你能把 callback 说成“资金真正进入攻击链的起点”，会比只说“回调函数很重要”更到位。

题卡 10：一个典型的价格操纵 / 闪电贷攻击链该怎么讲？

问题

  • 如果让我系统讲一遍价格操纵类攻击链，我该怎么有条理地回答？

短答

  • 典型链路是：借大额资金，短时扭曲某个关键状态或价格，让依赖方基于错误输入执行有利于攻击者的逻辑，最后获利并还款。
  • 关键不是“借到钱”，而是“依赖方错误地信了那个短时状态”。

展开

  • 一个经典模板通常分四步：
  • 第一步，借到大额资金，通常通过 flash loan 放大火力。
  • 第二步，利用这笔资金短时间改变目标池子的储备、价格或某个关键状态。
  • 第三步，让另一个依赖这个状态的协议做出错误决策，例如按虚高价格放贷、按错误估值接受抵押，或在错误价格下完成兑换。
  • 第四步，攻击者拿走收益，归还 flash loan，本次事务结束。
  • 这个模板之所以危险，是因为很多协议会错误地把“链上某一瞬间能读到的值”当成“稳健可信的真实价格”。
  • 所以你在面试里最好把问题说成“依赖方对数据来源的信任模型出了问题”，而不是只说“闪电贷很危险”。
  • 你还可以补一句常见防御：
  • 不直接信单点 spot price
  • 使用更稳健的 TWAP 或外部预言机
  • 提高操纵成本
  • 限制单区块内敏感操作

Day 5：Lending、Liquidation、Mock Interview

题卡 1：借贷协议的基本结构是什么？

问题

  • 如果让我抽象地讲 DeFi 借贷协议，它的核心结构是什么？

短答

  • 借贷协议的核心就是：存款人把资产存进池子提供流动性，借款人抵押资产后从池子里借出资产，协议再依赖预言机和风险参数持续评估谁安全、谁该被清算。
  • 它本质上是一个“资金池 + 会计系统 + 风险控制系统”的组合。

展开

  • DeFi 借贷协议里最基础的参与方有两类：
  • 一类是存款人，他们把资产放进协议，提供可借流动性，并期待赚取利息。
  • 一类是借款人，他们需要先提供抵押品，再借出另一种资产。
  • 但这还不够，协议还必须维护一套持续运行的会计系统：
  • 记录每个人存了多少
  • 借了多少
  • 利息累计到哪里了
  • 抵押品当前值多少钱
  • 另外还必须有一套风险控制系统：
  • 预言机负责提供价格
  • 参数系统负责定义 LTV、清算阈值、close factor、清算奖励等规则
  • 清算人负责在风险失控时把坏账及时处理掉
  • 所以你可以把借贷协议看成“链上的自动抵押借贷银行”，但这个银行没有人工风控，全靠代码、参数和外部价格源维持偿付能力。

题卡 2：为什么借贷协议需要 cToken / aToken 这类存款凭证？

问题

  • 存款凭证到底是干什么的？为什么不能只记一个余额？

短答

  • 因为协议需要一种可组合、可计息、可转移的方式来表示“你在池子里的份额”。
  • cToken / aToken 本质上都是存款头寸的链上凭证，只是记账方式不同。

展开

  • 如果协议只是内部记一个“你存了 100 USDC”，虽然也能运作，但表达力和组合性会很差。
  • 借贷协议更常见的做法是给存款人铸造一个代表其头寸的代币。
  • 这个代币有几个重要价值：
  • 它能证明你在池子里的权益
  • 它能承载利息增长逻辑
  • 它可以被钱包、前端、其他协议直接识别和组合
  • 它让“存款头寸”从抽象数据库记录变成了链上的一等公民
  • aToken 和 cToken 都属于这类设计，只是增值方式不同。
  • 所以不要把它们理解成“奖励积分”，更准确地说，它们是“生息头寸凭证”。

题卡 3：aToken 和 cToken 的高层差异是什么？

问题

  • Aave 的 aToken 和 Compound 的 cToken 都是存款凭证，但差别在哪？

短答

  • aToken 更像“余额直接增长”的模式，你看到的代币数量会随着利息累积而增长。
  • cToken 更像“数量相对稳定、兑换率上涨”的模式，真正变的是 exchangeRate。

展开

  • 对 Aave 来说，用户存款后拿到 aToken，它更直观地表现为“你钱包里这类资产余额在增长”。
  • 对 Compound 来说，用户拿到 cToken，它代表的是一种可兑换头寸，你最终能换回多少底层资产，取决于 cToken -> underlying 的兑换率。
  • 所以两者都能表达“存款会生息”，但用户感知层面不同：
  • aToken 更像余额动态增长
  • cToken 更像凭证数量相对固定、价值通过汇率增长
  • 从工程上看，这是两种不同的会计表达方式，而不是“一个会生息、一个不会生息”。
  • 面试里如果你能说出“一个偏余额模型，一个偏兑换率模型”，通常就够稳了。

题卡 4：借款利息为什么必须持续累计？协议是怎么理解这件事的？

问题

  • 借贷协议里的利息不是每秒真实转账，那它到底怎么累计？

短答

  • 借贷协议通常不会每秒给每个用户打款，而是通过索引、汇率或累积因子这类会计变量，把时间和利率折算进头寸价值。
  • 本质上是“会计更新”，而不是“实时逐笔转账”。

展开

  • 如果一个协议真要每秒对所有账户逐个改余额，成本会极高，也几乎不可扩展。
  • 所以更现实的做法是：
  • 维护一套全局或局部的利息累计指标
  • 在用户交互、查询或清算时，再把这些指标结算到具体头寸上
  • 这样协议就能做到：
  • 存款会随着时间产生收益
  • 借款会随着时间增加债务
  • 但链上不需要持续遍历所有账户更新状态
  • 所以借贷协议的关键不只是“会不会收利息”，而是“会计系统能不能正确反映利息随时间的累积”。
  • 这也是为什么很多借贷协议的核心风险并不只是权限漏洞，而是利率、索引、汇率、精度和状态同步问题。

题卡 5：LTV、liquidation threshold、close factor、liquidation bonus 分别是什么？

问题

  • 这些清算参数老是混在一起，它们到底各管什么？

短答

  • LTV 管的是“最多能借多少”。
  • liquidation threshold 管的是“什么时候开始危险到可清算”。
  • close factor 管的是“一次最多能被清多少债务”。
  • liquidation bonus 管的是“清算人为什么愿意来执行清算”。

展开

  • LTV 可以理解成放贷上线，它回答的是：用户拿这份抵押品，最多能借到什么程度。
  • liquidation threshold 更偏风控线，它通常高于或区别于 LTV，回答的是：什么情况下账户会跌入可清算区间。
  • close factor 是清算节奏控制器，它防止一次清算把整个头寸瞬间全部扫空，通常限制单次最多可偿还多少比例的债务。
  • liquidation bonus 则是给清算人的经济激励，因为如果没有奖励，没人愿意出 gas 和资金来帮协议处理坏账风险。
  • 所以这几个参数不是一回事：
  • 一个决定借款上限
  • 一个决定危险阈值
  • 一个决定每次清多少
  • 一个决定清算人为什么来
  • 面试里如果把这几个概念混了，通常会被继续追问。

题卡 6：健康因子 / 可清算条件应该怎么理解？

问题

  • 健康因子到底是什么？为什么不能只背一个公式？

短答

  • 健康因子本质上是在衡量“按协议当前风险参数和价格计算后，你的抵押是否还足以覆盖债务”。
  • 它不是一个脱离协议语义的纯数学数值，不同协议对参数和记账方式的定义会不同。

展开

  • 很多资料会给出类似：
  • Health Factor = (Collateral × 某种风险折扣) / Debt
  • 这个方向没问题，但面试里不能只背一个简化版，因为协议之间的定义并不完全一样。
  • 真正重要的是理解它在表达什么：
  • 抵押品值多少钱
  • 协议愿意按多保守的折扣承认这份价值
  • 当前债务值多少钱
  • 两者相比，安全垫还剩多少
  • 一旦这个安全垫低于某个阈值，协议就认为你的账户已经不再足够安全，可以被清算。
  • 所以健康因子不是“一个神秘分数”，而是风险参数、价格、债务和会计状态共同作用下的结果。

题卡 7：为什么清算不是“惩罚用户”，而是协议自救机制？

问题

  • 为什么借贷协议必须有清算？它的本质作用是什么？

短答

  • 清算的本质不是惩罚某个用户操作失误，而是在抵押不足时尽快回收风险敞口，防止协议整体出现坏账。
  • 它服务的是系统偿付能力，而不是情绪上的惩罚逻辑。

展开

  • 借贷协议的根本目标之一，是保证存款人的钱随时尽可能可兑付。
  • 如果借款人的抵押品价值跌得太多，但债务没同步下降，协议就面临坏账风险。
  • 这时如果不允许清算，损失最终会传导给整个资金池。
  • 所以清算机制的作用是：
  • 在账户还没彻底资不抵债前
  • 尽快让外部清算人介入
  • 还掉部分债务
  • 拿走对应抵押品和奖励
  • 这样协议可以缩小风险敞口，维持整体偿付能力。
  • 所以更准确的说法是：清算是系统级风险处置机制，而不是“惩罚不小心的用户”。

题卡 8：为什么借贷协议高度依赖预言机，而预言机又会成为核心风险？

问题

  • 借贷协议为什么离不开预言机？它又为什么这么危险？

短答

  • 因为借贷协议必须知道抵押品和债务当前值多少钱，否则无法判断用户还能借多少、是否该被清算。
  • 但一旦价格源被操纵或失真，整个风控逻辑就会跟着一起失真，所以预言机会从基础设施变成主要攻击面。

展开

  • 借贷协议不是只看“你有多少枚币”，而是要看“这些币现在值多少钱”。
  • 如果没有价格，协议就无法计算：
  • 你的抵押价值
  • 你的借款上限
  • 你的健康程度
  • 你是否已跌入清算区间
  • 所以预言机是借贷协议的眼睛。
  • 但问题也正出在这里：如果眼睛被欺骗，整个风控系统看到的世界就是错的。
  • 典型风险包括：
  • 直接读取易被操纵的 spot price
  • 低流动性池子价格被闪电贷扭曲
  • 预言机更新滞后
  • 多源聚合设计不当
  • 所以对借贷协议来说，预言机既是基础设施，也是最危险的单点之一。

题卡 9：为什么一个账户看起来安全，却可能突然变得可清算？

问题

  • 用户前一秒还没事，为什么下一秒就被清算了？

短答

  • 因为安全与否不是静态属性，而是由价格、利息、参数和市场状态共同决定的动态结果。
  • 只要其中某个关键变量快速变化，账户就可能从“看起来安全”瞬间跨进清算区间。

展开

  • 借贷账户是否安全，本质上是一个实时计算结果。
  • 所以它随时会被这些因素改变：
  • 抵押品价格下跌
  • 借入资产价格上涨
  • 借款利息持续累积
  • 预言机更新价格
  • 协议参数变化
  • 甚至其他关联市场的流动性变化
  • 这意味着“现在健康”不等于“下一个区块还健康”。
  • 很多清算并不是用户做错了某个新操作，而是市场状态变化后，原来的安全垫被迅速吃掉了。
  • 所以面试里可以把这类问题概括为：借贷安全性是动态状态，不是一次性判断。

题卡 10：Aave 和 Compound 的面试级差异该怎么讲？以及如何串成一轮追问？

问题

  • 如果面试官追问 Aave 和 Compound 的差异，或者把 Lending 和前几天题目串起来，我该怎么稳住？

短答

  • 高层上，两者都是超额抵押借贷协议，但 Aave 更常用 aToken 的余额增长式体验，Compound 更典型的是 cToken + 汇率增长式记账。
  • 如果被串题追问，可以把 Proxy、重入、AMM、预言机、清算全部落到“共享状态 + 风控输入 + 可组合性风险”这条主线上。

展开

  • Aave 和 Compound 的共性先讲清：
  • 都依赖超额抵押
  • 都依赖预言机
  • 都有存款凭证
  • 都通过参数系统和清算机制维持偿付能力
  • 再讲高层差异：
  • Aave 常见记忆点是 aToken、更直观的余额增长感知、参数体系和产品形态较丰富
  • Compound 常见记忆点是 cToken、兑换率增长模型、整体框架相对更“凭证化”
  • 不需要在题卡里展开成源码对比，只要讲到“记账方式和用户感知不同”就够了
  • 如果进入 mock 式追问，可以这样串：
  • Proxy / delegatecall：说明升级系统的权限和状态安全问题
  • Reentrancy：说明状态更新原子性
  • AMM / 预言机：说明价格从哪里来、为什么会失真
  • Lending liquidation：说明错误价格或状态如何最终落到清算和偿付风险上
  • 这样整轮回答会显得不是四个孤立知识点，而是一套能串起来的安全理解框架。

Day 6-7：Euler Finance 漏洞复现与 PoC

题卡 1：Euler Finance 攻击全貌——3 分钟怎么讲清？

问题

  • 面试官让你用 3 分钟讲清 Euler Finance 攻击链，你怎么组织？

短答

  • Euler 攻击不是权限泄露或经典重入，而是攻击者利用 donateToReserves 这条缺失健康检查的业务路径，主动让自己变成可清算状态，再通过清算奖励反向获利。
  • 本质是会计系统与风险约束失效叠加的逻辑漏洞。

展开

  • 讲述框架建议按时间线：背景 → 根因 → 攻击链 → 根本教训。
  • 背景：Euler 是借贷协议，核心安全前提是"账户在任何关键操作后都必须仍满足风险约束"。
  • 根因：donateToReserves 会改变抵押/债务/储备的会计关系，但缺了操作后的健康检查——主路径有检查，这条偏路径没有。这不是代码漏写，而是风险模型没有在所有状态变更路径上保持一致。
  • 攻击链：
    • 资金准备：flash loan 获取大额资金，做大借贷头寸
    • 状态畸变：调用 donateToReserves 把自己推入不健康区间
    • 清算获利：利用清算奖励机制从系统提取净利润
  • 教训：协议不是在单个权限点上失守，而是在系统性风险边界上失守。安全规则必须被所有入口统一执行。

题卡 3：为什么攻击者主动把自己推到可清算状态？

问题

  • 正常用户都想远离清算，攻击者为什么反其道而行？

短答

  • 因为清算伴随着清算奖励。杠杆放大头寸 + 故意制造可清算状态 → 清算奖励 > 自身损失 → 净利润。

展开

  • 清算机制给清算人 bonus 以激励坏账处理，这在正常场景下合理。
  • 但如果协议允许攻击者自己送自己进危险区，再从清算环节吃到奖励，那清算奖励就从"防御机制"变成"套利入口"。
  • Euler 攻击的反直觉核心：攻击者不是在躲清算，而是在设计一场对自己有利的清算。
    • 先把头寸做大
    • 用 donateToReserves 改变安全状态
    • 让自己变成可清算
    • 利用清算奖励机制吃出净收益

题卡 5：PoC 怎么拆、怎么验证、怎么写断言？

问题

  • 如果要写一个高质量的 Euler PoC，应该怎么组织，关键断言有哪些？

短答

  • 按三段拆：资金准备 → 状态畸变 → 清算获利。每段各自验证一个关键问题。
  • 断言不只是"盈利多少"，要证明协议不变量被打穿的过程。

展开

  • 资金准备：主网分叉 + flash loan，复现攻击者如何做大初始头寸。重点：资产、借贷关系和初始状态搭建正确。
  • 状态畸变：触发 donateToReserves，观察抵押 / 债务 / 健康状态变化。重点：证明协议允许了不该出现的状态。
  • 清算获利：复现清算动作、奖励结算和资产提取。重点：异常状态能转化成净利润。
  • 关键断言（把攻击脚本升级为漏洞解释器）：
    • 操作前后用户的抵押 / 债务 / eToken 余额变化
    • donateToReserves 前后健康状态变化
    • 原本不可清算的账户何时变成可清算
    • 清算后奖励与损失的净差额
  • Foundry 环境要求：fork 到正确区块高度、合约接口可访问、关键调用链走通。先保证"能跑、能观测、能断言"。
  • 断言的面试价值：说明你不是抄 PoC，而是理解状态机如何失守。

题卡 7：如果我是审计员，如何早期发现这类漏洞？

问题

  • 面试官追问"你会怎么在审计中发现类似问题"，怎么回答？

短答

  • 核心方法：枚举所有能改变头寸健康度的路径，逐一检查是否都有统一的健康状态校验。
  • 不要只盯主路径，要像攻击者一样问"哪些偏门操作也会动到风险相关的状态变量"。

展开

  • 审计清单思路：
    • 列出所有涉及抵押 / 债务 / 储备变化的函数（不只是 borrow/repay/withdraw）
    • 检查每条路径末尾是否都有健康检查
    • 特别关注"看似无害"的辅助函数（donate、transfer、migrate）
    • 验证风险模型在所有入口上是否一致执行
  • 不变量驱动审计：先定义协议的核心不变量（如"任何操作后账户必须健康或被阻止"），再反向验证每条路径是否维护了这些不变量。
  • 修复思路：在所有改变头寸风险的路径末尾统一加入健康检查，或提取成 modifier/internal function 确保不遗漏。

题卡 8：checkLiquidity() / deferLiquidityStatus 为什么是这次事故真正的风控闸门？

问题

  • 如果面试官继续往源码层追，checkLiquidity() 到底为什么这么关键？

短答

  • 因为它不是装饰性函数，而是 Euler 在账本变化后统一触发风险检查或挂脏标记的闸门。
  • donateToReserves() 致命就致命在：它改了用户风险相关状态，却既没立即检查，也没把账户标成稍后必须结算。

展开

  • Euler 不只是“有一个 RiskManager”，而是通过 checkLiquidity() 把很多状态变更路径统一收口到风控边界。
  • 它会先看 deferLiquidityStatus。
  • 如果当前没有延迟检查上下文，就直接把账户送去 requireLiquidity(account) 做硬校验。
  • 如果当前处在允许延迟结算的上下文里，它至少也会把状态从 CLEAN 标成 DIRTY，意思是“这个账户已经被改动过，稍后必须补总检查”。
  • 也就是说，这里保护的不是某个局部函数，而是“所有会影响头寸健康度的路径，最后都要经过统一闸门”。
  • donateToReserves() 的问题就在于它改了 balance 这类风险相关状态，却绕开了这个闸门：
  • 没有立即 requireLiquidity()
  • 没有留下延迟检查的脏标记
  • 等于让协议接受了一个本不该暂时存在的危险状态组合。

题卡 9：为什么 PoC 里必须拆成 violator 和 liquidator 两个角色？

问题

  • 为什么不能让攻击者自己清算自己？PoC 里拆双角色只是写法习惯吗？

短答

  • 不是写法习惯，而是协议约束决定的。
  • Euler 的清算逻辑禁止 self-liquidation，所以 PoC 必须把“持有坏账仓位的人”和“执行清算的人”拆开。

展开

  • 很多初看 Euler 的人会把攻击理解成“攻击者自己把自己清算掉赚奖励”，但链上实现不是这么跑的。
  • 协议会把不健康仓位的账户视为 violator。
  • 真正发起清算、拿到折价收益的人是另一个角色 liquidator。
  • 这也是为什么高质量 PoC 不能只写“跑通攻击脚本”，而要把协议约束也忠实还原出来。
  • 如果你把这两个角色混成一个，面试官很容易追问你是不是只是看了攻击文章，没有真正对着合约约束做复现。
  • 所以这道题的加分点在于：你不仅知道攻击链，还知道协议本身不允许哪种偷懒实现。

题卡 10：为什么既做 Foundry fork test，也做 Tenderly 广播调试？

问题

  • 既然本地 fork test 已经能跑通，为什么还要去 Tenderly 广播？

短答

  • 因为两者验证的不是同一件事。
  • Foundry fork test 更适合快速验证 PoC 成立，Tenderly 广播更适合把真实交易层次和状态变化逐层 debug 出来。

展开

  • Foundry fork test 的优势是快、可控、适合反复写断言。
  • 你可以在历史区块直接拉起状态，把攻击拆段验证，确认“这条逻辑链确实成立”。
  • 但它不天然等于“我已经从交易和部署语义上把真实 exploit 走了一遍”。
  • Tenderly 广播的价值在于：
  • 真实部署 exploit 合约
  • 真实发送调用
  • 在 Debugger 里逐层看 trace、storage 和中间状态
  • 这特别适合回答“你怎么确认脚本不是在本地自欺欺人”这种追问。
  • 还有一个很容易被忽略的细节：脚本里如果先部署再调用，广播时常常需要 --slow。
  • 原因不是性能，而是为了避免第二笔调用在第一笔部署确认前发出，导致你打到一个当时还没代码的地址。
  • 这个细节很工程化，面试里说出来会明显提高可信度。

Day 8-9：Sentiment 2023-04-04 基于 Balancer read-only reentrancy 的攻击复现

题卡 1：这个案例高层上到底发生了什么？

问题

  • 如果让我先用高层语言讲这个案例，我该怎么开口？

短答

  • 这是一个“上游池子状态短暂失真，下游借贷协议把失真读值当真”的真实攻击案例。
  • 更具体地说，攻击者利用 Balancer 的 read-only reentrancy / state desynchronization，让 Sentiment 在错误时机读取了错误估值，最终完成高估值借贷并获利。

展开

  • 这个案例不是单纯的闪电贷拉盘，也不是传统意义上“回调后把状态写坏”的重入。
  • 真正的关键是：上游系统在某个中间时刻，暴露出了一个看起来是 view 的、但实际已经不可信的状态快照。
  • 下游 Sentiment 把这个快照当成真实价格或真实估值输入，继续做风险判断和借贷决策。
  • 于是攻击链就形成了：
  • 上游状态失真
  • 下游信任错误数据
  • 攻击者利用错误估值借出真实资产
  • 所以这个案例最值得记住的不是某个函数名，而是“只读不等于安全，读错快照也能造成真钱损失”。

题卡 2：为什么这是“只读重入”，而不是普通重入？

问题

  • 这个案例为什么会被归类成 read-only reentrancy？

短答

  • 因为攻击核心不是反复进入去重复扣款或直接写坏同一状态，而是在重入窗口里读取到了一个中间态的错误数值。
  • 也就是说，真正被武器化的是“错误读取”，不是“重复写入”。

展开

  • 传统重入更容易让人想到：
  • 先外部调用
  • 再回调进入
  • 然后重复提款或重复消费资源
  • 只读重入则更隐蔽。
  • 它可能没有直接把某个关键状态永久写坏，但它让外部系统在错误的时机读到了错误的状态。
  • 在 Balancer / Sentiment 这类案例里，危险不在“读函数会改状态”，而在“被读取的状态本身处于去同步中间态”。
  • 下游协议如果把这个读值当成可信输入，就会在自己的系统里做出真实、可结算的错误决策。
  • 所以 read-only reentrancy 的关键不是“只读函数有副作用”，而是“中间态被下游当成了稳定真相”。

题卡 3：Balancer 这边的根因是什么？

问题

  • 如果把上游根因讲清楚，Balancer 这边到底出了什么问题？

短答

  • 根因是 join/exit 过程中某些关键状态的更新不是一个外部完全不可见的原子快照。
  • 这导致在特定重入窗口里，外部读取能看到“部分状态已经变了、部分状态还没变”的去同步状态。

展开

  • 在这类池子协议里，很多价格、份额和估值都依赖多个状态量共同成立。
  • 一旦这些状态量没有在外部完全不可见的前提下同步完成更新，就会出现中间态。
  • 比如可能出现：
  • totalSupply 已变化
  • balances 还没同步
  • 或某些内部缓存值还停留在旧状态
  • 单看每个变量本身都“像是合法值”，但它们组合起来已经不再代表一个真实可结算的池子状态。
  • 这就是所谓 state desynchronization。
  • 攻击者并不一定要永久篡改这些值，只要能卡在这个窗口里，让下游读到它，就足够了。

题卡 4：Sentiment 为什么会中招？

问题

  • 上游池子出了问题，为什么最终亏钱的是 Sentiment？

短答

  • 因为 Sentiment 把上游返回的估值当成了可信风险输入，用它决定账户价值、借贷额度或清算边界。
  • 一旦这个输入失真，Sentiment 的风控系统就会在自己这边“合法地”做出错误结算。

展开

  • 借贷协议本质上是价格和风控驱动系统。
  • 它不只看“你有几个 token”，更看“这些 token 当前值多少钱”。
  • 所以只要 Sentiment 依赖的估值源被抬高，它看到的就是：
  • 你的抵押价值变高了
  • 你的安全边际变厚了
  • 你似乎还能借出更多资产
  • 如果这个估值其实只是上游中间态造成的幻觉，那 Sentiment 就是在错误前提下放出了真实资产。
  • 所以从系统角度讲：
  • Balancer 暴露了错误快照
  • Sentiment 错信了这个快照
  • 攻击者把“读值错误”兑现成了“真实借款利润”
  • 这也是为什么 DeFi 组合性风险这么高：一个协议的只读问题，可能变成另一个协议的真金白银损失。

题卡 5：这条攻击链应该怎么拆？

问题

  • 如果我要把这个真实攻击讲成一条清晰链路，应该怎么拆？

短答

  • 最清晰的拆法是四步：
  • 制造上游中间态
  • 在中间态窗口读取错误估值
  • 用错误估值驱动下游借贷决策
  • 提取真实资产并完成获利

展开

  • 第一步，攻击者先构造能触发 Balancer 池子进入去同步中间态的动作。
  • 第二步，在这个窗口内，通过重入或相关调用路径让下游读取池子估值。
  • 第三步，Sentiment 把这个失真估值作为风险输入，错误地认为攻击者抵押品价值更高。
  • 第四步，攻击者基于这份虚高估值借出真实资产，随后完成整条攻击闭环。
  • 所以这条链里真正值得抓住的不是“某一个函数名”，而是数据流：
  • 上游错误状态
  • 错误读值
  • 下游错误信任
  • 真资产流出
  • 你只要把这个数据流讲顺，案例就不会碎。

展开

题卡 11：getPrice() 和 borrow() 分别是这条链上的哪两个边界？

问题

  • 如果我要把这个案例讲得更像源码审计，而不是复述文章，最该点哪两个函数？

短答

  • 上游边界是 WeightedBalancerLPOracle.getPrice()，它把 Balancer 的瞬时读数转成估值。
  • 下游边界是 AccountManager.borrow()，它把这份估值直接兑换成真实放款。

展开

  • 这条攻击链最容易讲散，解决办法就是抓住两个边界函数。
  • 第一是 getPrice()：
  • 它一边从 Vault.getPoolTokens(...) 读取底层余额
  • 一边从 BPT 合约读取 totalSupply()
  • 一旦这两个值不来自同一个干净快照，估值就会飘。
  • 第二是 borrow()：
  • 它不是“先算额度再借”，而是先 lendTo(account, amt)，最后才 isAccountHealthy(account)。
  • 这意味着只要健康检查吃到了脏价格，虚高抵押价值就会立刻兑换成真资产。
  • 所以面试里最稳的说法是：
  • getPrice() 是脏数据进入系统的边界
  • borrow() 是脏数据兑现成真实资金流出的边界

题卡 12：为什么 nonReentrant 没挡住这次攻击？

问题

  • Sentiment 不是有 nonReentrant 吗，为什么还是被打穿了？

短答

  • 因为这把锁保护的是 AccountManager 自己的执行栈，不保护外部依赖协议此刻是不是正处于脏状态。
  • 这次问题不是“我还没跑完你又进我”，而是“我去信了一个外部协议半路上的错快照”。

展开

  • 很多人看到 nonReentrant 就会产生安全错觉，以为“那重入类问题应该都挡住了”。
  • 但这把锁只能防一种情况：
  • Sentiment 自己某个受保护函数还没跑完
  • 攻击者又想回头进入 Sentiment 的另一个受保护函数
  • 这次真正制造中间态的关键动作发生在 Balancer 那边。
  • 也就是说，攻击窗口是在外部协议的 exitPool() 过程中出现的，而不是 Sentiment 自己的持锁执行栈里出现的。
  • 攻击者回到 Sentiment 调 borrow() 时，Sentiment 本地这把锁并不一定处于“施工中”状态。
  • 所以这次问题的本质不是“本地锁失效”，而是“本地锁根本不覆盖跨协议脏读”。

题卡 13：为什么攻击者要走 ETH 退出路径？

问题

  • 为什么案例里会特别强调 exitPool() 走 ETH 退出路径？

短答

  • 因为如果退出拿到的是普通 ERC20，标准 transfer 不会把执行权交还给攻击者。
  • 走原生 ETH 路径时，Balancer 会做一次 value call，攻击者才能在 receive()/fallback() 里卡进中间态窗口。

展开

  • 这类案例里真正值钱的不是“退池”这个动作本身，而是“退池过程里有没有一个可重入窗口”。
  • 如果只是普通 ERC20 转账，执行流通常不会自动回到攻击者自定义逻辑。
  • 但如果 Balancer 把资产以原生 ETH 形式退给攻击者合约，就会触发一次 value call。
  • 这时攻击者的 receive() 或 fallback() 就能在 exitPool() 还没完全收尾时执行。
  • 也正是在这个时间点：
  • BPT totalSupply() 已经缩了
  • Vault 里的 balances 还没同步扣完
  • 攻击者于是能趁这个窗口去让 Sentiment 读取价格。
  • 所以你可以把它讲成：ETH 退出路径不是“可有可无的实现细节”，而是把中间态窗口真正暴露给攻击者的执行机制。

题卡 14：priceBefore / priceDuring / priceAfter 这组数字到底证明了什么？

问题

短答

  • 核心不是“after 比 before 高一点”，而是 priceDuring > priceBefore。
  • 真正证明脏读成立的是回调窗口里的异常高价，而不是最终状态更新后的价格变化。

展开

  • 这组数据最适合拿来回答“你凭什么说这里读到了脏价格”：
  • priceBefore = 220120691697985632
  • priceDuring = 221005560649599490
  • priceAfter = 220122172406069243
  • 面试里最容易说错的是把 after 也当成漏洞核心证据。
  • 更准确的讲法是：
  • before -> during 的明显抬升，说明回调窗口里读到的不是正常价格，而是中间态脏读
  • after -> before 的细小差别，不一定说明漏洞持续存在，更可能是退池后真实新平衡、池子组成变化和舍入误差
  • 所以这组数字的意义不是“价格一直异常”，而是“在某个短暂窗口里，价格被错误抬高过”
  • 这恰好对应只读重入的本质：窗口短，但足够致命。

题卡 15：这类问题的修复思路应该怎么讲？

问题

  • 如果面试官继续追问“那怎么修”，这类案例应该怎么回答才不空？

短答

  • 修复要分上游和下游两层。
  • 上游要避免把可重入窗口里的 spot 读数暴露成可信价格，下游要避免在价格可信性没确认前就把资产先放出去。

展开

  • 上游 Balancer 侧的思路是：
  • 不把当前 view 读数直接当成天然安全
  • 在取价前做一次 no-op 的 reentrancy check，例如 manageUserBalance([]) 这类等价检查
  • 或者干脆避免让借贷风控路径直接依赖这种可重入窗口里的 spot 价格
  • 下游 Sentiment 侧的思路是：
  • 借款前先做额度预演和价格安全校验
  • 不要把 lendTo() 放在健康检查之前
  • 对这类 LP 抵押增加 oracle freshness、borrow cooldown 或专门的 reentrancy / stale-price 防护
  • 面试里最好的总结是：
  • 上游修的是“别暴露脏快照”
  • 下游修的是“别在快照可信性未确认前先放真钱”

题卡 10：如果面试官追问 Uniswap V2 对 FoT 代币为什么不兼容，该怎么答？

问题

  • 如果面试官追问 Uniswap V2 对 Fee-on-Transfer（FoT）代币为什么不兼容，最该抓哪条执行流？

短答

  • 根因首先在 swap 执行流
  • 普通 Router 会按“名义输入量”预先计算 amountOut，但 FoT 代币真正转进 Pair 的数量会更少。
  • UniswapV2Pair.swap() 末尾会按真实 balance 反推 amountIn 并做 K 值校验；实际到账不足时，这个 require 会失败，典型报错就是 UniswapV2: K。

展开

  • 这题先把两条路径分开：
  • swap：讨论“为什么默认换币会失败”时，核心应盯住这条路径
  • removeLiquidity / burn：也可能因为 FoT 导致到账和预期不一致，但那不是这道题的主根因
  • 普通 swapExactTokensForTokens 这类函数，会先基于储备和输入参数算出整条路径的 amounts[]
  • 但如果输入 token 是 FoT 代币，用户 transferFrom 进 Pair 时会被扣税，Pair 实际收到的数量小于 Router 预估输入
  • 到了 Pair.swap() 内部，协议并不信任外部“声称输入了多少”，而是读取当前 balance0 / balance1，用余额差分反推真实 amountIn
  • 随后执行带手续费的不变量检查；因为真实输入变小，但输出还是按未扣税前的名义输入量算出来的，最终就会在 swap 末尾的 K 值校验处 revert
  • 所以如果一定要落到源码终点，swap 场景下更直接的触发点是 UniswapV2Pair.swap() 末尾的 K 值检查，而不是泛泛地说“FoT 不兼容”
  • SupportingFeeOnTransferTokens 系列函数的意义，就是放弃信任名义输入量，改为在每一跳读取 Pair 的实际到账余额差，再动态计算可输出数量
  • 面试里最容易暴露源码不熟的三个信号是：
  • 把 swap 和 removeLiquidity / burn 的执行流混在一起
  • 说不出最终是哪一个 require 触发 revert
  • 把 FoT 问题和 EIP-150 的 63/64 Gas 转发规则，或和协议手续费概念混为一谈

题卡 2：Front-running 的原理是什么？

问题

  • 如果面试官直接问 front-running 原理，我怎么答更像懂链上机制？

短答

  • Front-running 本质上是利用公开 mempool 和排序权，在目标交易前插入一笔会改变其结果的交易，从价格、成交路径或状态变化里获利。

展开

  • 交易在正式上链前，通常会先进入公开或半公开的等待区。
  • 如果攻击者能看到用户准备买什么、卖什么、清算什么，就可以在它前面插入自己的交易。
  • 这样做的前提是：目标交易的结果会受前一笔交易影响。
  • 在 EVM 世界里这很常见，因为大家共享同一个全局状态。
  • 典型场景包括：
  • 抢先买入导致用户成交价变差
  • 抢先清算拿走本该属于别人的机会
  • Sandwich：先把价格推歪，再让用户在坏价格成交，最后自己反向平仓
  • 所以 front-running 不是孤立现象，它本质上是“公开可观测交易意图 + 排序权可变现”。

题卡 3：Solidity 0.8+ 之后，整数溢出还要怎么答？

问题

  • 现在编译器都自带检查了，面试再问整型溢出，我该怎么回答才不显得过时？

短答

  • 0.8+ 之后，默认算术溢出会触发 revert，所以“无意识溢出”风险明显下降。
  • 但这不等于可以不讲，因为 unchecked、类型缩窄、内联汇编和外部库边界里，溢出仍然可能变成真实问题。

展开

  • 如果只回答“0.8 之后没事了”，通常会显得你停留在表面。
  • 更完整的答法是：
  • 默认情况下，+ - * 等算术操作会自动做溢出检查，异常会 revert
  • 但如果开发者为了省 gas 手动用了 unchecked，那又回到旧语义
  • 即使主逻辑不溢出，uint256 -> uint128 这类 downcast 也可能在类型转换时截断高位
  • 内联汇编、某些底层库或跨合约接口边界，也可能绕开你对“默认安全”的直觉
  • 所以现在面试里更好的表述是：整数溢出已经从“高频默认坑”变成“边界条件和优化路径上的坑”
  • 这会比单纯背 SafeMath 或背版本差异更到位。

题卡 1：为什么 Searcher 不需要依赖 Event，也能发现可清算状态或套利空间？

问题

  • 如果链上事件还没发出来，Searcher 为什么还是能提前发现机会？

短答

  • 因为 Searcher 盯的是 mempool 里的待打包交易，不是链上已经确认的 Event。
  • 它会对 pending tx 做状态模拟，直接看“这笔交易如果打进去，储备量、价格、健康度、可清算性会怎么变”。

展开

  • 这是黑暗森林里最关键的认知之一：机会的发现往往早于事件广播。
  • Event 只有在交易真正上链执行后才会被写进日志。
  • 但 Searcher 的工作时点更早，它看到的是：
  • 用户刚发出的 pending tx
  • 这笔 tx 的 calldata、目标合约、参数
  • 基于当前链状态对这笔 tx 的模拟结果
  • 如果模拟后发现：
  • 某个账户会从健康变成可清算
  • 某个池子的价格会短时偏移
  • 某条路径会产生可套利价差
  • 它就能在事件还不存在时，先构造抢跑、夹子或回跑交易。
  • 所以安全工程师不能把“有没有 Event 广播”当成机会是否可见的分界线。
  • 真正的分界线是：这笔待打包交易是不是已经暴露在可观察 orderflow 里，以及别人能不能模拟出你的状态变化。

题卡 2：MEV 到底是什么？

问题

  • 如果面试官先问一句“MEV 是什么”，我该怎么答得既不空泛又不跑偏？

短答

  • MEV 现在通常指 Maximal Extractable Value，可以理解为“谁能影响交易顺序，谁就能从中额外榨出收益”。
  • 它不是单一攻击，而是一大类利用排序权、打包权和状态先后关系获利的现象。

展开

  • 早期很多人会说它是 Miner Extractable Value，因为矿工最直接控制交易打包顺序。
  • 现在更准确的说法是 Maximal Extractable Value，因为真正参与这条链路的不只是矿工，还有：
  • searcher
  • builder
  • validator
  • relay / orderflow 提供方
  • 它的本质不是“某个合约突然多了一个漏洞”，而是：
  • 交易都运行在同一份全局状态上
  • 谁先执行、谁后执行、谁被夹在中间，会直接改变收益分配
  • 于是排序权本身就变成了可以货币化的资源
  • 对安全分析来说，MEV 最重要的两个点是：
  • 第一，它不依赖链上 Event 发出来后才成立，很多机会在 pending tx 暴露时就已经能被模拟出来
  • 第二，它不只发生在 swap 上，也会发生在清算、套利、预言机窗口、NFT 铸造、桥接等所有强顺序敏感场景
  • 所以面试里一句更像样的话是：
  • MEV 本质上是排序权和状态先后关系的货币化。

题卡 3：Front-running、Back-running、Sandwich 到底怎么区分？

问题

  • 如果面试官不满足于“抢跑”两个字，我该怎么把三类典型 MEV 说清？

短答

  • Front-running 是在目标交易前插入会改变其结果的交易。
  • Back-running 是在目标交易后立刻接一笔，吃它制造出的状态变化。
  • Sandwich 是前后各打一笔，把用户夹在中间获利。

展开

  • 三者的共同前提都是：交易顺序本身会改变收益。
  • Front-running 最常见于：
  • 抢先买入
  • 抢先清算
  • 抢先执行某个会被状态变化影响的动作
  • Back-running 更像“吃尾气”：
  • 别人的交易先把状态打歪
  • 你在后面立刻接上，把这个新状态的利润吃掉
  • Sandwich 则是组合型打法：
  • 先在用户前面打第一笔，把价格推向对自己有利的方向
  • 用户在更差价格成交
  • 再在用户后面打第二笔，把头寸平掉并锁定利润
  • 所以面试里如果你只说“MEV 就是抢跑”，通常不够。
  • 更像专业回答的说法是：MEV 的核心是排序权可变现，而 front-run / back-run / sandwich 是排序权变现的三种常见形态。

题卡 4：Flashbots 和 Private RPC 到底解决了什么？又解决不了什么？

问题

  • 面试官如果追问“那用 Flashbots 或私有 RPC 不就行了”，我该怎么答？

短答

  • 它们主要解决的是“公开 mempool 暴露交易意图”这个问题，减少被普通 Searcher 看到和抢跑的概率。
  • 但它们不是万能护城河，因为它们不改变合约本身的脆弱设计，也不保证所有 orderflow 都绝对私密。

展开

  • 公开 mempool 的核心问题是：你的意图先暴露，别人就有时间模拟和构造反制交易。
  • Flashbots / Private RPC 的思路是尽量把交易从公开广播改成私下投递给 builder / relay / validator。
  • 这样可以减少：
  • 普通抢跑
  • public mempool 里的夹子攻击
  • 某些基于监听 pending tx 的机会捕获
  • 但它解决不了 3 件事：
  • 第一，合约设计本身如果可被 deterministic 地利用，私有投递只是降低被看到的概率，不是修复漏洞
  • 第二，不是所有路径都天然私密，系统边界和信任对象本身也要考虑
  • 第三，如果状态变化一旦上链后仍然会被后续交易利用，它也挡不住链上确认后的 back-run 或后续抢占
  • 所以它更像“orderflow 保护层”，不是“协议安全性证明”。

题卡 5：Uniswap V2 里 Factory、Router、Pair 各自负责什么？谁发 PairCreated？

问题

  • 如果面试官要我精确讲 Uniswap V2 架构，我该怎么答才不会把接口说混？

短答

  • Factory 负责创建交易对并维护 pair 映射，PairCreated 事件也是它发的。
  • Router 主要负责路径编排、加减流动性和 swap 的用户入口。
  • Pair 才是真正持有储备量、LP 份额和 swap 规则的状态合约。

展开

  • 这类题最容易错在把“用户常调的入口”和“真正持状态的合约”混成一件事。
  • 更准确地讲：
  • Factory
  • 创建 Pair
  • 记录 token 对应的 pair 地址
  • 发出 PairCreated(token0, token1, pair, ...)
  • Router
  • 帮用户做路径选择和多步调用编排
  • 处理加流动性、移除流动性、代币换币等入口
  • 自己通常不持有 AMM 核心状态，更像无状态协调层
  • Pair
  • 持有 reserve0、reserve1
  • 管 LP token
  • 执行 mint、burn、swap
  • 更新储备并维持不变量
  • 所以如果面试官问“监听流动性池创建该监听谁”，准确答案是：监听 Factory 的 PairCreated，不是监听 Pair 自己。

题卡 6：mint、burn、swap 底层到底是怎么改储备量的？

问题

  • 如果继续追问 Uniswap V2，mint / burn / swap 这几个动作底层是谁在改储备量？

短答

  • 真正改 AMM 储备量的是 Pair 合约，不是 Router。
  • Router 负责把 token 转进正确地址并发起调用，Pair 再根据实际余额变化计算输入输出并调用 _update() 刷新 reserve0 / reserve1。

展开

  • 这题的关键是把“资金转移”与“储备记账”区分开。
  • 在 V2 里，常见流程是：
  • 用户通过 Router 进入
  • Router 把 token 转到目标 Pair
  • Pair 在 mint / burn / swap 内部根据当前余额与旧储备做计算
  • 最后通过 _update(balance0, balance1, reserve0, reserve1) 更新储备状态
  • 对应到三类动作：
  • mint
  • 用户把两边资产打入 pair
  • pair 计算应铸造多少 LP token
  • 更新 reserves
  • burn
  • 销毁 LP token
  • pair 把底层资产按份额退回
  • 更新 reserves
  • swap
  • 用户先把输入 token 转入 pair
  • pair 按不变量约束给出输出
  • 完成后更新 reserves
  • 所以更专业的表述是：Router 是入口和编排层，Pair 才是储备与定价状态的最终真相源。

题卡 7：reserve0/reserve1 为什么必须通过 _update() 和 balanceOf(this) 对齐？

问题

  • 面试官如果追问 Uniswap V2 状态机，你怎么解释 mint / burn / swap 结束后为什么一定要 _update()？

短答

  • reserve0/reserve1 是 Pair 的“记账快照”，balanceOf(address(this)) 是该时刻真实余额。
  • mint / burn / swap 都会先发生转账与计算，最后统一调用 _update(balance0, balance1, _reserve0, _reserve1)，把快照对齐到真实余额。
  • 只有完成这一步，下一笔交易和价格累积逻辑才能基于正确状态继续运行。

展开

  • 这题本质是区分两个世界：
  • reserve0/reserve1：上一次同步后的缓存状态（用于定价和校验）
  • balance0/balance1：当前交易执行到某一刻时，Pair 地址在 ERC20 上的真实余额
  • 在 V2 里，状态机是“先动作，后对账”：
  • mint：用户先把两种 token 转入 pair，合约按增量铸造 LP，最后 _update
  • burn：先销毁 LP 并把底层资产转出，最后 _update
  • swap：先乐观转出 amountOut，若有回调则先执行回调，再根据 balance 反推 amountIn 并做不变量校验，最后 _update
  • swap 的关键不是“信任传入参数”，而是以余额差分反推输入：

$$ amount0In = \max(balance0 - (reserve0 - amount0Out), 0) $$

$$ amount1In = \max(balance1 - (reserve1 - amount1Out), 0) $$

  • 然后用带手续费的不变量检查（0.3% fee）：

$$ (balance0 \cdot 1000 - amount0In \cdot 3) \cdot (balance1 \cdot 1000 - amount1In \cdot 3) \ge reserve0 \cdot reserve1 \cdot 1000^2 $$

  • 校验通过后再 _update，把新余额写回 reserve0/reserve1，并更新价格累计与 blockTimestampLast。
  • 这就是你在审计里必须抓住的核心：V2 的真实状态演进是“balance 先变化，reserve 后同步”。

题卡 9：如果不使用 EIP-1967，黑客会怎么利用 Storage Collision 接管 Proxy？

问题

  • 如果面试官继续高压追问，具体会怎么利用 storage collision 拿走代理合约控制权？

短答

  • 如果 proxy 把 implementation、admin 等关键地址随便放在普通 slot，比如 slot0 / slot1，而 implementation 的业务变量也恰好写到这些槽位，攻击者就能通过合法业务调用把关键槽位覆盖掉。
  • 一旦 implementation 或 admin 被覆盖成攻击者控制的值，proxy 的升级权或执行逻辑就会被接管。

展开

  • delegatecall 的危险点在于：代码在 implementation，storage 在 proxy。
  • 这意味着 implementation 里的状态变量布局，本质上是在解释 proxy 的 storage。
  • 如果没有像 EIP-1967 那样把关键元数据放到特殊 slot，而是粗暴地放在常规业务槽位：
  • slot0 可能存 proxy admin
  • slot1 可能存 implementation 地址
  • 同时 implementation 第一、第二个状态变量也可能落在 slot0 / slot1
  • 攻击路径就出现了：
  • 攻击者调用某个看似普通的业务函数
  • 该函数通过 delegatecall 在 proxy 上执行
  • 它把业务变量写进 slot0 / slot1
  • 结果等于把 proxy 的 admin / implementation 覆盖了
  • 一旦 implementation 被改成攻击者合约，后续所有调用都能被导向恶意逻辑
  • 一旦 admin 被改成攻击者地址，升级控制权也等于被拿走
  • 所以 EIP-1967 的价值不是“神奇防御一切”，而是把这些关键槽位放到普通编译器几乎不会自然碰到的位置，极大降低业务变量误撞上去的概率。
