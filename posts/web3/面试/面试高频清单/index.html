面试高频清单

这份清单用于最后冲刺，不代替完整题卡。使用方式：

  • 面试前 20 分钟：只看这份
  • 被某一类问题追问得发虚：再回对应题卡
  • 每次真实面试后：把新问法补进这里，再回填到题卡

1. EVM / Proxy / 调用语义

  • call / delegatecall / staticcall：storage、msg.sender、msg.value、是否允许写状态
  • storage layout：packing、mapping、dynamic array、slot 推导
  • storage collision：为什么本质是“同一 slot 被不同语义解释”
  • 不使用 EIP-1967 的真实风险：业务变量误撞 implementation / admin 槽位，进而覆盖升级控制权
  • Transparent / UUPS / Diamond：升级逻辑在哪、风险边界在哪
  • EIP-1967：为什么标准 slot 能降低碰撞风险
  • EIP-150：为什么是“保留上层 gas 控制权”，不是单纯背 63/64
  • permit：nonce、deadline、重放、钓鱼、前置执行

临场金句：

  • delegatecall 是“借别人的代码，操作自己的状态”。
  • 代理最怕的不是“能升级”，而是“升级后布局不兼容”。
  • EIP-150 保护的是调用者的控制权，不是用户的总 gas。

2. 重入 / 静态分析 / 交易模型

  • 不带 DeFi 术语定义重入：时序失控 + 临界资源竞争 + 状态原子性被破坏
  • 单函数 / 跨函数 / 跨合约 / 只读重入
  • UTXO vs Account：局部资源消费模型 vs 全局共享状态模型
  • 为什么 Account 模型更容易产生 MEV、清算竞争、排序依赖
  • Searcher 的关键能力：观察 pending tx、simulate 状态变化，不依赖 Event 广播
  • Slither 在做什么：IR、CFG、数据流、规则匹配、静态近似分析
  • 为什么静态分析需要人工复核：可达性、权限前提、经济可利用性
  • 自动化漏扫方案：采集、归一化、静态分析、组件识别、风险排序、PoC 验证、人工闭环

临场金句：

  • 重入不是 Solidity 特有 bug，而是状态机时序失守。
  • 工具给的是线索，不是判决。
  • UTXO 解决双花更自然，Account 组合性更强，但排序权也更值钱。

3. AMM / Oracle / Flash Loan / MEV

  • Uniswap V2：x * y = k、储备变化、手续费如何进模型
  • Uniswap V2 架构精度：Factory 创建交易对并发 PairCreated，Router 编排路径，Pair 持有 reserves 并在 mint / burn / swap 后 _update()
  • Uniswap V3：Tick、区间流动性、资本效率 vs 复杂度
  • TWAP：为什么比 spot 更难操纵，但不是银弹
  • Flash loan：为什么是资金放大器，不是根因
  • Callback：为什么是攻击真正展开的执行窗口
  • Oracle misuse：为什么问题核心常常不是“价格变了”，而是“依赖方错信了这个价格”
  • Front-running / sandwich：公开 mempool + 排序权 + 状态依赖
  • Flashbots / Private RPC：解决公开意图暴露，不等于修复协议脆弱性
  • 代码级防抢跑：commit-reveal、batch auction、延迟执行、签名报价 + expiry；minOut / deadline 更偏用户保护，不是根治

临场金句：

  • 闪电贷回答两层：为什么能打，为什么能做大。
  • 预言机安全不是“有没有喂价”，而是“信任模型对不对”。
  • 排序权本身就是可变现资源，这就是很多 MEV 的起点。

4. Lending / Liquidation

  • Aave / Compound 基础结构：存款凭证、借款利息、LTV、liquidation threshold、close factor
  • 清算为什么不是惩罚，而是维持系统偿付能力
  • 为什么“看起来安全”的账户会在状态变化后突然可清算
  • 风险参数不是静态值，而是价格、利息、市场状态共同决定的动态结果
  • 借贷协议为什么天然依赖预言机，也为什么最怕预言机

临场金句：

  • 借贷协议最危险的不是借钱动作本身，而是“真钱放款建立在什么输入上”。
  • 清算的本质是让坏账尽快被市场化处理。

5. 项目与 PoC

MemeScan

  • 一句话：不是普通爬虫，而是“可验证的安全分析系统”
  • 真实链路：Coordinator -> Sandbox -> Auditor -> Reporter
  • 亮点：动态仿真、Rug-Pull Replay、规则引擎 + LLM 分层、Streamlit 下的异步工程化
  • 不要说错：
  • Coordinator 不是最终深度分析决策者
  • 当前不是完全自治 Agent
  • DB 还没完整承接主链路
  • 可主动承认的局限：规则引擎仍偏 MVP、主链路仍以 Markdown 报告落盘为主、当前断言还需从“单身份单时刻”升级到“多身份多时刻多权限路径”

Uniswap V2 DEX

  • 一句话：不是只看过 Uniswap V2 原理，而是亲手实现过恒定乘积做市、流动性池和 LP 份额铸造/销毁
  • 核心要讲：
  • 你实现了什么
  • x * y = k 如何落到状态更新
  • 流动性池如何支持双边存入/提取
  • LP 份额如何铸造/销毁
  • 如果继续追问：
  • Factory / Router / Pair 的职责边界
  • mint / burn / swap 后是谁更新 reserves
  • 你的实现与真实 Uniswap V2 的简化取舍

Euler

  • 一句话：donateToReserves 漏掉统一流动性检查，使攻击者能主动制造坏账仓位，再通过清算奖励获利
  • 核心边界：
  • checkLiquidity() / deferLiquidityStatus
  • donateToReserves()
  • 必须会讲：
  • 为什么不是权限漏洞 / 普通重入
  • 为什么要拆 violator / liquidator
  • 为什么要区分 Foundry fork test 和 Tenderly 广播

Sentiment

  • 一句话：上游 Balancer 脏读，下游 Sentiment 把错价格直接兑现成真钱放款
  • 核心边界：
  • WeightedBalancerLPOracle.getPrice()
  • AccountManager.borrow()
  • 必须会讲：
  • 为什么 nonReentrant 没挡住
  • 为什么要走 ETH 退出路径
  • priceBefore / priceDuring / priceAfter 里真正值钱的是 during
  • 修复要分上游和下游两层

6. 真实面经补位

  • 自动化漏扫方案怎么设计
  • front-running 原理
  • Solidity 0.8+ 后整数溢出怎么答
  • UTXO vs Account
  • callback 为什么关键
  • 哪个项目最能代表你的成长

7. 最后 5 分钟自检

  • 我能把每个项目讲成“问题 -> 核心机制/根因 -> 证据 -> 修复/收获”
  • 我能把每个漏洞讲成“根因 vs 放大器”
  • 我能把抽象概念落到具体函数、状态和断言
  • 我知道哪些地方能讲满，哪些地方应该主动承认边界
  • 我能在 3 分钟内讲清一个代表性项目或复现案例
