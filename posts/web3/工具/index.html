Foundry（偏向合约测试）

理解这三个角色的关系：

  1. Anvil（铁砧）：它是游戏服务器。它会在你电脑本地启动一条区块链。只要它开着，你的世界就存在；它关了，世界就消失。
  2. Forge（锻造）：它是造物主工具。用来编写和把合约（比如一个自动售货机）放到服务器里。
  3. Cast（施法）：它是游戏手柄。用来和合约互动（比如查询余额、按下按钮）。

设置环境变量，部署合约上链

RPC 就是让程序（Foundry、脚本、工具）连接区块链节点的 URL。
你要和 Sepolia testnet 通信，就必须有 Sepolia 的 RPC。 用它才能部署合约、发送交易、读区块

export RPC_URL=https://eth-sepolia.g.alchemy.com/v2/你的APIKEY
forge script ./script/DeploySimpleStorage.s.sol --broadcast  --rpc-url $RPC_URL  --private-key $PRIVATE_KEY

//anvil本地链模板如下

测试

forge test --match-path ./test/unstoppable/Unstoppable.t.sol -vv //运行指定测试合约
forge test -vv //只有一个测试文件直接
forge test --match-test 函数 -vv //测试单个函数
forge test --match-path test/poc.t.sol --match-test testExploit -vv


forge test 默认不显示 console.log 的输出，需要加 -v 参数提高日志等级：

-v → 显示基本信息
-vv → 显示 console.log 输出 
-vvv → 还显示失败测试的 trace
-vvvv → 显示所有 trace

因为 Foundry 的测试会自动读取 test 中的脚本，但所有合约逻辑仍然来自 src 文件夹。

换句话说：

  • src/ 里是你要审计 / 攻击 / 部署的真实合约
  • test/ 里是 Foundry 自动部署 src 合约、构建攻击场景、执行 exploit 的脚本

代码干的事是：

  1. 执行 test/unstoppable/*.t.sol 文件
  2. 测试脚本内部
    • 使用 src 里的 UnstoppableLender.sol / DamnValuableToken.sol
    • 然后自动部署它们
    • 自动设置初始状态
    • 自动运行 exploit
  3. 最后验证 challenge 是否被解掉

测试模式

Fork Test（分叉测试）

Fork Test = 基于真实链状态的本地 EVM 沙箱。

Foundry 允许你从主网/测试网某个区块高度创建一份“链状态快照”，然后在本地执行交易。

换句话说：

  • 代码在本地跑
  • 状态从链上来
  • 读是真实的，写是本地的，不影响链上
使用
  1. 在 foundry.toml 里写 RPC
[rpc_endpoints]
sepolianet = "https://eth-sepolia.g.alchemy.com/v2/DPuDLu_tUJwkWFmyEsT7x"
  1. 测试代码中这样调用（注意选择的区块要注意高度，否则有些调用合约不在链上）
contract FundMeTest is Test {
    FundMe fundme;
    address constant SEPOLIA_ETH_USD = 0x694AA1769357215DE4FAC081bf1f309aDC325306;

    function setUp() external {  //部署合约
        //fork测试
        vm.createSelectFork("sepolianet", 9_000_000); //fork 到 Sepolia（会从 foundry.toml 里读 rpc_endpoints.sepolia）
        fundme = new FundMe(SEPOLIA_ETH_USD);// 2. 用真实的 priceFeed 地址部署 FundMe
    }
  }

monk

Mock = 在测试环境中，用合约模拟真实外部依赖的行为

特点：

  • 不访问链上真实合约
  • 不依赖预言机、治理、多签等外部状态
  • 可控（返回值可随时设定）
  • 可预测（不随链上状态变化）
  • 允许边界条件/攻击场景测试

Web3 测试中常 mock 的组件：

  • Price Feed（Chainlink AggregatorV3Interface）
  • ERC20 token
  • 外部协议接口（Uniswap、Aave、Lido）
  • Oracle 和跨链模块

适用场景：
你需要测试自己写的逻辑，而外部依赖本身不是测试重点。

Foundry 常用作弊码 (Cheatcodes) 指南

这些代码只能在测试环境 (.t.sol) 中运行，能够让我们像“上帝”一样操控区块链状态。

作弊码 (Cheatcode)功能解释常见使用场景
vm.addr(privateKey)私钥推导地址：输入一个 uint256 格式的私钥，计算并返回它对应的以太坊地址。当我们意外获得私钥时，需要先算出它的公钥地址以便后续伪装。
vm.prank(address)伪造身份 (单次)：改变下一笔外部调用的 msg.sender。执行完一行代码后，身份自动恢复。我们需要用多个不同的 Oracle 账号依次发送价格时，比如：
vm.prank(oracle1);
oracle.postPrice(...)
vm.startPrank(address)
vm.stopPrank()		伪造身份 (持续)：在这两个调用之间的所有外部调用，msg.sender 都会被固定为目标地址。需要同一账号连续执行多步操作时。比如先买入，再授权，再卖出，全程保持 player 身份。
vm.warp(timestamp)时间穿越：强制修改当前区块链的时间戳 (block.timestamp)。绕过时间锁、延迟期。比如在 Selfie 挑战中，排队提案后强制快进 2 天（+ 2 days）来执行操作。
vm.deal(address, amount)凭空生钱：直接修改某个地址的 ETH 余额（无论他是谁）。初始化测试环境时，给攻击者或金库塞初始资金。
vm.roll(blockNumber)修改区块高度：强制更改当前合约的区块号 (block.number)。攻击依赖于特定区块才生效的治理投票或解锁逻辑。
vm.label(address(Smoofs), "Smoofs")标记地址，方便在调试日志中查看

Slither（智能合约静态分析工具）

Slither 是一个基于 Python 编写的 Solidity 跨平台静态分析框架。它可以在不运行代码的情况下快速扫描智能合约，发现常见的安全漏洞（例如重入漏洞、未初始化的变量、权限控制缺陷等）。

它就像代码的“全自动 X 光机”，找出隐藏的病灶。

2. 基本使用方法

进入包含目标合约的项目根目录（它能自动识别 Hardhat, Foundry, Truffle 等框架），然后执行扫描：

# 自动识别框架并扫描整个项目的所有合约
slither .

# 只扫描特定的单个合约文件
slither src/MyContract.sol

3. 常见高级命令

有时默认扫描输出的无关信息太多，你可以通过参数过滤或生成可视化报告：

# 屏蔽掉低危漏洞和纯信息提示（只看中高危漏洞，减少噪音）
slither . --exclude-informational --exclude-low

# 把扫描结果输出为 JSON 格式，方便用其他脚本处理
slither . --json results.json

# 【需要额外安装 Graphviz 工具】打印合约的继承图
slither src/MyContract.sol --print inheritance-graph

# 【需要额外安装 Graphviz 工具】打印函数控制流图（CFG）
slither src/MyContract.sol --print cfg

4. 如何看懂输出结果？

Slither 会给终端输出不同颜色的警告，按严重等级分类：

  • 红色 (High)：高危漏洞。如重入攻击 (reentrancy)、任意外部调用 (arbitrary-send) 等，看到红色必须仔细排查。
  • 黄色 (Medium)：中危漏洞。如对不可信的返回数据未作校验等。
  • 绿色 (Low / Informational)：低危漏洞或代码规范提示。如未使用的变量 (unused-state)、solc 编译器版本未锁定、可以使用 immutable 或 constant 优化 Gas 等。

终端输出的最后会有一段总结，如： x issues (y high, z medium)，你可以根据这个直观看到合约当前的安全状况。每次跑完针对对应的行号去修改即可。

Tenderly + Foundry

用 Tenderly 作为 fork RPC 跑测试

让 Tenderly 作为 fork RPC 的意思是：

  • 测试代码仍然在本地 Foundry 里执行
  • 链上状态来自 Tenderly 的 Virtual TestNet / fork
  • 你本地通过 RPC 去读取和修改这条 fork 的状态

也就是说：

  • forge test --rpc-url $TENDERLY_RPC_URL ... 不是“让 Tenderly 帮你跑测试”
  • 它是“让 Foundry 把 Tenderly 当成远程 fork 节点来用”

例如：

source .env && forge test --match-test testExploit -vvv --rpc-url $TENDERLY_RPC_URL --fork-block-number 16817995

这条命令的含义是：

  1. forge test 在本地运行 .t.sol 测试文件
  2. 当前 EVM 状态不是本地空链，而是从 TENDERLY_RPC_URL 对应的 fork 链读取
  3. 读到的合约代码、余额、存储都来自 Tenderly fork
  4. 测试中的状态修改只发生在这次 fork 执行上下文里

它和普通测试文件的区别

1. 普通 Foundry 单元测试

特点：

  • 完全本地执行
  • 通常依赖 anvil 或 Foundry 默认本地 EVM
  • 常用 vm.deal、vm.prank、mock 等 cheatcode 构造环境
  • 不一定依赖真实链状态

适合：

  • 测自己写的业务逻辑
  • 做边界条件、权限、重入、数学计算测试
  • 不关心主网真实状态时

2. Fork Test（包括 Tenderly fork RPC）

特点：

  • 代码本地跑，状态从真实链/远程 fork 来
  • 可以直接复用主网已有合约地址、池子状态、余额和存储
  • 非常适合漏洞复现、主网合约交互、DeFi 攻击路径分析

适合：

  • 复现历史攻击
  • 调试主网 DeFi 状态依赖
  • 验证某个 exploit 在特定区块高度是否成立

3. Tenderly 上广播脚本

特点：

  • 使用 forge script --broadcast
  • 会真的向 Tenderly Virtual TestNet 发送交易
  • Tenderly UI 里能看到真实交易、调用栈和 Debugger
  • 更接近“链上演示”而不是“本地测试”

例如：

source .env && forge script script/EulerTenderly.s.sol:EulerTenderlyScript --rpc-url $TENDERLY_RPC_URL --broadcast -vvvv

适合：

  • 想在 Tenderly Explorer / Debugger 里逐步看 exploit 调用栈
  • 想看到真实交易哈希、部署记录、内部调用
  • 想把本地 PoC 变成可视化演示交易

为什么不建议直接广播 .t.sol

.t.sol 是测试文件，不是脚本入口。

它的主要职责是：

  • 写测试逻辑
  • 写断言
  • 用 Test 框架驱动执行

而 --broadcast 需要的是：

  • 明确的脚本入口
  • 明确的部署步骤
  • 明确的链上交易顺序

所以通常分工是：

  • test/poc.t.sol：用于理解、验证、教学、断言漏洞链成立
  • script/*.s.sol：用于真正广播到 Tenderly Virtual TestNet，方便在 UI 中调试

一个实战判断方法

如果你的目标是：

只是验证漏洞能不能跑通

用：

forge test --rpc-url $TENDERLY_RPC_URL ...

想在 Tenderly UI 里点开交易、看 Debugger

用：

forge script --broadcast --rpc-url $TENDERLY_RPC_URL ...

Tenderly UI 里怎么查看 exploit 交易

当你已经用下面这种命令成功广播后：

source .env && forge script script/EulerTenderly.s.sol:EulerTenderlyScript --rpc-url $TENDERLY_RPC_URL --broadcast -vvvv

就可以去 Tenderly 页面里看这笔交易。

查看步骤

  1. 打开你对应的 Tenderly Virtual TestNet
  2. 进入 Transactions 或 Explorer
  3. 找到刚刚广播出来的交易
    • 一般会有一笔部署合约交易
    • 以及一笔真正执行 exploit 的交易
  4. 点开你要分析的那笔交易
  5. 进入 Debugger

在 Debugger 里重点看什么

做DeFi 漏洞复现时，建议按调用栈顺序看：

  1. flashLoan(...)
    • 确认借出的资产和数量是否正确
  2. executeOperation(...)
    • 这是 Aave V2 自动回调到攻击合约的入口
  3. leverUp / deposit / mint / repay
    • 看攻击者如何建立高杠杆仓位
  4. donateToReserves(...)
    • 看抵押物如何被主动转进储备池，但债务没有同步减少
  5. checkLiquidation(...)
    • 看 healthScore、repay、yield、discount
  6. liquidate(...)
    • 看清算时资产和债务如何从 violator 转到 liquidator
  7. withdraw(...)
    • 看 Euler 池里真实剩余的 DAI 如何被提走
  8. Aave 还款阶段
    • 看 amountOwing 是否被成功扣回

看状态时重点关注哪些变量

建议重点观察这些值在调用前后的变化：

  • eDAI balance / eDAI underlying
  • dDAI debt
  • healthScore
  • repay
  • yield
  • discount
  • 合约最终 DAI 余额

如何理解交易层级

Tenderly Debugger 通常会把一笔交易拆成多层调用：

  • 最外层：你广播出去的交易
  • 中间层：协议间的外部调用（Aave -> 攻击合约 -> Euler）
  • 内层：模块内部调用、token transfer、清算细节

你在分析 exploit 时，不要只看最外层成功失败，要一直往里点，直到看到：

  • 闪电贷回调入口
  • 核心漏洞函数
  • 清算函数
  • 资金回收函数

Tenderly 广播时为什么有时需要 --slow

在 forge script --broadcast 场景里，如果你的脚本会：

  1. 先部署一个新合约
  2. 再马上调用这个新合约上的函数

有时会遇到这样的问题：

  • 本地 simulation 显示一切成功
  • 但真正广播到链上后，只看到一笔部署交易
  • 第二笔调用交易 gas 很低（比如只有 21160）
  • 在 Tenderly 里看不到完整 exploit 调用栈

这通常说明：

第二笔调用发出时，第一笔部署交易还没被链确认，新合约代码还没真正落到链上。

结果就是：

  • 第二笔交易虽然 status = success
  • 但它其实只是对“当时还没有代码的地址”发了一次普通 call
  • 所以不会进入你期望的 attack()、flashLoan()、executeOperation() 等真实逻辑

典型现象

如果出现下面这种情况，就要怀疑广播顺序问题：

  • 一笔部署交易 gas 很高（几百万）
  • 第二笔调用交易 gas 只有 21160 左右
  • Tenderly 页面里只看到一层空调用
  • 没有内部调用栈

解决方法

给广播命令加 --slow：

source .env && forge script script/EulerTenderly.s.sol:EulerTenderlyScript --rpc-url $TENDERLY_RPC_URL --broadcast --slow -vvvv

--slow 的作用是：

  • 等前一笔交易先确认
  • 再发送下一笔交易

这样在“先部署、再调用”的脚本里会更稳。

Tenderly 到底怎么用（按按钮理解）

如果你第一次看 Tenderly，会很容易被一堆页面和调用树绕晕。 最实用的理解方式不是先背概念，而是：按页面按钮理解它在回答什么问题。

先记住：Tenderly 主要分成两类用途

1. 作为远程 fork 节点

你在本地运行：

forge test --rpc-url $TENDERLY_RPC_URL ...

这时：

  • 测试代码还是在本地 Foundry 执行
  • Tenderly 提供的是远程 fork 状态
  • 适合做 fork test、漏洞复现、真实链状态调试
2. 作为可视化调试器

你在本地运行：

forge script ... --broadcast --slow --rpc-url $TENDERLY_RPC_URL -vvvv

这时：

  • 会真的向 Tenderly Virtual TestNet 发交易
  • Tenderly UI 里能看到真实交易
  • 适合看完整调用栈、事件、资金流、状态变化

一、左边菜单分别是干什么的

1. Explorer

这是交易浏览器。

你要看：

  • 最近发了哪些交易
  • 哪笔是部署交易
  • 哪笔是 exploit 主交易
  • 交易哈希、gas、from、to、区块号

怎么判断：

  • To = contract creation 或带 Contract Address 的，通常是部署交易
  • 调用已部署合约地址、gas 很高的，通常是主交易

2. Contracts

这是合约中心。

你要看：

  • 某个合约地址有没有部署成功
  • 这个地址是不是你的 exploit 合约
  • 是否已经验证源码
  • 是否能直接看源码和方法

如果交易页右边出现：

  • No source for this contract

就说明：

  • Tenderly 还没拿到这个合约的源码
  • 你可以先去 Contracts 页面找该地址，再做验证

3. Wallets

这是钱包页。

你要看：

  • 当前 Virtual TestNet 上有哪些测试钱包
  • 哪个地址在发交易
  • 哪个地址没 ETH 付 gas

广播时报：

  • insufficient funds for gas * price + value

第一反应就去这里查广播地址有没有 ETH。

4. Faucet

这是给测试地址充值的地方。

主要给：

  • ETH（付 gas）
  • 有时也可配合 Virtual TestNet 资金工具给测试资产

5. JSON-RPC Calls / RPC Builder

这是调试 RPC 请求的地方。

适合：

  • 你想验证某个 eth_call、eth_getBalance、eth_getCode 是不是返回了预期结果
  • 你怀疑不是 Solidity 逻辑错，而是 RPC 或链状态错

6. Simulator

这是模拟器。

适合：

  • 不想立刻广播交易
  • 想先手工模拟一次调用看看会不会成功

二、点开某笔交易后，上方几个标签怎么看

你截图里最常用的是这几个：

  • Summary
  • Contracts
  • Events
  • State
  • Fund Flow
  • Gas Profiler

1. Summary

这是交易总览。

主要回答：

  • 这笔交易成功还是失败
  • from / to 是谁
  • gas 用了多少
  • 这是一笔部署，还是普通调用
  • 调用树大概长什么样

2. Contracts

这是这笔交易里涉及了哪些合约。

适合回答：

  • 这笔 exploit 到底调了哪些协议
  • Aave、Euler、DAI、helper 合约有没有都出现

3. Events

这是日志事件页。

最适合看：

  • Transfer
  • Approval
  • FlashLoan
  • Liquidation

如果你不想先读复杂调用树，可以先看 Events，快速确认：

  • 资产有没有转走
  • 闪电贷有没有发出
  • 清算有没有发生

4. State

这是状态变化页。

最适合看：

  • 某个 storage slot 有没有变化
  • 合约执行前后状态怎么变

在看漏洞时，它适合确认：

  • donate 后抵押有没有变
  • 清算后债务有没有转移

5. Fund Flow

这是资金流页面。

非常适合新手。

它主要回答：

  • 钱从哪来
  • 钱到哪去
  • 哪个地址收到了 DAI
  • 哪个协议被抽走了流动性

如果你对调用树看不懂，就先看 Fund Flow。

6. Gas Profiler

这是gas 分析页。

适合回答：

  • 这笔交易最耗 gas 的步骤在哪
  • 是 flashLoan、liquidate 还是 withdraw 最重

对理解漏洞本身不是第一优先，但对性能和交易复杂度很有用。

三、Debugger 页面到底怎么看

很多人第一次卡在这里。

你可以把 Debugger 分成 4 个区域理解：

1. 左侧调用树（Execution）

这是调用链目录。

它回答的是：

  • 谁调了谁
  • 调用顺序是什么
  • 当前你点开的代码片段属于哪一层调用

对 exploit 来说，应该顺着这条主线找：

  1. attack()
  2. flashLoan(...)
  3. executeOperation(...)
  4. leverUp / deposit / mint / repay
  5. donateToReserves(...)
  6. checkLiquidation(...)
  7. liquidate(...)
  8. withdraw(...)

如果左侧只有一层普通调用，而且 gas 特别低，通常说明：

  • 你点错交易了
  • 或者这笔调用没有真正进入 exploit 主逻辑

2. 中间源码区

这是当前调用对应的源码位置。

如果这里显示：

  • No source for this contract

说明 Tenderly 没拿到源码。

你仍然可以看 trace，但看不到高亮源码。

3. 底部输出区 / 变量区

这里适合看：

  • 当前函数输入参数
  • 当前日志事件数据
  • 当前步骤产出的返回值

如果你点到某条 FlashLoan 或 Transfer 事件，这里通常能直接看到金额和地址。

4. 左下角 Stack Trace

这是当前步骤在整个调用链里的位置。

当你看不懂左边大树时，可以看 Stack Trace 来判断：

  • 现在是在 Aave 内部
  • 还是已经回调到 exploit 合约
  • 还是已经进了 Euler 的模块

四、看一笔 exploit 交易时，建议固定顺序

不要乱点，按下面顺序最稳：

第一步：先看 Summary

确认：

  • 这是不是主交易
  • gas 是否足够高
  • from / to 是否正确

第二步：再看 Events 或 Fund Flow

确认：

  • 闪电贷有没有发出
  • DAI 有没有流入攻击合约
  • 清算和提现有没有发生

第三步：最后进 Debugger

按调用树从外到内看：

  • attack()
  • flashLoan()
  • executeOperation()
  • Euler 关键调用

五、一个最实用的判断口诀

看交易是不是主 exploit 交易

如果满足下面几点，通常才是：

  • 不是部署交易
  • gas 很高
  • 有很多内部调用
  • 能看到 flash loan / liquidation / withdraw 之类的路径

如果只有：

  • 单层调用
  • gas 只有 21160
  • 没有内部调用

通常不是主 exploit 交易。

看不懂调用树时怎么办

优先看：

  1. Summary
  2. Events
  3. Fund Flow
  4. 最后才是 Debugger

因为对新手来说：

  • Events 告诉你有没有发生关键动作
  • Fund Flow 告诉你钱到底去了哪
  • Debugger 才是最后深入的地方

六、把 Tenderly 当成什么工具最容易理解

可以这样记：

  • Explorer = 区块浏览器
  • Contracts = 合约通讯录
  • Debugger = 单步调试器
  • Fund Flow = 资金流向图
  • Events = 日志事件板
  • State = 存储变化板
  • Gas Profiler = 费用分析器

这样你就不会把所有页面都当成“都要先看”的东西。
