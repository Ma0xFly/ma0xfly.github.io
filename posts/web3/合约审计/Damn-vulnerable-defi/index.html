Unstoppable

有一个代币化金库，里面存有100万枚DVT代币。在宽限期结束前，它提供免费闪电贷服务。

为了在完全无需许可的情况下发现任何漏洞，开发人员决定在测试网上进行实时测试。他们还设置了监控合约来检查闪电贷功能的运行状况。

player账户余额为 10 个 DVT 代币，证明可以关闭金库。金库必须停止提供闪电贷服务

你起始有 10 DVT，目标是让 Vault 停止提供闪电贷服务（flashLoan 调用失败，Monitor 触发暂停）。

漏洞点

UnstoppableVault

UnstoppableVault 是一个 ERC4626 vault + ERC3156 闪电贷出借方 + Pausable

关键点：

  • totalAssets() 返回 asset.balanceOf(address(this))（直接读 Token 余额），如果你不通过 deposit，而是直接通过 transfer 把代币转给合约（“强制喂食”），totalAssets() 会增加，但 totalSupply() 不会变。
  • ERC4626 的 share 与 asset 之间的转换由 convertToShares/convertToAssets 负责
  • flashLoan 的关键检查：

function flashLoan(IERC3156FlashBorrower receiver, address _token, uint256 amount, bytes calldata data)
external
returns (bool)
{
  if (amount == 0) revert InvalidAmount(0); // 早期失败：借款量不能为 0
  if (address(asset) != _token) revert UnsupportedCurrency(); // 强制 ERC3156 规范：资产必须匹配
  uint256 balanceBefore = totalAssets();
  // 校验：将 totalSupply 按照 ERC4626 转换为 shares 后，应等于实际资产余额
  // 这是一个关键的安全检查，若不匹配说明内部会计错乱
  if (convertToShares(totalSupply) != balanceBefore) revert InvalidBalance(); // enforce ERC4626 requirement

  // 将代币转给 borrower，并执行回调
  ERC20(_token).safeTransfer(address(receiver), amount);

  // 计算费用并执行回调
  uint256 fee = flashFee(_token, amount);
  if (
    receiver.onFlashLoan(msg.sender, address(asset), amount, fee, data)
    != keccak256("IERC3156FlashBorrower.onFlashLoan")
  ) {
    // 回调未返回正确的魔数，视为失败
    revert CallbackFailed();
  }

  // 从 borrower 拉回本金 + 费用（需要 borrower 允许金库转移）
  ERC20(_token).safeTransferFrom(address(receiver), address(this), amount + fee);
  // 将费用支付给 feeRecipient
  ERC20(_token).safeTransfer(feeRecipient, fee);

  return true;
}

注意这句：

if (convertToShares(totalSupply) != balanceBefore)
revert InvalidBalance();

含义是：

Vault 认为：用当前总 share 数量 totalSupply 换算出的 asset 数量，应该等于 Vault 实际持有的 asset 余额 totalAssets()。

这在“所有资产都通过 deposit/mint 流入 Vault”时是成立的。

UnstoppableMonitor（监控 + 暂停）

UnstoppableMonitor 实现了 IERC3156FlashBorrower，用来检测 Vault 是否还能正常闪电贷

伪代码逻辑：

function checkFlashLoan(uint256 amount) external {
  try vault.flashLoan(
    IERC3156FlashBorrower(address(this)),
    address(vault.asset()),
    amount,
    ""
  ) {
    emit FlashLoanStatus(true);
  } catch {
    // 闪电贷失败 -> 直接暂停 Vault，并把所有权转回 deployer
    vault.setPause(true);
    vault.transferOwnership(deployer);
    emit FlashLoanStatus(false);
  }
}

测试里：

  • setUp() 里先由 deployer 调用一次 checkFlashLoan(100e18)，确保一开始是成功的（FlashLoanStatus(true)）
  • _isSolved() 再由 deployer 调用一次 checkFlashLoan(100e18)，此时必须失败才算过关

因此，本关的本质目标是：找到一种方式，让后续任何 flashLoan 都因为 revert 而失败。

Naive Receiver (Context Confusion 上下文混淆)

There’s a pool with 1000 WETH in balance offering flash loans. It has a fixed fee of 1 WETH. The pool supports meta-transactions by integrating with a permissionless forwarder contract.

A user deployed a sample contract with 10 WETH in balance. Looks like it can execute flash loans of WETH.

All funds are at risk! Rescue all WETH from the user and the pool, and deposit it into the designated recovery account. 关卡情报

  • 场景： 有一个闪电贷资金池 (NaiveReceiverLenderPool)，它提供 ETH 闪电贷，每次贷款收取固定的 1 ETH 手续费。
  • 受害者： 有一个不知情的合约 (FlashLoanReceiver)，它目前持有 10 ETH。这个合约实现了闪电贷接收接口，意味着它可以接收闪电贷。
  • 目标： 榨干受害者合约里的所有 10 ETH，直到它余额归零。最后提款到指定账户
  • 你的资源： 你几乎身无分文（或者说不需要花自己的钱）。

审计漏洞

第一步：鸟瞰全貌 (Reconnaissance)

目的：不看具体逻辑，先弄清楚“这合约是干嘛的？”、“它依赖谁？”。

  1. 看名字：NaiveReceiverPool -> 一个池子，可能和“天真（Naive）”的接收者有关。

  2. 看继承 (Inheritance)：

    contract NaiveReceiverPool is Multicall, IERC3156FlashLender

    • IERC3156FlashLender：这是一个闪电贷借出方。我知道它一定有 flashLoan 函数。

    • Multicall：这是一个由 OpenZeppelin 或 Solady 提供的功能，允许在一个交易里批量调用函数。（警觉：这通常是组合攻击的入口）。

  3. 看 Import：

    • WETH：它只处理 Wrapped ETH，不是原生 ETH。

    • FlashLoanReceiver：它会和这个接收者合约交互。

第一步结论：这是一个只借出 WETH 的闪电贷池子，支持批量调用。

第二步：找“钱”在哪 (Follow the Money)

目的：合约最危险的地方就是资金进出的地方。找到存储资金的变量。

  1. 状态变量 (State Variables)：

    uint256 private constant FIXED_FEE = 1e18; // 1 ETH
WETH public immutable weth;
mapping(address => uint256) public deposits;
uint256 public totalDeposits;

    • 审计嗅觉：FIXED_FEE = 1 ETH。太贵了！ 借 1 块钱也要付 1 ETH 手续费？这绝对是经济模型设计上的隐患。

    • 资金流向：资金是 WETH，内部用 deposits 记账。

第三步：核心逻辑审计 (Core Business Logic)

目的：盯着最复杂的那个函数看。在闪电贷合约里，就是 flashLoan。

我们要逐行“翻译”它的行为，并扮演恶意用户的角色。

function flashLoan(IERC3156FlashBorrower receiver, address token, uint256 amount, bytes calldata data)
    external
    returns (bool)
{
    // ... 省略代币检查 ...

    // 动作 1: 给钱
    weth.transfer(address(receiver), amount);

    // 动作 2: 外部调用 (External Call) -> ⚠️ 高危区域
    // 这里的 msg.sender 是我（攻击者），但 receiver 是受害者地址
    // 关键点：合约没有检查 receiver 是否等于 msg.sender！
    if (receiver.onFlashLoan(msg.sender, address(weth), amount, FIXED_FEE, data) != CALLBACK_SUCCESS) {
        revert CallbackFailed();
    }

    // 动作 3: 收钱 + 强制收税
    uint256 amountWithFee = amount + FIXED_FEE;
    weth.transferFrom(address(receiver), address(this), amountWithFee);

    // ... 省略记账 ...
    return true;
}

审计自问自答（关键步骤）：

  • Q1: 谁能调用 flashLoan?

    • A: external 修饰，任何人都能调用。我是黑客，我能调用。

  • Q2: 我调用的时候，receiver 参数能填别人吗？

    • A: 代码里没有 require(receiver == msg.sender)。所以我可以填任何实现了 onFlashLoan 接口的合约地址（比如那个“天真的接收者”）。

  • Q3: 谁付钱？

    • A: weth.transferFrom(address(receiver), ...)。代码强制从 receiver 也就是受害者那里扣钱。

  • Q4: 我如果借款金额 amount 填 0 会怎样？

    • A: 手续费是 FIXED_FEE (1 ETH)。即使借 0，受害者也要付 1 ETH。

第三步结论：发现逻辑漏洞！我可以发起交易，指定别人当冤大头付手续费。

第四步：组合攻击 (Check for Combinations)

目的：回到第一步看到的 Multicall。

  • 单次攻击：我调用一次 flashLoan，受害者亏 1 ETH。

  • 既然继承了 Multicall，我能不能在一个交易里调用 10 次 flashLoan？

    • 完全可以。这样一笔交易的 Gas 费就能耗尽受害者 10 ETH。效率极高。

第五步：利用上下文混淆withdraw提取所有余额

wp中当执行 calls[10] = abi.encodePacked(withdrawOp, deployer); 时，构造的数据流（Payload）在内存中长这样：

|------------ 标准 withdraw 调用数据 (withdrawOp) --------|--- 附加数据 ---|
|  Selector|    Amount (32字节)      |   Receiver (32字节)  |   Deployer   |
|  4字节   | 000...0003f2 (1010 ETH) | 000...recovery_addr |  20字节地址   |

当这段数据被传入 pool.withdraw 执行时，发生了以下两件事：

withdraw 函数的视角 (正常的参数解析)

Solidity 的 ABI 解码规则有一个特性：忽略尾部多余数据 (Ignore Trailing Garbage)

withdraw(uint256 amount, address receiver) 函数只需要读取前 4+32+32 个字节。它读完 receiver 后，认为任务完成了，根本不关心后面是不是还粘着一个 deployer 地址。

  • amount: 解析成功

  • receiver: 解析成功

  • 后面多余的 20 字节：直接无视

_msgSender() 函数的视角 (特殊的身份解析)

在 withdraw 内部，执行到了 deposits[_msgSender()] -= amount;。 这时，_msgSender() 被调用了。

function _msgSender() internal view override returns (address) {
    // 条件1: 调用者是 trustedForwarder (我们通过 Forwarder 发起的，满足)
    if (msg.sender == trustedForwarder && msg.data.length >= 20) {
        // 条件2: 截取 msg.data 的最后 20 个字节作为“真实发送者”
        return address(bytes20(msg.data[msg.data.length - 20:])); 
    }
    // ...
}

  • msg.data 是什么？ 由于 Multicall 使用了 delegatecall，当执行到第 11 个操作时，msg.data 正是我们构造的那串 “withdrawOp + deployer”。

  • 切片操作： 它直接跳到数据流的最末尾，切下了那 20 个字节。

  • 结果： 它读到了我们粘上去的 deployer 地址！ setUp 函数揭示deployer 是金主：Pool 里的初始 1000 WETH 和赚到的 10 WETH 手续费，在账本上都归 deployer 所有。

Truster

越来越多的贷款池开始提供闪贷。在这种情况下，一个新的池已经启动，免费提供DVT代币闪贷。

该池持有100万个DVT代币。你什么都没有。

为了通过这个挑战，拯救池中的所有资金，执行单个交易。将资金存入指定的回收账户。

收到，让我们按照**“五步法”**审计标准，立即对 TrusterLenderPool 进行安全分析。

第一步：鸟瞰全貌 (Reconnaissance)

  • 合约名: TrusterLenderPool 1。暗示了某种“信任”关系，通常在安全领域，“Trust”往往意味着“过度信任”或“权限过大”。

  • 继承: ReentrancyGuard 2。这说明开发者意识到了重入攻击的风险，并使用了 OpenZeppelin 的标准库来防御。

  • 库的使用: using Address for address; 3。引入了 OpenZeppelin 的 Address 库，这通常意味着会用到 .functionCall 这种低级调用封装。

  • 核心功能: 提供 Flash Loan（闪电贷）服务。

初步结论：这是一个标准的闪电贷池，防了重入，但名字里的 “Truster” 让人不安。

第二步：资金锚定 (Follow the Money)

  • 核心资产: DamnValuableToken public immutable token; 4。

  • 余额检查: 合约没有使用 mapping 记账，而是直接在 flashLoan 中通过 token.balanceOf(address(this)) 检查余额变化 5。

  • 目标: 我们的目标是把池子里的 token 搬空。

第三步：核心逻辑精读 (Core Business Logic)

我们聚焦于唯一的业务函数 flashLoan。让我们扮演攻击者进行拷问：

function flashLoan(uint256 amount, address borrower, address target, bytes calldata data)
    external
    nonReentrant
    returns (bool)
{
    // 记录闪电贷之前的余额
    uint256 balanceBefore = token.balanceOf(address(this));

    // 1. 给钱：把 token 转给借款人
    token.transfer(borrower, amount);

    // 2. 外部调用：执行 target 上的任意代码
    target.functionCall(data); // <--- !!! 极其可疑

    // 3. 检查还款：只要余额不减少就行
    if (token.balanceOf(address(this)) < balanceBefore) {
        revert RepayFailed();
    }

    return true;
}

Attacker Mindset 自问自答：

  • Q1: 谁能调？ external 修饰，任何人都能调 6。

  • Q2: 参数受控吗？ amount, borrower, target, data 全部由我（攻击者）传入，完全受控 7。

  • Q3: 谁付钱？ 只要最后余额没少，不需要额外付手续费。

  • Q4: 外部调用在哪？ 就在 target.functionCall(data) 8。

第四步：威胁建模 (Threat Modeling)

我们发现了绝对的高危操作：任意外部调用 (Arbitrary External Call)。

漏洞点分析：

代码行：target.functionCall(data);

  1. 身份 (Context)：

    当这一行代码执行时，是谁在发起调用？

    是 TrusterLenderPool 合约自己。也就是说，msg.sender 是池子的地址。

  2. 能力 (Capability)：

    由于 target 和 data 是我们可以控制的，我们可以让池子以它自己的身份，去调用任何合约的任何函数。

  3. 攻击向量构思：

    • 如果我们借款 amount = 0，那么 balanceBefore 等于当前余额。

    • 只要我们执行的操作不减少池子的余额，第3步的检查就能通过。

    • 关键问题：有什么操作是不移动资金，但能让我们获利的？

    • 答案：ERC-20 的 approve (授权)。

攻击剧本：

  1. 调用 flashLoan。

    • amount: 0

    • target: DVT Token 合约地址

    • data: approve(attacker, 1000000 ether) 的编码数据

  2. 执行过程：

    • 池子调用 token.approve(attacker, 100W)。

    • 此时，Token 合约里的 allowance[pool][attacker] 变成了 100W。

    • 池子余额未变，Flash Loan 检查通过。

  3. 收割：

    • 交易结束后，攻击者调用 token.transferFrom(pool, attacker, 1000000 ether)。

    • 因为已经获得了授权，资金被瞬间转走。

side-entrance

一个令人惊讶的简单池允许任何人存入ETH，并在任何时间点提取它。

它已经有1000个ETH余额，并且正在使用存入的ETH提供免费闪贷来推广他们的系统。

一开始你有1个ETH的余额。通过从池中拯救所有ETH并将其存入指定的回收账户来通过挑战。

第一步：鸟瞰全貌 (Bird’s Eye View)

目的：快速定位合约角色与工具。

  • 看名字：SideEntranceLenderPool（侧门借贷池）。

    • 暗示：这像是一个隐喻。正门可能有守卫（检查），但也许我们可以通过“侧门”把钱运出去。

  • 看引入：

    • import {SafeTransferLib} from "solady/utils/SafeTransferLib.sol";

    • 使用了我们刚才讨论过的 Solady 库。

    • 注意：withdraw 函数使用了 SafeTransferLib.safeTransferETH，这意味着转账是安全的，且处理了返回值，但它不管业务逻辑对不对。

  • 看接口：

    • IFlashLoanEtherReceiver：定义了 execute 函数。这是闪电贷回调的标准模式。

第二步：资金锚定 (Follow the Money)

目的：锁定金库位置，区分“物理资金”和“账本资金”。

这是这一关最核心的地方，请务必仔细区分：

  1. 物理资金 (Reality)：

    • address(this).balance：合约实际上持有的 ETH 总量。

  2. 账本资金 (Accounting)：

    • mapping(address => uint256) public balances：合约在小本本上记下来的“谁拥有多少钱”。

关键问题：这两个系统是强绑定的吗？

  • 在 deposit() 中：你给物理资金（msg.value），账本给你记账。✅ 绑定。

  • 在 withdraw() 中：账本销账（delete），物理资金给你。✅ 绑定。

  • 漏洞嗅觉：如果这两个系统在某一刻解绑了，或者我们可以利用其中一个系统去欺骗另一个系统，漏洞就出现了。

第一步：鸟瞰全貌 (Bird’s Eye View)

第二步：资金锚定 (Asset Anchoring)

1. 资金流入 (Inflow)

  • createDistribution: 调用 SafeTransferLib.safeTransferFrom 将代币从 msg.sender 拉入合约。此时更新 remaining 余额。这是资金的信任源头。

第三步：核心逻辑精读 (Core Logic Deep Dive)

我们将显微镜对准 claimRewards 函数。开发者为了省 Gas，采用了一种 “延迟写入 + 累加器” 的模式。

代码逻辑拆解：

  1. 循环遍历：合约遍历用户提供的 inputClaims 数组。

  2. 分组处理：它试图把针对同一个 Token 的多次领取合并成一次状态更新（SSTORE）和一次转账（CALL），以此节省 Gas。

  3. 两个累加器：

    • amount (uint256): 算术累加。 $Total = Claim_1 + Claim_2 + …$

    • bitsSet (uint256): 位操作累加。 $Mask = Bit_1 | Bit_2 | …$

致命的逻辑不对称性：

在循环内部处理同一个 Token 的连续 Claim 时：

  • 金额计算：amount += inputClaim.amount;

    • 如果你传入两次相同的 Claim（金额 100），结果是 $100 + 100 = 200$。

  • 状态记录：bitsSet = bitsSet | 1 << bitPosition;

    • 如果你传入两次相同的 Claim（Batch Index 0），结果是 $(1 \ll 0) | (1 \ll 0) = 1$。

    • 注意：这里没有检查 bitsSet 中是否已经包含了该位！

检查点滞后：

_setClaimed 函数是唯一检查“是否已领取”并“写入状态”的地方。但它是在一组循环结束后才调用的，而且传入的是最终计算出的 bitsSet（值为 1）和 amount（值为 200）。

合约会去存储槽（Storage）里看：位置 0 的位是 1 吗？

  • 如果是第一次领，答案是 否。

  • 检查通过。

  • 合约写入位置 0 为 1。

  • 合约转账 200 代币。

  • 结果：用户用一份凭证，领了两份钱。

第四步：威胁建模 (Threat Modeling)

1. 身份分离检查 (Auth Check)

  • msg.sender 是领取者。

  • MerkleProof.verify 验证了 msg.sender 有资格领取某笔钱。

  • 漏洞点：验证逻辑没有阻止 msg.sender 在同一个交易中多次使用同一个资格。

2. 攻击向量构建 (Attack Vector)

场景：重复索赔攻击 (The “Double Dip” Attack)

  • 前提：攻击者在 Batch 0 有 100 USDT 的奖励资格，且尚未领取。

  • 攻击载荷 (Payload)：

    • 构造一个 Claim 数组，包含 N 个完全相同的元素：

      • batchNumber: 0

      • amount: 100

      • tokenIndex: 0

      • proof: [有效的 Merkle Proof]

  • 执行流程：

    1. 循环第 1 次：bitsSet 变为 ...001，amount 变为 100。

    2. 循环第 2 次：bitsSet 仍为 ...001 (OR 操作幂等)，amount 变为 200。

    3. …

    4. 循环第 N 次：bitsSet 仍为 ...001，amount 变为 $N \times 100$。

    5. 调用 _setClaimed：检查链上状态，Batch 0 未被标记。通过。

    6. 转账：发送 $N \times 100$ USDT。

  • 后果：直接掏空该 Token 在合约内的所有余额（只要 $N \times 100 \le$ 合约余额）。

3. 另一个攻击向量：Word 覆盖攻击 (Word Overwrite)

  • 原理：wordPosition 变量在循环中被更新，但 _setClaimed 只使用循环结束时的那个 wordPosition。

  • 载荷：[Claim(Batch 0), Claim(Batch 256)]。

    • Batch 0 对应 Word 0。

    • Batch 256 对应 Word 1。

  • 结果：循环结束时 wordPosition 是 1。bitsSet 包含了 Batch 0 和 Batch 256 的位。合约会将这些位都写到 Word 1 中。

    • 后果：Word 0（Batch 0 的真实位置）没有被写入。攻击者可以再次合法领取 Batch 0。
