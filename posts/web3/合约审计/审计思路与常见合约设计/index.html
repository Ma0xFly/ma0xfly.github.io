审计思路

这是每一位 Web3 安全工程师新手的必经之路。面对几百行代码感到不知所措是非常正常的。

别担心，专业的审计并不是从第一行读到最后一行。我们有一套标准化的审计“战术流程”。

第一步：鸟瞰全貌

目的：搞清楚“这合约是干嘛的”、“它使用了什么工具”。

  • 看名字：通常暗示了业务逻辑（如 Pool, Vault, Lender）。

  • 看继承 (Inheritance)：

    • is Ownable $\rightarrow$ 有权限控制。

    • is ERC20 $\rightarrow$ 它是一个代币。

    • is Multicall $\rightarrow$ 警觉！ 支持批量调用（组合拳入口）。

    • is ReentrancyGuard $\rightarrow$ 开发者考虑了防重入（但也可能用错）。

  • 看引入 (Imports)：使用了 OpenZeppelin 还是 Solmate？（Solmate 通常更激进，少检查）。

第二步：资金锚定

目的：找到“金库”在哪里。合约最危险的地方就是存钱的地方。

  • 找状态变量：

    • mapping(address => uint256) balances：内部记账。

    • IERC20 public token：它操作的是哪种资产？

  • 找常量/不可变量：

    • 是否有硬编码的费率？（如 FIXED_FEE = 1 ether）。

    • 是否有硬编码的地址？

    • 是否有特殊的时间锁？

第三步：核心逻辑精读

目的：逐行翻译最复杂的那个函数（通常是 flashLoan, swap, deposit）。

  • 扮演攻击者 (Attacker Mindset)：在读每一行时，自问自答：

    • Q1: 谁能调？ (external/public 且无 onlyOwner？那我能调)。

    • Q2: 参数受控吗？ (target, data, amount 是我填的吗？)。

    • Q3: 谁付钱？ (是用 msg.sender 的钱，还是 address(this) 的钱，还是 target 的钱？)。

    • Q4: 外部调用在哪？ (见第四步)。

第四步：威胁建模

目的：扫描高危操作和组合漏洞。

  • 外部调用 ：

    • 搜索关键字：call, delegatecall, staticcall, transfer, transferFrom。

    • 黄金法则：这是控制权移交的地方。如果目标地址 (target) 和数据 (data) 由用户控制，这里就是 任意外部调用漏洞 (Arbitrary External Call)。

  • 鉴权缺失 ：

    • 检查 msg.sender 是否与受影响的账户一致。

    • 是否存在 tx.origin 这种错误的鉴权。

  • 上下文混淆 ：

    • 如果涉及 delegatecall 或 Multicall，检查 msg.sender 是否被保留或混淆（如第二关）。

总结：新手审计的“四板斧”

下次看到大段代码，按这个顺序切入：

  1. 看门面 (Interface)：看继承了什么，引入了什么库。（确定是 DeFi、NFT 还是 DAO？）

  2. 找常量 (Constants)：找有没有离谱的费率、时间锁、或者硬编码地址。（比如这里的 1 ETH 费率）。

  3. 盯着外部调用 (External Calls)：搜索 call, delegatecall, transfer, onFlashLoan 等关键字。

    • 这是控制权移交的地方，也是最容易出事的地方。

  4. 身份分离检查 (Auth Check)：

    • 调用者 (msg.sender) 和 受影响者 (account / receiver) 是同一个人吗？

    • 如果不是，msg.sender 有权利动 receiver 的钱吗？

现在，你再回头看那段代码，是不是感觉清晰多了？ 不要在意 _deposit 那种辅助函数，盯着 flashLoan 这种核心业务逻辑猛攻。

智能合约安全审计 SOP (v1.0)

基于你提供的 “合约分析四步法”，我已将其标准化为一份 Web3 安全审计 SOP (Standard Operating Procedure)。

这份文档将你的个人经验（鸟瞰、资金、逻辑、威胁）转化为可执行、可复查的工程标准，适用于审计报告生成或漏洞挖掘实战。

目标 (Objective): 系统化拆解智能合约逻辑，确保从宏观架构到微观代码行的全覆盖，最小化安全死角。

适用范围 (Scope): Solidity 智能合约（DeFi, NFT, DAO 等）。

工具链 (Tools): Foundry (Forge), VS Code (Solidity Metrics), Slither.

第一阶段：全貌扫描 (Bird's Eye View)

核心目标： 确定“这是什么”，识别系统边界与静态风险。

1.1 协议指纹识别 (Protocol Fingerprinting)

  • 动作： 检查 interface 和继承树（Inheritance Tree）。

  • 检查项：

    • 是否继承了标准库（OpenZeppelin, Solmate）？版本是否兼容？

    • 分类判定：

      • DeFi (AMMs, Lending, Staking) -> 重点关注数学精度与流动性。

      • NFT (ERC721/1155) -> 重点关注 Mint 逻辑与元数据权限。

      • DAO (Governor) -> 重点关注投票权重与提案执行阈值。

1.2 静态配置审查 (Constants & Config Scan)

  • 动作： 全局搜索 constant, immutable 关键字及硬编码数值。

  • 风险检查点：

    • 费率 (Fees)： 是否存在超过 10% 的异常费率？费率分母（Basis Points）是否正确？

    • 时间锁 (Timelock)： 关键操作是否有冷却期（Cooldown）？

    • 特权地址： 是否存在硬编码的 EOA 地址（Admin/Owner）？

SOP 输出物： 系统架构图，包含合约继承关系及外部依赖库列表。

第二阶段：资金锚定 (Funds Anchoring)

核心目标： 追踪“钱去哪了”，锁定控制权移交点（The Sink）。

2.1 外部调用审计 (External Calls Analysis)

  • 动作： 正则搜索高危关键字：call, delegatecall, transfer, transferFrom, safeTransfer, onFlashLoan。

  • 关键检查：

    • 控制权移交： 所有的 external call 前后是否遵循“检查-生效-交互”（CEI）模式？

    • 重入风险： 调用外部合约后，是否更新了状态变量？是否加了 nonReentrant 修饰符？

    • 返回值检查： 低级调用 call 的布尔返回值是否被处理？

2.2 资金流向图谱 (Asset Flow Mapping)

  • 动作： 绘制资金流入（Deposit）与流出（Withdraw）路径。

  • 关键检查：

    • 谁能触发转账？

    • 合约内资金是否可能被“意外”锁死（没有 withdraw 函数）？

第三阶段：核心逻辑精读 (Core Logic Deep Dive)

核心目标： 理解“怎么运行”，验证状态机转换的正确性。

(注：此阶段连接资金流与威胁模型)

3.1 状态变化审计 (State Transition Check)

  • 动作： 逐行分析修改 storage 变量的函数。

  • 检查项：

    • 记账逻辑： balances[user] 的增减是否原子化？是否存在“凭空增发”或“双重扣除”？

    • 数学运算： 是否使用 unchecked？是否存在溢出/下溢风险（尤其是 Solidity < 0.8.0 或涉及位运算时）？

3.2 业务逻辑一致性 (Business Logic Consistency)

  • 动作： 对照白皮书或注释，验证代码意图。

  • 反直觉检查：

    • 代码逻辑是否与函数名（如 emergencyWithdraw）描述一致？

    • 极端情况（如 TotalSupply = 0 或 极大值）下逻辑是否崩坏？

第四阶段：威胁建模 (Threat Modeling)

核心目标： 尝试“坏人视角”，通过身份伪造与权限绕过进行攻击模拟。

4.1 身份分离检查 (Auth & Spoofing Check)

  • 核心原则： 永远不要信任 msg.sender。

  • 检查清单：

    • 调用者 vs 受益人：

      • 代码中 msg.sender（操作者）和 receiver（接收资金者）是同一个参数吗？

      • 如果是分离的（如 transferFrom(from, to, amount)），检查 msg.sender 是否有权操作 from 的资金（即 allowance 检查）。

    • Tx.origin 陷阱： 确保未混淆 msg.sender 与 tx.origin（防止钓鱼攻击）。

4.2 权限越权测试 (Privilege Escalation)

  • 动作： 梳理所有 onlyOwner 或 onlyRole 修饰的函数。

  • 攻击面：

    • 初始设置函数（initialize）是否被保护？能否被 Front-run 抢跑？

    • 权限转移逻辑是否可能导致权限丢失（如转给零地址）？

总结：SOP 执行速查表 (Checklist)

阶段 核心动作 关键指令/搜索词 必须确认 (Definition of Done)
1. 鸟瞰 继承与常量检查 interface, constant, immutable 确认是不是标准 Fork 项目？
2. 资金 外部调用追踪 call, delegatecall, transfer 所有的“出钱”出口都已标记？
3. 逻辑 状态与数学 unchecked, storage 变更 记账逻辑是否闭环？
4. 威胁 身份与权限 msg.sender vs account 确认我不能动别人的钱？

常见合约设计

Multicall

Multicall (多重调用) 是 Web3 开发和审计中非常核心的一个设计模式。

简单来说，它的作用是：把多个独立的函数调用，打包进唯一的一笔交易 (Transaction) 中执行。

作为安全工程师，你需要从 “业务价值” 和 “底层原理” 两个维度来理解它，因为这通常是产生高阶漏洞（如Context Confusion）的根源。

1. 为什么我们需要 Multicall？(业务价值)

如果没有 Multicall，你要执行 10 次操作（比如给 10 个人发工资，或者像第DVD第二关那样借 10 次钱），你需要：

  1. 发起 10 笔交易。

  2. 用户点 10 次钱包确认。

  3. 支付 10 次基础 Gas 费（每笔交易自带 21,000 gas base fee）。

  4. 原子性缺失：可能第 3 笔成功了，第 4 笔失败了，导致状态不一致。

有了 Multicall：

  1. 省钱 (Gas Efficiency)：只付一次 21,000 的基础费用。

  2. 原子性 (Atomicity)：所有操作要么全部成功，要么（通常）全部失败回滚。这在 DeFi 组合操作中至关重要（例如：Swap -> Add Liquidity -> Stake，中间任何一步断了都不行）。

  3. 用户体验 (UX)：用户只需签一次名。

2. 底层原理 (代码审计视角)

在 Solidity 中，Multicall 通常是一个抽象合约（Abstract Contract），你的 DVDNaiveReceiverPool 继承了它。

核心代码逻辑通常长这样（OpenZeppelin 实现版）：

abstract contract Multicall {
    function multicall(bytes[] calldata data) external virtual returns (bytes[] memory results) {
        results = new bytes[](data.length);
        
        for (uint256 i = 0; i < data.length; i++) {
            // ⚠️ 关键点：delegatecall
            (bool success, bytes memory ret) = address(this).delegatecall(data[i]);
            require(success, "Multicall: internal error");
            results[i] = ret;
        }
        return results;
    }
}

审计重点：delegatecall 与上下文 (Context)

这是你刚刚通关第二关的核心知识点。

  1. 自调用 (Self-Interaction)： 当通过 pool.multicall(...) 调用时，合约是在自己调用自己。

  2. delegatecall 的特性： 它会保留原始的 msg.sender 和 msg.value。

    • 正常情况：

      • 你 (Player) -> 调用 -> Pool.multicall

      • Multicall 内部 -> delegatecall -> Pool.flashLoan

      • 在 flashLoan 里，msg.sender 依然是 Player。

    • 受信任转发器 (Trusted Forwarder) 情况 (本关漏洞)：

      • Forwarder -> 调用 -> Pool.multicall

      • 此时 msg.sender 是 Forwarder。

      • Multicall 内部 -> delegatecall -> Pool.withdraw

      • 在 withdraw 里，msg.sender 依然是 Forwarder！

      • 这就是为什么 _msgSender() 检查通过了，因为它看到了 Forwarder，并进而去解析 calldata 末尾的“伪造地址”。

3. 常见的 Multicall 类型

  1. Read-Only Multicall (链下查询用)：

    • 例如 MakerDAO Multicall 合约。

    • 用于前端一次性查询 100 个代币的余额。

    • 通常使用 staticcall。

    • 风险低。

  2. State-Changing Multicall (交易执行用)：

    • 例如 Uniswap V3 Router, NaiveReceiverPool。

    • 用于打包复杂的交易逻辑。

    • 使用 delegatecall。

    • 风险高：必须检查 payable 问题（是否多次计算了同一个 msg.value？）和 上下文混淆问题（Context Confusion）。

总结： Multicall 就是一个“批处理循环”。在审计时，只要看到 Multicall，立刻警觉：“如果我在这个批处理中混合了不同权限的函数，或者通过代理合约调用它，会不会导致权限校验被绕过？”
