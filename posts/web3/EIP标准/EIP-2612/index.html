EIP-2612: ERC-20 Permit — 基于签名的授权扩展 (Permit Extension for EIP-20 Signed Approvals)

一句话总结：EIP-2612 为标准 ERC-20 代币引入了一个 permit() 函数，允许用户通过 链下 EIP-712 签名 来完成代币授权（approve），彻底消除了"用户必须先发一笔 approve 交易再发一笔操作交易"的两步操作痛点，实现了 Gasless Approval（无 Gas 授权）。

📜 核心元数据 (Metadata)

信息源原则：底层网络协议必须以 Ethereum EIP 官方文档 (eips.ethereum.org) 和以太坊黄皮书 (Yellow Paper) 为准。

  • 官方 EIP 提案编号与链接：ERC-2612: Permit Extension for EIP-20 Signed Approvals
  • 协议类型 (Type)：Standards Track / ERC（应用层标准，不修改 EVM 底层）
  • 所属分叉/升级名称 (Hard Fork)：无。这是一个纯合约层面的标准，不依赖任何硬分叉，任何 ERC-20 代币都可以自行选择实现。
  • 当前标准状态：Final
  • 前置依赖 EIP：
    • EIP-20：标准 ERC-20 代币接口（本标准是其扩展）。
    • EIP-712：类型化结构数据签名标准（permit 的签名格式严格遵循 EIP-712）。

1. 核心理念与全网级痛点 (Core Concept & Motivation)

  • 传统 ERC-20 的交互痛点： 在标准的 ERC-20 中，如果用户想让一个智能合约（如 Uniswap Router）代自己花费代币，必须完成两笔链上交易：

    1. 第一笔：approve()：用户先发一笔交易，授权合约可以使用多少代币。
    2. 第二笔：实际操作：用户再发一笔交易调用目标合约（合约内部调用 transferFrom() 从用户账户转走代币）。

    这意味着：

    • 用户需要支付两次 Gas 费（审批一次 + 操作一次）。
    • 对于没有 ETH 的用户（比如你只有 USDC 却没有 ETH），你连第一步 approve 的 Gas 都付不起，根本无法与 DeFi 交互。
    • 用户体验极差：每次操作都需要确认两次钱包弹窗。

  • 破局思路：把 approve 搬到链下去做。用户不需要发送一笔链上交易来授权，而是在钱包里用私钥对一条结构化消息签名（就像"签一张支票"），然后把这个签名交给任何人。任何人拿着这个签名，都可以调用合约的 permit() 函数来替用户完成链上授权。

  • 专属核心概念：

    • permit()：EIP-2612 引入的核心函数。任何人调用它并提交一个合法的 EIP-712 签名，就能代替代币持有者完成 approve() 操作。
    • Gasless Approval（无 Gas 授权）：用户自己不需要花 Gas 来 approve，Gas 由调用 permit() 的第三方（如 Relayer、DApp 后端）支付。
    • EIP-712 签名：一种结构化的链下签名标准，钱包（如 MetaMask）弹出的签名窗口会以人类可读的格式展示签名内容（如"你正在授权 XX 合约花费 100 USDC"），远比签名一堆十六进制哈希安全得多。

2. 核心架构与底层修改细节 (Core Specification)

EIP-2612 是纯合约层标准，不修改 EVM 指令集、不改变区块结构、不引入新交易类型。它只要求实现 EIP-2612 的 ERC-20 合约额外暴露三个函数。

A. 新增接口（3 个函数）

// 核心函数：任何人都可以调用，用链下签名来完成 approve 操作
function permit(
    address owner,     // 代币持有者（签名者）
    address spender,   // 被授权人（谁可以花钱）
    uint256 value,     // 授权额度
    uint256 deadline,  // 签名过期时间（Unix 时间戳）
    uint8 v,           // ┐
    bytes32 r,         // ├── ECDSA 签名的三个分量
    bytes32 s          // ┘
) external;

// 查询某个地址当前的 nonce（防重放攻击的计数器）
function nonces(address owner) external view returns (uint256);

// 返回该合约的 EIP-712 Domain Separator（用于签名域隔离）
function DOMAIN_SEPARATOR() external view returns (bytes32);

B. permit() 函数的执行逻辑

当 permit() 被调用时，合约必须验证以下条件全部满足才能成功：

  1. 时间校验：block.timestamp <= deadline（签名未过期）。
  2. 地址校验：owner != address(0)（防止授权给零地址的僵尸资金）。
  3. Nonce 校验：传入的 nonce 必须等于 nonces[owner] 的当前值（防重放攻击）。
  4. 签名校验：(v, r, s) 是 owner 对以下 EIP-712 结构化消息的合法 secp256k1 签名。

如果所有条件都通过：

  • 设置 allowance[owner][spender] = value（等同于执行了 approve）。
  • 将 nonces[owner] 加 1（使得同一个签名无法被第二次使用）。
  • 触发标准的 Approval 事件。

任何条件不满足 → 必须 revert。

C. EIP-712 签名消息结构

用户在链下签名的消息，严格遵循 EIP-712 的结构化格式：

1. DOMAIN_SEPARATOR（域隔离哈希）：

DOMAIN_SEPARATOR = keccak256(abi.encode(
    keccak256('EIP712Domain(string name,string version,uint256 chainId,address verifyingContract)'),
    keccak256(bytes(name)),        // 代币名称，如 "USD Coin"
    keccak256(bytes(version)),     // 版本号，如 "1"
    chainId,                       // 当前链 ID（防止跨链重放）
    address(this)                  // 合约地址（防止跨合约重放）
));

2. Permit 类型哈希 + 最终签名数据：

bytes32 structHash = keccak256(abi.encode(
    keccak256("Permit(address owner,address spender,uint256 value,uint256 nonce,uint256 deadline)"),
    owner,
    spender,
    value,
    nonce,
    deadline
));

// 最终签名的消息（EIP-712 标准格式）
bytes32 digest = keccak256(abi.encodePacked(
    "\x19\x01",
    DOMAIN_SEPARATOR,
    structHash
));

// 用 ecrecover 恢复签名者地址，必须等于 owner
address recoveredAddress = ecrecover(digest, v, r, s);
require(recoveredAddress == owner, "INVALID_SIGNER");

D. 交易类型与区块结构 (Transaction & Block Header Changes)

无。EIP-2612 是纯合约层标准，不涉及任何底层协议修改。

3. 从宏观到微观的生命周期 (Lifecycle Workflow)

一笔 EIP-2612 permit + Swap 的完整流程（以 Uniswap V2 为例）：

用户钱包（链下）                      链上合约
    │                                    │
    │ ① 用户在 DApp 前端点击"Swap"       │
    │    钱包弹出 EIP-712 签名请求      │
    │    显示："授权 Router 花费 100 USDC"│
    │                                    │
    │ ② 用户点击"签名"（不花 Gas！）     │
    │    获得 (v, r, s) 签名            │
    │                                    │
    │ ③ DApp 将签名 + swap 参数打包     │
    │    发送一笔交易到 Router 合约     │
    │           ─────────────────────►   │
    │                                    │ ④ Router 合约先调用
    │                                    │    USDC.permit(owner, router, 100, ...)
    │                                    │    → USDC 合约验证签名 → approve 生效
    │                                    │
    │                                    │ ⑤ Router 合约再调用
    │                                    │    USDC.transferFrom(owner, pool, 100)
    │                                    │    → 代币转入流动性池
    │                                    │
    │                                    │ ⑥ 完成 swap，用户收到目标代币
    │           ◄─────────────────────   │

关键区别：传统流程需要用户发送 2 笔交易（approve + swap），现在只需要 1 个签名 + 1 笔交易，用户体验和 Gas 成本直接减半。

4. 关键机制深度解析 (Deep Dive)

A. 核心难点：为什么需要 nonce？

如果没有 nonce，同一个签名可以被无限次重复提交（重放攻击）。攻击者截获一个合法签名后，就可以反复调用 permit() 来重新授权。

nonce 机制确保：

  • 每个签名中嵌入了一个唯一的递增计数器。
  • 合约验证时要求签名中的 nonce 必须等于链上存储的 nonces[owner] 当前值。
  • 签名被使用一次后，nonces[owner]++，旧签名中的 nonce 就永远对不上了。
  • 和以太坊交易本身的 nonce 是独立的，互不干扰。

B. 核心难点：DOMAIN_SEPARATOR 为什么要包含 chainId 和 address(this)？

这是为了防止跨链重放和跨合约重放：

攻击场景 防御字段
以太坊主网上的签名被拿到 BSC 上重放 chainId 不同 → 签名无效
USDC 合约的签名被拿到 USDT 合约上重放 address(this) 不同 → 签名无效
硬分叉后旧链上的签名在新链上重放 chainId 不同 → 签名无效

⚠️ 安全注意：如果 DOMAIN_SEPARATOR 在合约部署时计算并缓存为 immutable，那么万一以太坊发生链分裂/硬分叉（chainId 改变），缓存的旧 DOMAIN_SEPARATOR 就会导致跨链重放漏洞。建议每次调用时动态计算，或在 chainId 改变时重新计算。

C. 核心难点：ecrecover 返回零地址的陷阱

EVM 的 ecrecover 预编译合约有一个危险特性：当传入畸形签名（无效的 v, r, s 值）时，它不会 revert，而是静默返回零地址 address(0)。

如果合约没有检查 owner != address(0)，攻击者就可以：

  1. 构造一个 owner = address(0) 的 permit 调用，配上任意无效签名。
  2. ecrecover 返回 address(0) == owner → 校验通过！
  3. 攻击者获得了零地址上所有代币的 approve 权限（"僵尸资金"攻击）。

因此 EIP-2612 明确要求：必须校验 owner != address(0)。

5. 对生态与智能合约的系统级冲击 (Impact Analysis)

  • 对 DApp 开发者的影响：

    • 用户体验革命：用户不再需要两步操作（approve → 操作），现在可以一步完成（签名 + 操作合并为一笔交易）。
    • 无 ETH 操作成为可能：用户只持有 ERC-20 代币（如 USDC）但没有 ETH 时，只要有 Relayer 愿意代付 Gas，用户就可以链下签名完成所有操作。
    • 主流协议全面采用：Uniswap V2/V3 的 LP Token、USDC (Circle)、DAI (MakerDAO)、AAVE aToken 等主流代币均已实现 EIP-2612。

  • 对安全审计的影响：

    • Permit 钓鱼攻击：用户在不知情的情况下签署了恶意的 permit 签名（如钓鱼网站诱导签名 value = type(uint256).max），攻击者就可以拿走用户的所有代币。这已成为当前 Web3 最大的钓鱼攻击向量之一。
      • 防范：钱包需要清晰展示 EIP-712 签名内容；用户需要仔细阅读签名弹窗。
    • 前端运行 (Front-running)：如果用户把 permit 签名发送给 DApp 后端，后端在提交链上交易前，签名可能被第三方截获并抢先提交（front-run）。尽管最终效果对用户一致（授权确实生效了），但可能干扰到特定的业务逻辑流程。
    • 签名审查 (Censorship)：Relayer 收到签名之后可以选择不提交——用户的签名被"扣留"了。deadline 参数可以缓解这个问题：到期后签名自动作废，用户可以重新签名。

  • 与 DAI 的先驱实现的差异： DAI 是最早实现 permit 的代币之一，但其实现与 EIP-2612 标准略有不同：

    差异点 DAI 的 permit EIP-2612 标准
    授权额度 bool allowed（全额或零） uint256 value（精确额度）
    过期参数名 expiry deadline
    规范性 先驱实现，不完全兼容标准 统一标准（Uniswap V2 采用）

  • 长远倒逼演进：

    • EIP-2612 只解决了 approve 的 Gasless 问题。更进一步的 ERC-4337（Account Abstraction 账户抽象） 试图从钱包层面彻底抹去 EOA 限制，使得所有交易都可以由智能合约钱包发起并代付 Gas。
    • EIP-2612 的 permit 模式已被证明是 Account Abstraction 普及前最实用的过渡方案。
