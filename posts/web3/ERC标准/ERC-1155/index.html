ERC-1155: 多代币标准 (Multi-Token Standard)

一句话总结：ERC-1155 是链游和复杂衍生品体系的超级大本营。它打破了“发一个币就得建一个新合约”的铁律，允许在一个独立的智能合约中同时管理无数种真金白银（ERC-20）和独一无二的稀有装备（ERC-721），并引入了史诗级的批量转账 (Batch Transfer) 功能。

📜 核心元数据与参阅资料 (Metadata & References)

信息源原则：底层的通信/代币标准必须以 Ethereum EIP 官方文档 (ethereum.org/EIPS) 和最主流的防呆实现库（如 OpenZeppelin 或 Solmate）的源码作为唯一第一手真理源。切忌大量摘抄二手的科普文章。

  • 官方 EIP 提案编号与链接：EIP-1155: Multi Token Standard
  • 当前标准状态：Final
  • 权威参考实现 (Reference Implementation)：OpenZeppelin contracts/token/ERC1155/ERC1155.sol
  • 依赖基座标准：必须与 ERC-165 (接口自省) 配合使用。

1. 核心理念与痛点解决 (Core Concept)

  • 传统痛点/背景：如果一家链游公司要发售：1 种作为金币的代币（ERC-20），10 把独一无二的神器宝剑（10 个 ERC-721 合约？），以及数千瓶功效相同的红药水。在 ERC-1155 出现之前，项目方必须部署十几个智能合约。更要命的是，如果玩家要把这批物资交易给另一个人，他必须对十几个合约分别发起 approve，然后分十几次付天价 Gas 费转账。
  • 破局思路：由区块链游戏开发商 Enjin 提出的 ERC-1155 引入了**“大一统的代币 ID (Token ID)”** 以及对应的**“份额机制 (Amount)”**。所有的代币种类（无论是 FT 还是 NFT）都被统摄在这个唯一的合约大金库里。
  • 专属名词解释：
    • Semi-Fungible Token (半同质化代币)：比如演唱会 VIP 区的门票，它们作为一个品类（tokenId = 5）总体有 100 张（amount = 100），这 100 张互相之间是完全等价的。但对于整个金库来说，它们又和普通区门票（tokenId = 6）截然不同。

2. 核心接口与数据结构 (Core Interfaces)

相比 ERC-20 的一维哈希表，和 ERC-721 的所有权查寝表，ERC-1155 有着极其雄心勃勃的二维账本结构。

A. 核心状态结构 (State Variables)

// 1. 终极二维账本：查库房。
// 输入 物品分类编号(id) 和 玩家地址(account)，精准返回这个玩家在这个分类下的物资数量（amount）。
mapping(uint256 => mapping(address => uint256)) private _balances;

// 2. 全局授权账本：和 ERC-721 一样，谁能全权代理我名下所有的资产？（注意 1155 没有单品授权）
mapping(address => mapping(address => bool)) private _operatorApprovals;

B. 关键机制与底层逻辑 (Core Functions)

1. safeTransferFrom (单品过户)

将指定分类（ID）的代币进行转移。

  • 白话逻辑解析：底层就是纯粹的二维账本加减法。并在末尾附带了类似 ERC-721 的防呆敲门机制 onERC1155Received。
  • 核心源码骨架： 
    function safeTransferFrom(address from, address to, uint256 id, uint256 amount) public {
    require(from == msg.sender || _operatorApprovals[from][msg.sender], "NOT_AUTHORIZED");

    // 纯粹的二维账本平账逻辑
    uint256 fromBalance = _balances[id][from];
    require(fromBalance >= amount, "INSUFFICIENT_BALANCE");
    unchecked { _balances[id][from] = fromBalance - amount; }
    
    _balances[id][to] += amount;

    emit TransferSingle(msg.sender, from, to, id, amount);
    
    // ...这里省略了安全敲门机制...
}

2. safeBatchTransferFrom (打包批量过户)

1155 的杀手级绝对核心：将一大批不同的药水、宝剑打包在一个包裹里发给对方，省去十几次独立转账的 Gas 费。

  • 白话逻辑解析：用数组将多个物品 ID 和各自对应的数量传进去，在合约内部只做一次非常紧凑的 for 循环扣款。
  • 核心源码骨架： 
    function safeBatchTransferFrom(address from, address to, uint256[] memory ids, uint256[] memory amounts) public {
    require(ids.length == amounts.length, "LENGTH_MISMATCH");
    require(from == msg.sender || _operatorApprovals[from][msg.sender], "NOT_AUTHORIZED");

    // 最硬核的部分：在一次交易内疯狂循环平账所有物资
    for (uint256 i = 0; i < ids.length; ++i) {
        uint256 id = ids[i];
        uint256 amount = amounts[i];

        uint256 fromBalance = _balances[id][from];
        require(fromBalance >= amount, "INSUFFICIENT_BALANCE");
        unchecked { _balances[id][from] = fromBalance - amount; }
        _balances[id][to] += amount;
    }

    emit TransferBatch(msg.sender, from, to, ids, amounts);
    // ...这里省略了批量的安全敲门机制...
}

C. 关键事件 (Events)

  • TransferSingle(...)：发了一件或者同种类的几件单品。
  • TransferBatch(...)：发了一个混合资产大礼包。这是 ERC-1155 独有的高效率事件，要求所有的链下索引器（The Graph/区块浏览器）具备强大的数组解包渲染解析能力。

3. 标准业务工作流 (Standard Workflow)

你在链游中打怪掉落（Mint）并把战利品打包交易给商人的流程：

  1. 统一印钞 (Mint / MintBatch)：怪物死亡，系统调用内部函数 _mintBatch，一次性向你的地址写入：100 枚金币 (id=1) 和 1 把屠龙刀 (id=2)。触发 TransferBatch 事件。
  2. 委托商会 (Approve)：你找地精商人代卖，调用 setApprovalForAll(商人合约, true)。注意：ERC-1155 为了简化代码，直接砍掉了只针对单品的 approve，只要授权，就是允许对方动用你在这个1155全宇宙合约里的所有物资。
  3. 一口气交割 (Batch TransferFrom)：买家付款，商人合约被唤醒，直接调用 safeBatchTransferFrom(你的地址, 买家地址, [1, 2], [100, 1])。只需一笔微不足道的运费（Gas），100 块钱和 1 把刀瞬间过户到了买家账上。

4. 关键机制与难点解析 (Key Mechanics)

对于协议中那些绕人的数学计算、状态机流转或者是舍入取整规则，必须配以核心公式推导或精简版代码辅助理解。

A. 核心难点一：FT 和 NFT 在同一个账本里的坍缩表达

  • 原理讲解：初学者最迷惑的是，如果大一统了，那我在代码层面到底怎么向系统证明我发的是一个不可替代的 NFT？答案极其粗暴简单：控制单据印量 (Supply)。
  • 数学推导/公式： $$NFT \iff TotalSupply(ID) == 1$$ $$FT \iff TotalSupply(ID) > 1$$
  • 业务逻辑：在 ERC-1155 的内部，它根本不认识什么是 NFT，它只认数字。如果你告诉系统，铸造一把名为“埃辛诺斯战刃”的武器（tokenId = 123），并且规定这个 123 全宇宙只铸造唯一的一张 (amount = 1)，那么它在现实物理表现上，就是那个永远无法被分割和等价替换的超级 ERC-721 NFT。如果 amount 发出去 1 亿个，那就是 ERC-20。一统江湖，大道至简。

B. 核心难点二：成倍暴击的“批量接收”重入攻击

  • 原理讲解：刚才在 ERC-721 里讲过，如果目标是合约，我们要去敲门（触发接收方的 onERC721Received 回调）。ERC-1155 为了打包传输，更是直接开创了宏伟的 onERC1155BatchReceived 钩子。 这极度危险！在执行 safeBatchTransferFrom 时，黑客如果在这个批量的钩子中写上恶意的再入攻击代码，相当于他拿着一把无数次触发回调的机关枪，可以在你一笔交易未落账的极短缝隙里，把你上亿价值的资产全数转移。
  • 核心逻辑代码 (Solidity)： 
    // 在打包转移函数的极度危险地带
function safeBatchTransferFrom(address from, address to, uint256[] memory ids, uint256[] memory amounts, bytes memory data) public virtual override {
    // 永远记住：先把自己的内帐平掉（Checks & Effects）
    for (uint256 i = 0; i < ids.length; ++i) {
        uint256 id = ids[i];
        uint256 amount = amounts[i];
        // 如果这里没扣款就先去调下面那个敲门函数，金库瞬间搬空
        _balances[id][from] -= amount; 
        _balances[id][to] += amount;
    }
    // 所有账本全部扣死锁定了，再去敲那些图谋不轨的黑客合约的门（Interactions）
    _doSafeBatchTransferAcceptanceCheck(operator, from, to, ids, amounts, data);
}
    代码拆解：注意 _doSafeBatchTransferAcceptanceCheck 永远被放在那个计算余额的 for 循环之后。这是 EVM 世界的生存底线。如果把顺序写反，黑客在回调里反复调用 safeBatchTransferFrom，而旧余额却还没被减去，等于无损刷钱机。

5. 周边生态与后续演进 (Ecosystem)

  • 主流采用方：Enjin, The Sandbox, Adidas Originals 衍生品等绝大部分包含不同会员等级 / 多游戏道具的大型 Dapp。
  • 标准的局限性：
    1. 它太庞大笨重了。虽然它的出发点是好的，但在很多纯粹只需要处理加密艺术品 (纯粹的 1/1 NFT) 的场景下，强制引入 _balances[id][account] 的二维表显得极为冗余和低效。
    2. 强制剥夺了用户的单点防御能力。刚才说过它取消了针对单品 Approve 的权限，用户一旦把 ERC-1155 系列授权给某个市场，就意味着彻底暴露了自己在该宇宙下的所有资产。一旦市场被黑，裤衩都不剩。
