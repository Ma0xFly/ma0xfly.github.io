ERC-2612: Permit 扩展标准

一句话总结：通过 EIP-712 签名机制替代传统 approve 交易，实现链下授权链上执行，节省一笔交易并提升用户体验。

📜 核心元数据与参阅资料 (Metadata & References)

信息源原则：底层的通信/代币标准必须以 Ethereum EIP 官方文档 (ethereum.org/EIPS) 和最主流的防呆实现库（如 OpenZeppelin 或 Solmate）的源码作为唯一第一手真理源。切忌大量摘抄二手的科普文章。

  • 官方 EIP 提案编号与链接：[EIP-2612: Permit (https://eips.ethereum.org/EIPS/eip-2612)]
  • 当前标准状态：Review
  • 权威参考实现 (Reference Implementation)：OpenZeppelin contracts/token/ERC20/extensions/ERC20Permit.sol
  • 依赖基座标准：基于 ERC-20 扩展，依赖 EIP-712 类型化数据签名标准

1. 核心理念与痛点解决 (Core Concept)

  • 传统痛点/背景：在没有 ERC-2612 之前，用户想要将 ERC-20 代币转账给第三方合约（如 Uniswap），必须先调用 approve(spender, amount) 授权，然后再调用合约执行交易。这不仅消耗两笔 Gas，还容易发生授权金额过大导致的安全风险。
  • 破局思路：引入 permit 函数，允许用户在链下对授权消息进行 EIP-712 类型化签名，第三方合约直接调用 permit + transferFrom 或合二为一，用户无需发起单独的 approve 交易。
  • 专属名词解释：
    • Permit：通过签名替代 approve 的授权机制
    • Nonce：每个地址独立的计数器，防止签名重放攻击
    • Deadline：签名的有效期截止时间戳

2. 核心接口与数据结构 (Core Interfaces)

极度重要：必须在这里贴出协议最本质的状态变量（State Variables），因为任何智能合约的本质都是对底层哈希账本（mapping）的读写。核心函数的解释也必须发掘到源码，附带最核心的 Solidity 代码骨架（剥离安全防呆校验后的纯净业务逻辑）。

A. 核心状态结构 (State Variables)

mapping(address => uint256) private _nonces;

bytes32 private constant _PERMIT_TYPEHASH = keccak256(
    "Permit(address owner,address spender,uint256 value,uint256 nonce,uint256 deadline)"
);

bytes32 private _DOMAIN_SEPARATOR;
  • _nonces：记录每个地址的签名使用次数，防止重放攻击
  • _PERMIT_TYPEHASH：EIP-712 类型的哈希值，用于验证签名
  • _DOMAIN_SEPARATOR：EIP-712 域分隔符，隔离不同合约/链的签名

B. 关键机制与底层逻辑 (Core Functions)

这里不要用简单的表格。必须展示核心函数实际改变底层账本的简版源码代码片段。

permit(owner, spender, value, deadline, v, r, s) (通过签名授权)

  • 白话逻辑解析：验证签名合法性后，直接修改 _allowances 账本，相当于链下签名链上生效。推模式（用户主动授权）。
  • 核心源码骨架 (Solidity)：
    function permit(
    address owner,
    address spender,
    uint256 value,
    uint256 deadline,
    uint8 v,
    bytes32 r,
    bytes32 s
) public {
    require(block.timestamp <= deadline, "Permit expired");
    
    bytes32 structHash = keccak256(abi.encode(
        _PERMIT_TYPEHASH,
        owner,
        spender,
        value,
        _nonces[owner]++,
        deadline
    ));
    
    bytes32 digest = keccak256(abi.encodePacked(
        "\x19\x01",
        _DOMAIN_SEPARATOR,
        structHash
    ));
    
    address signer = ecrecover(digest, v, r, s);
    require(signer == owner, "Invalid signature");
    
    _allowances[owner][spender] = value;
    
    emit Approval(owner, spender, value);
}

nonces(owner) (查询当前 nonce 值)

  • 白话逻辑解析：返回地址当前的 nonce，用于链下构建正确的签名。
  • 核心源码骨架 (Solidity)：
    function nonces(address owner) public view returns (uint256) {
    return _nonces[owner];
}

C. 关键事件 (Events)

  • Approval(owner, spender, value)：permit 成功时触发，与 ERC-20 标准一致，方便链下索引授权状态变化
  • 事件对于 The Graph 索引器追踪授权历史、钱包 UI 显示授权状态至关重要

3. 标准业务工作流 (Standard Workflow)

将上面零散的函数串联起来，还原一个真实的交互业务闭环。

一个完整的 Permit 授权 + 转账流程：

  1. 链下签名：用户在钱包或前端构建 EIP-712 签名消息，包含 owner、spender、value、nonce、deadline，生成 (v, r, s) 签名。
  2. 提交 Permit：用户或第三方合约调用 permit(owner, spender, value, deadline, v, r, s)，合约验证签名并更新 _allowances[owner][spender]。
  3. 执行 TransferFrom：合约立即调用 transferFrom(owner, recipient, amount)，基于刚才的授权进行转账。
  4. Nonce 自增：permit 函数内部执行 _nonces[owner]++，该签名使用后失效。

4. 关键机制与难点解析 (Key Mechanics)

这是 ERC 标准笔记中最具价值的部分。对于协议中那些绕人的数学计算、状态机流转或者是舍入取整规则，必须配以核心公式推导或精简版代码辅助理解。

A. 核心难点一：EIP-712 类型化数据签名

  • 原理讲解：传统的 eth_sign 方式容易被钓鱼攻击，EIP-712 引入结构化数据签名，用户看到的是可读的 JSON 对象而非乱码，大幅提升安全性。

  • 数学推导/公式： $$\text{digest} = \text{keccak256}("\x19\x01" || \text{DOMAIN_SEPARATOR} || \text{structHash})$$ $$\text{structHash} = \text{keccak256}(\text{encode}(\text{TYPEHASH}, \text{owner}, \text{spender}, \text{value}, \text{nonce}, \text{deadline}))$$

  • 核心逻辑代码 (Solidity)：

    // 提取出协议中最能体现该机制的 5-10 行核心、去除了防呆修饰符的伪代码/源码。
function _hashTypedDataV4(bytes32 structHash) internal view returns (bytes32) {
    return keccak256(abi.encodePacked("\x19\x01", _DOMAIN_SEPARATOR, structHash));
}

// _DOMAIN_SEPARATOR 构建
bytes32 _DOMAIN_SEPARATOR = keccak256(abi.encode(
    keccak256("EIP712Domain(string name,string version,uint256 chainId,address verifyingContract)"),
    keccak256(bytes(name)),
    keccak256(bytes(version)),
    block.chainid,
    address(this)
));

    代码拆解："\x19\x01" 是 EIP-191 的版本标识，_DOMAIN_SEPARATOR 绑定了合约名、版本、链 ID 和合约地址，确保不同链/合约的签名互不通用。structHash 对用户输入的参数进行哈希，保证签名的唯一性。

  • 主流采用方：

    • Uniswap V2/V3：支持 permit 授权交易池
    • 1inch：DEX 聚合器广泛采用
    • Aave/V3：借贷协议支持 permit 抵押资产
    • USDC/USDT (部分实现)：部分稳定币已集成 permit

  • 标准的局限性：

    • 增加了合约代码复杂度和 Gas 成本（需要存储 nonce 和 DOMAIN_SEPARATOR）
    • 用户可能不理解签名含义仍需教育
    • 不是所有 ERC-20 代币都实现了此标准

  • 后续打补丁的扩展提案：

    • ERC-4494：将 permit 机制扩展到 ERC-721（基于签名的 NFT 授权）
    • Dai 的 permit：在 ERC-2612 定稿前 Dai 协议先行实现了类似功能
