ERC-1363: 可支付型代币扩展 (Payable Token)

一句话总结：ERC-1363 是对 ERC-20 标准最安全、最成功的演进补丁。它吸取了前辈们的教训，用标准化的 TransferAndCall 解决了双重审批（Approve + Execute）的痛点，允许用户在一笔交易中完成代币转移和调用外部合约逻辑。

📜 核心元数据与参阅资料 (Metadata & References)

信息源原则：底层的通信/代币标准必须以 Ethereum EIP 官方文档 (ethereum.org/EIPS) 和最主流的防呆实现库（如 OpenZeppelin 或 Solmate）的源码作为唯一第一手真理源。切忌大量摘抄二手的科普文章。

  • 官方 EIP 提案编号与链接：EIP-1363: Payable Token
  • 当前标准状态：Final
  • 权威参考实现 (Reference Implementation)：OpenZeppelin contracts/token/ERC20/extensions/ERC1363.sol
  • 依赖基座标准：完全继承自 ERC-20。且大量借鉴了 ERC-721 的 onReceived 安全回调设计。

1. 核心理念与痛点解决 (Core Concept)

  • 传统痛点/背景：ERC-20 的 approve + transferFrom 的两步付账极其繁琐，耗时且消耗双倍 Gas。虽然历史上出现过试图一步解决转账加执行的 ERC-827（因安全漏洞退市）和 ERC-223（因兼容性太差未普及），但市场对“一键式可支付代币”的需求从未停止。
  • 破局思路：ERC-1363 完美平衡了安全和易用性。它并没有盲从 ERC-827 那样允许携带炸弹包裹（任意 calldata 代签），而是强制规定：外部的目标合约必须实现两个固定的、像门铃一样的专用钩子函数（onTransferReceived 和 onApprovalReceived）。这样代币合约转账后，只会去按这两个专用的门铃，绝不会发生越权的任意函数调用。

2. 核心接口与数据结构 (Core Interfaces)

该标准完全复用了 ERC-20 的 _balances 余额体系，它并未创造新的账本类型，而是从执行流入手，开辟了附带回调提醒的高铁直达专线。

A. 核心状态结构 (State Variables)

// 无任何新增映射，完全共享原生 ERC-20 的底层防篡改记账池：
mapping(address => uint256) private _balances;
mapping(address => mapping(address => uint256)) private _allowances;

B. 关键机制与底层逻辑 (Core Functions)

1. transferAndCall(to, value, data) (转账并强制敲门)

一站式划转资金并触发业务的灵魂函数。

  • 白话逻辑解析：底层依旧走 _balances 的减发加收，钱结清后的一瞬间，该代币合约化身顺丰快递员，拿着 data（附言包裹）去猛按目标 to 地址门上固定的 onTransferReceived 门铃。如果对方不开门（没实现该收款函数或执行报错），那么钱将原路全额退还，交易连坐取消。
  • 核心源码骨架：
    function transferAndCall(address to, uint256 value, bytes calldata data) public returns (bool) {
    // 1. 先把底层原生的转账账本平平稳稳地结清
    transfer(to, value); 
    
    // 2. 强制触发目标的专属接客回调门铃！(彻底堵死 827 那个危险的底层裸 call)
    require(
        _checkAndCallTransfer(msg.sender, to, value, data),
        "ERC1363: _checkAndCallTransfer reverts"
    );
    return true;
}

function _checkAndCallTransfer(address from, address to, uint256 value, bytes memory data) internal returns (bool) {
    if (!to.isContract()) return false; 
    // 必须安全、受限地调用标准写死的 onTransferReceived 对接接口
    bytes4 retval = IERC1363Receiver(to).onTransferReceived(msg.sender, from, value, data);
    return (retval == IERC1363Receiver.onTransferReceived.selector);
}

2. approveAndCall(spender, value, data) (授权并主动唤醒)

  • 白话逻辑解析：与上面的转账敲门异曲同工。你签发了天量授权批文后，代币合约主动去敲那个业务金库的 onApprovalReceived 专用门铃并大喊：“喂！有位散户刚批给你 100 U 的刷卡额度，这是他托我带来的执行参数 data，你想干啥现在赶紧去划卡执行吧！”。这就彻底终结了古典 DeFi 时代用户先点 Approve 确认，等慢吞吞出块后再去单独点一次 Swap 执行的剧痛交互。

C. 关键事件 (Events)

该标准并未额外规定花哨的高级事件，仍旧依靠并重度复用底层 ERC-20 强制下发的 Transfer 和 Approval 经典事件播报。

3. 标准业务工作流 (Standard Workflow)

你在支持 1363 标准的去中心化域名注册局里购买一个域名的极速流程：

  1. 一笔带过 (Execute)：你在前端点击“支付 100 币注册域名”。系统直接调用代币的 transferAndCall(注册局合约地址, 100, "域名数据")。
  2. 结算资金 (Accounting)：代币合约扣除你的 100 个币，加给注册局。
  3. 安全唤醒 (Safe Calling)：代币合约主动敲注册局的门，调用了注册局早早写好的专职迎客机器人 注册局.onTransferReceived(...)。
  4. 自动化发货 (Completion)：注册局在此函数中看到钱已到账，立刻用传进来的 data 给你分配了域名。一切操作在短短几毫秒的一个区块交易中水到渠成。

4. 关键机制与难点解析 (Key Mechanics)

对于协议中那些绕人的数学计算、状态机流转或者是舍入取整规则，必须配以核心公式推导或精简版代码辅助理解。

A. 核心难点一：如何避免重走 827 借刀杀人的老路？

  • 原理讲解：为了防止黑客利用代币合约的无上权限去执行恶意提款命令。ERC-1363 在底层代码里彻底封死了 call 任意函数的自由度。它强制使用接口调用。
  • 核心逻辑代码 (Solidity)：
    // 在 transferAndCall 的最后一步，有一段极其严格的回调验证
function _checkOnTransferReceived(address from, address to, uint256 value, bytes memory data) private returns (bool) {
    if (to.code.length == 0) {
        return false; // 不是合约，就不敲门了，兼容普通人转账
    }
    try IERC1363Receiver(to).onTransferReceived(msg.sender, from, value, data) returns (bytes4 retval) {
        // 这里的逻辑直接照抄了极度成熟安全的 ERC-721 防呆敲门机制
        return retval == IERC1363Receiver.onTransferReceived.selector;
    } catch (bytes memory reason) {
        // 不懂规矩的接收方，直接拉黑全盘回滚
        revert("ERC1363: transfer to non ERC1363Receiver implementer");
    }
}
    代码拆解：你看，由于代币合约在这里调用的是明明白白、死死写定名字的 .onTransferReceived 函数，所以对方就算收到了调用，也由于被局限在这个专门用来记账收钱的特定黑盒子里，无法让代币合约去帮它越权提款。这就叫真正的“戴着镣铐起舞”。

5. 周边生态与后续演进 (Ecosystem)

  • 主流采用方：作为 OpenZeppelin 的标配扩展库，越来越多新发行的复杂全链游戏代币、DAO 门票资产选择附加这一协议以优化前端体验。
  • 标准的局限性：
    1. 需要目标接收方（比如老牌的 DEX 池子）同样配合升级去实现 IERC1363Receiver 接口。如果对方是个老当益壮的古董合约，这套高级的敲门系统依然用不上。
  • 扩展提案：
    • ERC-2612 (Permit)：正如前面所说，当今的 Web3 更倾向于在链下用私钥签好授权额度，让旧式金库也能在一次交易内完工，而不是指望万千老金库为了 1363 标准全部推翻重写。
