ERC-20: 同质化代币通用标准

一句话总结：ERC-20 是以太坊上最早、最普及的“同质化代币（Fungible Token）”智能合约记账标准。它统一了全网所有代币的授权、转账接口，是 DeFi 和整个 Web3 金融大厦的基础地砖。

📜 核心元数据与参阅资料 (Metadata & References)

信息源原则：底层的通信/代币标准必须以 Ethereum EIP 官方文档 (ethereum.org/EIPS) 和最主流的防呆实现库（如 OpenZeppelin 或 Solmate）的源码作为唯一第一手真理源。切忌大量摘抄二手的科普文章。

  • 官方 EIP 提案编号与链接：EIP-20: Token Standard
  • 当前标准状态：Final
  • 权威参考实现 (Reference Implementation)：OpenZeppelin contracts/token/ERC20/ERC20.sol
  • 依赖基座标准：无，它是大多数标准的基础。

1. 核心理念与痛点解决 (Core Concept)

  • 传统痛点/背景：在 2015 年以太坊刚诞生时，每个人发行的“积分”或“代币”合约里的函数名千奇百怪。有的转账叫 send()，有的叫 transferMoney()。这导致早期的去中心化交易所（DEX）或钱包（如 MetaMask）根本无法兼容和显示不同的代币。
  • 破局思路：ERC-20 强制规定了一套仅有 6 个函数和 2 个事件的极简状态机接口。只要你的合约拥有名为 balanceOf、transfer 和 approve 等特定名称和参数的函数，全网所有的钱包和协议就能无缝识别和划转你的代币。
  • 专属名词解释：
    • Fungible Token (同质化代币)：每一个单位的代币与另一个单位的代币在功能和价值上完全等同（就像两张崭新的百元实体现钞），这与 ERC-721 (非同质化 NFT) 形成鲜明对比。

2. 核心接口与数据结构 (Core Interfaces)

ERC-20 的本质是账本。任何复杂的业务逻辑最终都是在修改以下两个底层 mapping。

A. 核心状态结构 (State Variables)

// 1. 余额账本：记录每个地址持有的真金白银数量
mapping(address => uint256) private _balances;

// 2. 授权账本：记录 授权人(owner) -> 被授权人(spender) -> 额度(amount)
mapping(address => mapping(address => uint256)) private _allowances;

// 3. 代币元数据
uint256 private _totalSupply; // 总供应量
string private _name;         // 代币名称 (如 "Tether USD")
string private _symbol;       // 代币符号 (如 "USDT")

B. 关键机制与底层逻辑 (Core Functions)

1. _mint (底层铸造) 与 _burn (底层销毁)

这是代币诞生和消亡的唯一途径，通常只由特定身份（如矿池合约、项目方）调用。

  • 核心逻辑：不仅修改 _totalSupply 和指定地址的 _balances，核心规矩是无论印钞还是销毁，都必须强制触发 Transfer 事件，借用 0x0 原生黑洞地址来充当对手方。
  • 核心源码骨架： 
    // 铸造示例
_totalSupply += amount;
_balances[account] += amount;
emit Transfer(address(0), account, amount); // 从 0x0 凭空变出钱

// 销毁示例
_balances[account] -= amount;
_totalSupply -= amount;
emit Transfer(account, address(0), amount); // 把钱打入 0x0 销毁

2. transfer (直接转账)

最基础的“推 (Push)”模式。你主动把自己的钱推给别人。

  • 核心源码骨架： 
    function transfer(address to, uint256 amount) public returns (bool) {
    uint256 fromBalance = _balances[msg.sender];
    require(fromBalance >= amount, "ERC20: transfer amount exceeds balance");
    
    // 修改底层状态变量（最纯粹的加减法）
    unchecked { _balances[msg.sender] = fromBalance - amount; }
    _balances[to] += amount;

    emit Transfer(msg.sender, to, amount);
    return true;
}

3. approve (授权) 与 transferFrom (代为拉取)

DeFi 大厦的基石接口。允许第三方（如 Uniswap Router）以“拉 (Pull)”模式从你账上提款。

  • 核心源码骨架： 
    // 你在前端点击“授权”：给第三方签发一张额度支票
function approve(address spender, uint256 amount) public returns (bool) {
    // 直指核心账本：我(msg.sender)允许那个合约(spender)花 amount 这么多钱
    _allowances[msg.sender][spender] = amount;
    emit Approval(msg.sender, spender, amount);
    return true;
}

// 第三方拿着支票来提钱：检查支票额度，扣减支票，最后再进行底层转账
function transferFrom(address from, address to, uint256 amount) public returns (bool) {
    // 这里的 msg.sender 是那个第三方合约
    uint256 currentAllowance = _allowances[from][msg.sender];
    if (currentAllowance != type(uint256).max) { // 无限授权免扣额度的 Gas 优化
        require(currentAllowance >= amount, "ERC20: insufficient allowance");
        unchecked { _allowances[from][msg.sender] = currentAllowance - amount; }
    }
    
    // 复用底层的 transfer 转账账本加减法逻辑
    _balances[from] -= amount;
    _balances[to] += amount;
    emit Transfer(from, to, amount);

    return true;
}

C. 关键事件 (Events)

  • Transfer(address indexed from, address indexed to, uint256 value)：只要发生了余额变动（无论是主动转账还是被拉走，甚至是 Mint/Burn），必须强制触发此事件。所有的区块链浏览器（如 Etherscan）和数据看板（如 Dune）都是通过这个事件来计算个人的账户余额和代币轨迹的。如果少写了这个核心事件，前端界面将永远无法察觉资产转移。
  • Approval(address indexed owner, address indexed spender, uint256 value)：当调用 approve 修改授权额度时触发。

3. 标准业务工作流 (Standard Workflow)

在 DeFi 协议（如 Uniswap 兑换）中最经典的双步“拉取支付 (Pull-Payment)”流程：

  1. 签订契约 (Approve)：你（User）在前端点击“授权 USDT”。底层调用 USDT 合约的 approve(Uniswap Router 地址, 1000)，允许 Router 动用你 1000 个 USDT。此时钱还在你账上。
  2. 发起指令 (Execution)：你点击“确交换”。底层调用 Uniswap Router 的 swapExactTokensForTokens 函数。
  3. 金库划款 (TransferFrom)：Router 被唤醒，它在中途调用 USDT 合约的 transferFrom(你的地址, 对手方/金库地址, 1000)。
  4. 验证与记账 (Accounting)：USDT 合约检查内部的 allowance 账本，确认你确实给过 Router 1000 的额度。验证通过后，扣除这 1000 的授权额度，并将你 balanceOf 里的 1000 USDT 真实地转移走。

4. 关键机制与难点解析 (Key Mechanics)

对于协议中那些绕人的数学计算、状态机流转或者是舍入取整规则，必须配以核心公式推导或精简版代码辅助理解。

A. 核心难点一：授权踩踏漏洞（Approve Race Condition / Front-running）

  • 原理讲解：这是一个极其经典的**条件竞争（抢跑）**陷阱。假设你先授权了 Alice 100 额度（approve(Alice, 100)）。几天后你反悔了，想把额度改成 50，于是你提交了一笔 approve(Alice, 50) 的交易。恶意节点 Alice 在内存池（Mempool）中看到了你这笔要把额度降为 50 的交易。她立刻支付极高的 Gas 费，抢在你的交易确认前，先执行 transferFrom 把原来的 100 额度全部提走。紧接着你的 approve(50) 交易确认，Alice 的账本又刷新出了 50 的额度，她再次提走。最终你本意是想降额到 50，却一共损失了 150！
  • 数学推导/公式：$\text{损失总量} = \text{旧的Approve额度} + \text{新的Approve额度}$
  • 核心逻辑代码 (Solidity)： 
    // 导致漏洞的原始 ERC-20 approve 实现方式（直接覆盖变量）
function approve(address spender, uint256 amount) public returns (bool) {
    // 漏洞根源：无论之前额度是多少，直接暴力覆盖赋值
    allowance[msg.sender][spender] = amount; 
    emit Approval(msg.sender, spender, amount);
    return true;
}
    代码拆解：因为使用了 = 直接覆盖，导致攻击者可以在覆盖生效前耗尽旧值。业界通用的解决方案是：在重置额度前，强制要求先将其归零 approve(spender, 0)（早期的 USDT 就是这么强烈限制的），或者使用 OpenZeppelin 后续补充的安全扩展库工具 safeIncreaseAllowance 和 safeDecreaseAllowance 来做加减法而不是直接覆盖。

B. 核心难点二：沉睡的无主资产（代币丢失）

  • 原理讲解：用户使用手机钱包时，如果直接用 transfer 将 ERC-20 代币转给了一个不支持该代币记账和转出的智能合约地址。这笔钱即使到了合约账上，由于合约没写提取逻辑，这笔代币将永远锁死在里面，无法追回。为了解决这种“推模式”带来的本金误打丢失问题，诞生了后续的 ERC-223 和带有钩子接收函数的 ERC-777 标准，但因为历史基数，目前市场主要还是妥协使用 ERC-20。

5. 周边生态与后续演进 (Ecosystem)

  • 主流采用方：整个 Ethereum 和 EVM 兼容链 99% 的流通货币（USDT, USDC, UNI, SHIB 等）。
  • 标准的局限性：
    1. 必须使用两笔独立的交易（先 Approve 签发额度合约，后 Execute 让协议拉走资金），这导致用户体验极其糟糕，且浪费了双倍的 Gas 费。
    2. 收款方（尤其是智能合约）无法主动得知自己收到了钱，难以触发后续的自动化操作。
  • 后续打补丁的扩展提案：
    • ERC-2612 (Permit 签名授权)：极大地优化了双重交易问题。用户可以在链下使用私钥签名一份授权证明（产生 v, r, s 签名值），然后合并在一次调用中发给协议。协议用 permit() 解析签名并在一次交易中同时完成授权与扣款。目前 Uniswap 的 Permit2 正是基于这一理念发扬光大的。
