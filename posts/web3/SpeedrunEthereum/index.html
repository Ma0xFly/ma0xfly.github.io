通关ai使用指南

SpeedrunEthereum（基于 Scaffold-ETH 2）是练习“高分交互模式”的绝佳靶场。因为它不仅涉及 Solidity 合约逻辑（你的专业核心），还强制要求你理解 前端交互（React/Wagmi）——这往往是安全审计员的盲区，却是理解“业务逻辑漏洞”的关键。

结合你的 Web3 安全背景 和 Foundry 偏好，这里有一套专门针对 SpeedrunEthereum 的**“三层通关法”**，旨在把每个挑战吃透，而不是飘过。

第一层：核心合约逻辑（采用 Mode C - 概念引导 + 自己手写）

你是安全方向的学生，绝对不要让 AI 直接写合约代码（特别是 Staker.sol 或 Vendor.sol）。这些逻辑是你未来的饭碗。

❌ 低分做法：

“请给我写一个去中心化众筹合约，要求有 deadline 和提款功能。”（复制粘贴 → 结束）

✅ 高分做法（Mode C）：

在写代码前，先用 AI “对齐伪代码”。

  • 场景： Challenge 1 (Decentralized Staking App)

  • Prompt 模板：

    “我正在做 Staking Challenge。请不要给我代码。

    我想理清 execute() 函数的状态机逻辑：

    1. 如果我在 deadline 之前调用会发生什么？

    2. 如果 balance 没达到 threshold 但时间到了，状态应该怎么变？

      请用文字描述这几种状态转换，并指出如果不加 external 修饰符可能带来的安全隐患。”

  • 你的行动： 根据 AI 的逻辑描述，自己在 Solidity 里逐行实现。

第二层：前端与交互钩子（采用 Mode A - 生成 + 强迫理解）

Scaffold-ETH 2 封装了很多 React Hooks（如 useScaffoldWriteContract）。作为安全工程师，你不需要成为前端专家，但你必须看懂前端是如何构建交易参数的，因为很多攻击（如前端欺骗）发生在这里。

❌ 低分做法：

“前端报错了，帮我修好这个按钮。”

✅ 高分做法（Mode A）：

让 AI 生成前端组件，但必须逼问数据流向。

  • 场景： Challenge 2 (Token Vendor) - 前端购买 Token 逻辑

  • Prompt 模板：

    “这段 React 代码调用了 vendor.buyTokens()。

    1. 请解释 msg.value 是如何在前端被封装进交易数据的？

    2. useScaffoldWatchContractEvent 这个钩子是如何监听到链上事件并更新 UI 的？

    3. 如果恶意用户直接绕过前端调用合约，这段前端逻辑里有哪些检查是链上合约必须重复做的？”

  • 你的行动： 能够指着前端代码说出：“这行代码对应着 EVM 里的 calldata 构建。”

第三层：红蓝对抗（你的专属 Bonus - Mode B + 威胁建模）

这是最适合你“Web3 安全工程师”人设的环节。通关不是结束，Hacking 才是开始。由于 SpeedrunEthereum 自带测试通常是 JS/Hardhat 写的，你可以利用这一点进行 Foundry 迁移训练。

异常处理

为什么是require不是if else

这是一个非常深刻的问题，触及了区块链与传统编程最大的逻辑区别。

在传统编程（如 C++、Java、Python）中，if-else 是用来控制流程的。 但在 Solidity 中，require 不仅仅是控制流程，它是一个**“带有时间倒流功能的断路器”**。

主要有三个核心原因：状态回滚（Atomicity）、Gas 退款、代码防御性。

核心原因：原子性与“时间倒流” (State Reversion)

这是最关键的一点。区块链交易必须是原子的（Atomic）——要么全成功，要么全失败（什么都没发生）。

  • 使用 if-else (软失败)： 如果你用 if 发现条件不对并 return，程序停止了，但之前已经发生的修改会被保留。

  • 危险场景：你先扣了用户的钱，然后用 if 检查库存，发现没货了，于是 return。

  • 结果：钱扣了，货没给，函数结束了。这是灾难。

  • 使用 require (硬回滚)： 一旦 require 失败，EVM 会触发 REVERT 操作码。

  • 结果：它会像按下“Ctrl+Z”一样，撤销这笔交易中做过的所有状态修改（包括余额变动、变量赋值）。就像这笔交易从未发生过一样。

erro revert为什么比传统的 require String 好

A. 极致的 Gas 节省（最重要原因）

这是核心差异。

  • 传统的写法：

    // 这种写法很昂贵！
require(block.timestamp >= deadline, "Wait until the deadline has passed");

    • 原理：EVM 需要把 "Wait until the deadline has passed" 这串长字符串存储在字节码中，并且在报错时通过内存返回。

    • 成本：字符串越长，Gas 越贵。哪怕是一个字母，都要占空间。

  • 你的写法（Custom Error）：

    // 定义错误（类似定义一个函数签名）
error TooEarly(); 

if (block.timestamp < deadline) {
    revert TooEarly(); // 这种写法非常便宜！
}

    • 原理：EVM 不存字符串。它只存 TooEarly() 的哈希值前 4 个字节（Selector）。

    • 成本：无论错误名字叫 A() 还是 ThisIsAVeryLongErrorName()，它在运行时只占 4 个字节。这是固定成本，极其便宜。

B. 支持动态参数（调试神器）

如果你用字符串 require，想告诉用户“还要等多久”，你得疯狂拼接字符串（Solidity 拼接字符串非常痛苦且费 Gas）。

但在 revert 中，你可以像传函数参数一样传值：

Solidity

// 定义带参数的错误
error TooEarly(uint256 currentTime, uint256 deadline);

if (block.timestamp < deadline) {
    // 直接把数据抛出去，前端能收到具体数值！
    revert TooEarly(block.timestamp, deadline);
}

前端或者区块链浏览器会收到：TooEarly(1700000000, 1800000000)。用户瞬间明白：“哦，还要等一亿秒”。

2. 进化史：从 require 到 revert

为了让你更清楚 Web3 开发的演进，请看这个对比：

阶段 写法 评价 典型场景
石器时代 require(cond, "Error String") 耗气、笨重 快速原型开发、老旧合约
青铜时代 revert("Error String") 同上 复杂的逻辑分支中手动触发
**黄金时代 ** if (!cond) revert CustomError() 省气、专业 当前的行业标准
未来时代 (Solidity 0.8.26+) require(cond, CustomError()) 完美 最新版编译器支持直接在 require 里用自定义错误

✅ Checkpoint 3: Price（价格计算）

需要实现：price 函数

测试在检查什么：

xInput xReserves yReserves 预期 yOutput
1 ETH 5 ETH 5 Token 831248957812239453 wei
1 ETH 10 ETH 15 Token 1359916340820223697 wei

函数签名：

price(uint256 xInput, uint256 xReserves, uint256 yReserves) public pure returns (uint256)

核心公式（带 0.3% 手续费）：

$$yOutput = \frac{yReserves \times xInput \times 997}{xReserves \times 1000 + xInput \times 997}$$

伪代码：

inputAmountWithFee = xInput * 997

numerator = yReserves * inputAmountWithFee

denominator = (xReserves * 1000) + inputAmountWithFee

return numerator / denominator

⚠️ 注意事项：

  • 必须是 pure 或 view 函数
  • 整数除法自动向下取整
  • 测试会精确检查输出值，公式必须完全正确

4.1 需要实现：ethToToken 函数

测试在检查什么：

  1. 发送 1 ETH 后，DEX 的 ETH 余额增加 1
  2. 发送 0 ETH 应该 revert
  3. 连续交易时，后一次获得的 Token 更少（滑点效应）
  4. 必须 emit EthToTokenSwap 事件
  5. Token 必须转给调用者

ethToToken() public payable returns (uint256)

实现逻辑：

  1. require(msg.value > 0) — 防止发送 0 ETH
  2. 获取 ETH 储备：address(this).balance - msg.value（关键！）
  3. 获取 Token 储备：token.balanceOf(address(this))
  4. 调用 price() 计算获得多少 Token
  5. 把 Token 转给 msg.sender
  6. emit EthToTokenSwap 事件

事件格式：

event EthToTokenSwap(address swapper, string action, uint256 ethInput, uint256 tokenOutput)

测试用 getEventValue(tx1_receipt, 2) 读取第 3 个参数（index=2），期望值是 ethInput

⚠️ 注意事项：

  • 计算储备时，msg.value 已经包含在 balance 里了！必须减掉

4.2 需要实现：tokenToEth 函数

测试在检查什么：

  1. 发送 1 Token 后，DEX 的 Token 余额增加 1
  2. 发送 0 Token 应该 revert
  3. 必须 emit TokenToEthSwap 事件
  4. 连续交易时，后一次获得的 ETH 更少

函数签名：

tokenToEth(uint256 tokenInput) public returns (uint256)

实现逻辑：

  1. require(tokenInput > 0)
  2. 获取两个储备量
  3. 调用 price() 计算获得多少 ETH
  4. transferFrom 把用户的 Token 转入合约
  5. 用 call 把 ETH 发给用户
  6. emit TokenToEthSwap 事件

⚠️ 注意事项：

  • 先转入 Token，再转出 ETH（重入保护）
  • 用户必须先 approve

✅ Checkpoint 5: Liquidity（流动性管理）

5.1 需要实现：deposit 函数

测试在检查什么：

  1. 存入 5 ETH 后，totalLiquidity 增加 5
  2. 用户的 LP 余额变成 5
  3. 存入 0 ETH 应该 revert
  4. 必须 emit LiquidityProvided 事件

函数签名：

deposit() public payable returns (uint256)

事件格式（顺序很重要！）：

event LiquidityProvided(address provider, uint256 liquidityMinted, uint256 ethDeposit, uint256 tokenDeposit)

核心逻辑：

  1. require(msg.value > 0)
  2. 计算 ETH 储备：address(this).balance - msg.value
  3. 计算 Token 储备
  4. 计算需要存入多少 Token：

$$tokenDeposit = \frac{msg.value \times tokenReserve}{ethReserve}$$

  1. 计算铸造多少 LP Token：

$$liquidityMinted = \frac{msg.value \times totalLiquidity}{ethReserve}$$

  1. 用 transferFrom 转入 Token
  2. 更新 totalLiquidity 和用户的 LP 余额
  3. emit 事件

⚠️ 注意事项：

  • 如果池子比例是 1:1，存入 5 ETH 应该附带 5 Token
  • LP 增发比例 = 资产增加比例

5.2 需要实现：withdraw 函数

测试在检查什么：

  1. 池子 1:1 比例时，提取 1 LP = 提取 1 ETH + 1 Token
  2. 如果 LP 余额不足，应该 revert
  3. totalLiquidity 必须减少
  4. 必须 emit LiquidityRemoved 事件

函数签名：

withdraw(uint256 amount) public returns (uint256, uint256)

事件格式：

event LiquidityRemoved(address provider, uint256 amount, uint256 ethWithdrawn, uint256 tokenWithdrawn)

测试验证了：withArgs(deployer.address, 1.5 ETH, 1.5 ETH, 1.5 Token)

核心逻辑：

  1. require(用户 LP 余额 >= amount)
  2. 计算能提取多少 ETH 和 Token：

$$ethWithdrawn = \frac{amount \times ethReserve}{totalLiquidity}$$

$$tokenWithdrawn = \frac{amount \times tokenReserve}{totalLiquidity}$$

  1. 先减少 用户的 LP 余额和 totalLiquidity（防重入！）
  2. 发送 ETH 和 Token 给用户
  3. emit 事件

    /**
     * @notice 初始化 DEX，将代币从 ERC20 合约转入 DEX。同时存入 ETH 和 Balloons。

        token.transferFrom(msg.sender, address(this), tokens);
        totalLiquidity = msg.value;
        liquidity[msg.sender] = msg.value;

    }

deposit

tokenDeposit 的意义

保持池子比例不变！

假设当前池子：

ETH 储备 = 10 ETH

Token 储备 = 20 Token

比例 = 20 / 10 = 2:1 （每 1 ETH 对应 2 Token）

用户存入 1 ETH，需要多少 Token？

tokenDeposit = 1 × (20 / 10) = 2 Token

存入后：

ETH 储备 = 11 ETH

Token 储备 = 22 Token

比例 = 22 / 11 = 2:1 ✅ 比例不变！

如果不按比例存入，会发生什么？

  • 池子比例被打乱
  • 价格会突然变化
  • 套利者可以攻击你的池子

🎯 liquidityMinted 的意义

公平分配 LP Token！

LP Token 代表你在池子中的 份额。

假设当前池子：

ETH 储备 = 10 ETH

totalLiquidity = 10 LP

（每 1 LP 代表池子的 1/10）

用户存入 1 ETH：

liquidityMinted = (1 × 10) / 10 = 1 LP

存入后：

ETH 储备 = 11 ETH

totalLiquidity = 11 LP

用户持有 1 LP = 池子的 1/11

为什么这样计算？

  • 用户存入了池子的 1/11（1 ETH / 11 ETH）
  • 所以应该获得总 LP 的 1/11（1 LP / 11 LP）

要是用户没有token呢，只有eth怎么添加流动性

如果用户只有 ETH，没有 Token？

在这个简化版 DEX 中：用户无法直接添加流动性。

但在真实世界中，有几种解决方案：

解决方案 1：先用 ETH 买 Token

用户可以分两步操作：

Step 1: 用一半 ETH 买 Token

ethToToken() 用 0.5 ETH → 得到约 0.5 Token

Step 2: 用剩余 ETH + 刚买的 Token 添加流动性

deposit() 用 0.5 ETH + 0.5 Token → 得到 LP

缺点： 用户需要操作两次，而且 Step 1 会产生滑点和手续费。

解决方案 2：单边流动性（Zap In）

一些高级 DEX（如 Uniswap V2 的 Router）提供 "Zap" 功能：

solidity

function zapIn() public payable {

// 1. 用一半 ETH 换成 Token

uint256 halfEth = msg.value / 2;

uint256 tokensBought = _swapEthForToken(halfEth);

// 2. 用另一半 ETH + 换来的 Token 添加流动性

_addLiquidity(halfEth, tokensBought);

}

这是一个一键操作，帮用户自动完成两步！

$$\text{抵押率} = \frac{\text{抵押物价值 (CORN计价)}}{\text{债务总额 (CORN)}}$$

抵押率 ≥ 120%

抵押率 ≥ 1.2

1.2

原式: (A / B) >= 1.2

变形: A >= B × 1.2

整数版: A × 100 >= B × 120

  • 借款人欠 100 CORN
  • 抵押物价值 90 CORN (水下了)
  • 清算人还 50 CORN

清算人应该拿走多少 ETH？

$$\text{清算人获得的ETH} = \frac{\text{还款金额}}{\text{ETH价格}} \times (1 + \text{奖励比例})$$

代入数字： $$\text{清算人获得的ETH} = \frac{50}{价格} \times 1.10$$

完整推导:

假设 ETH 价格 = 1500 CORN (已经暴跌)

清算人还款: 50 CORN

基础可获得 ETH = 50 / 1500 = 0.0333 ETH

10% 奖励 = 0.0333 × 0.10 = 0.00333 ETH

最终获得 = 0.0333 + 0.00333 = 0.0366 ETH

或者直接: 50 / 1500 × 1.10 = 0.0366 ETH

用 CORN 验算清算人的利润:

  • 清算人支付: 50 CORN
  • 清算人获得: 0.0366 ETH × 1500 = 55 CORN 价值的 ETH
  • 净赚 5 CORN！ (10% 奖励)

📐 整数运算版本

把 10% 奖励表示为整数：

奖励分子 = 110

奖励分母 = 100

清算人获得ETH = (还款金额 × 奖励分子) / (价格 × 奖励分母)

          = (还款金额 × 110) / (价格 × 100)

伪代码:

function liquidate(借款人地址, 还款金额):

// 检查：借款人确实不健康

require(

    借款人ETH × 价格 × 100 < 借款人债务 × 清算线,

    "该用户还很健康"

)

// 计算清算人应得的 ETH (含 10% 奖励)

应得ETH = (还款金额 × 110) / (价格 × 100)

// 检查：借款人有这么多抵押物

require(借款人ETH >= 应得ETH, "抵押物不足")

// 清算人支付 CORN

CORN.transferFrom(清算人, this, 还款金额)

// 更新借款人状态

账本_CORN债务[借款人] -= 还款金额

账本_ETH存款[借款人] -= 应得ETH

// 发送 ETH 给清算人

发送ETH(清算人, 应得ETH)

⚠️ 潜在的坑 (Gotchas)

  1. 部分清算 vs 全额清算：允许清算一部分债务更灵活
  2. 清算奖励可能超过抵押物：需要处理"坏账"情况
  3. 闪电贷攻击：清算人可能用闪电贷获取资金，一个交易完成清算并套利
  4. 抢跑 (Front-running)：矿工可能看到清算交易后，自己先执行

知识

仓位

"仓位"来自金融交易术语：

术语 含义
仓位 (Position) 你在市场中持有的资产组合
开仓 建立新头寸（借款 = 开仓）
平仓 关闭头寸（还清债务 = 平仓）
爆仓 仓位不健康被强制清算

简单说：仓位 = 你存了多少 + 你欠了多少，我们通过计算它们的比率来判断是否"健康"。

🏠 现实世界类比：房屋抵押贷款

假设你去银行贷款买房：

你的"仓位" 金额
🏠 抵押物 (房子) 价值 100 万
💰 债务 (欠银行的钱) 借了 60 万

仓位 = 抵押物 + 债务的组合

银行会计算一个比率来判断你的仓位是否"健康"：

$$\text{抵押率} = \frac{100万}{60万} = 166% $$

银行说：只要你的抵押率 >= 120%，我们就认为你的仓位是安全的。

📉 仓位变"危险"的场景

房价暴跌！

你的"仓位" 变化后
🏠 抵押物 (房子) 价值 100万 → 65万
💰 债务 还是欠 60 万

$$\text{新抵押率} = \frac{65万}{60万} = 108% $$

仓位不安全了！ 银行可能会"清算"你——强制卖掉房子来收回贷款。
